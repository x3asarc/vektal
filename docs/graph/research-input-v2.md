# **Architectural Integration of Static Code Graphs, Temporal Episodes, and Observability Streams for the Vektal Platform**

The engineering of a unified observability framework for a complex Shopify multi-supplier platform like Vektal necessitates a transition from traditional linear logging to a multidimensional graph-based architecture. As Vektal scales to accommodate over eight vendors and thousands of SKUs, the operational complexity of managing disparate data streams—static source code structures, dynamic execution episodes, and runtime error reports—becomes a primary bottleneck for root-cause analysis and system stability. By leveraging Neo4j Aura as a centralized high-performance graph database, the platform can unify these currently disconnected datasets into a single, queryable Knowledge Graph (KG). This report details the technical design and implementation strategies for the Vektal Knowledge Graph, focusing on the integration of static Abstract Syntax Tree (AST) metadata, Graphiti-generated temporal episodes, and Sentry-sourced observability data.

The fundamental objective is to move beyond "point-in-time" debugging toward a "context-aware" forensic capability. In the Vektal ecosystem, where the behavior of a function may vary based on the store\_id (vendor context) or the specific proxy environment variables currently in use, the graph model provides the necessary connective tissue. The following sections outline the precise mechanisms for bridging these domains, ensuring bi-temporal accuracy, and maintaining strict multi-tenant isolation.

## **The Graphiti-Static Bridge: Linking Implementation to Episodic Memory**

The integration of Graphiti's episodic memory with a static code graph is the cornerstone of the Vektal observability mission. Graphiti is designed as a Python framework for building and querying temporally-aware knowledge graphs, particularly suited for AI agents that require long-term memory of interactions and system states.1 In the context of Vektal, these interactions are not just user-facing but represent the internal "life" of the platform, recorded as discrete episodes such as FAILURE\_PATTERN, ORACLE\_DECISION, and CODE\_INTENT.3

### **Designing the Cypher Schema for Cross-Domain Linkage**

To bridge the static codebase with dynamic episodes, the Labeled Property Graph (LPG) schema must establish a direct relationship between the "blueprint" (the static function definition) and the "experience" (the episode). The static graph, populated via sync\_to\_neo4j.py, identifies functions and classes based on their location within the repository. The bridge is realized through a specialized relationship, :REFERS\_TO, which links a Graphiti :Episode node to a static :Function node.

| Node Label | Core Properties | Source System | Functional Role |
| :---- | :---- | :---- | :---- |
| :Function | module\_path, function\_signature, lineno, store\_id | sync\_to\_neo4j.py (AST) | Defines the static logic of the platform. |
| :Module | module\_path, package\_name, store\_id | sync\_to\_neo4j.py (AST) | Groups functions into logical file-based units. |
| :Episode | episode\_id, content, type, timestamp, store\_id | graphiti-core | Records a discrete event or observation during execution. |
| :Entity | name, label, summary, embedding, store\_id | graphiti-core | Represents extracted concepts (e.g., Supplier A, Proxy B). |

This schema allows for sophisticated traversals. For instance, an episode of type FAILURE\_PATTERN generated by a failed scraping task can be linked to the specific :Function node that executed the scraping logic. By ensuring that both the static code graph and the episodic graph utilize the same store\_id property, the platform maintains per-vendor isolation throughout the bridge.5

### **Shared Properties and Semantic Alignment**

The efficacy of the bridge depends on the consistency of the properties used for mapping. The module\_path and function\_signature serve as the primary identifiers. The module\_path represents the relative path of the Python file within the Vektal repository, while the function\_signature is a unique string constructed from the module path, class name (if applicable), and function name.3

Graphiti's "Episodic Processing" maintains data provenance by ingesting data as discrete episodes.2 When the Vektal platform ingests a CODE\_INTENT episode, the metadata must include the function\_signature. This enables queries such as: "Find all functions called by API X that have associated FAILURE\_PATTERN episodes in Graphiti from the last 24 hours." Such a query traverses from an :APIRoute node (derived from AST @app.route decorators) to the :Function it triggers, and then follows the :REFERS\_TO incoming edges from :Episode nodes filtered by the FAILURE\_PATTERN type and a temporal predicate.7

Furthermore, Graphiti's ability to extract and deduplicate entities (e.g., people, organizations, products) creates a "Semantic Entity Subgraph".4 If an ORACLE\_DECISION episode records that the system switched from Supplier X to Supplier Y, the episode is linked to both the :Function responsible for the decision and the :Entity nodes representing the suppliers. This tripartite link—Code, Episode, and Entity—provides the forensic depth required to understand why a specific code path was chosen in a given operational context.4

## **Bi-Temporal Versioning for Neo4j Aura: Preserving Code History**

A significant challenge in graph-augmented observability is the evolution of the codebase. As functions are refactored, moved, or deleted, a simple "current state" graph becomes insufficient for historical debugging. If a Sentry issue occurred 48 hours ago, it must be mapped to the version of the code that existed at that precise moment, not the current version on the main branch. To solve this, Vektal implements a bi-temporal versioning strategy within Neo4j Aura.9

### **The SCD Type 2 Pattern in a Graph Context**

The versioning strategy avoids the use of DETACH DELETE, which is both computationally expensive on Aura and destructive to historical context. Instead, the platform adopts a pattern similar to Slowly Changing Dimensions (SCD) Type 2\.9 Every code-related node (e.g., :Function, :Class, :Module) is assigned a StartDate and an EndDate property.

* **Current Active Node:** The node representing the current version of the code has a StartDate set to the time of deployment and an EndDate set to a future sentinel value (e.g., 9999-12-31) or null.9  
* **Historical Node:** When a code change is detected by sync\_to\_neo4j.py, the existing node's EndDate is updated to the current timestamp. A new node is then created for the updated function version with a new StartDate.9  
* **Deletions:** If a function is removed from the source code, its corresponding node is not deleted. Instead, its EndDate is set to the current time, and it is marked with a :Deleted label to exclude it from "current state" queries while preserving it for historical traversals.12

### **Transactional Batching and Aura Performance**

Neo4j Aura, as a managed service, has specific constraints regarding Java heap memory and transaction sizes. Large-scale updates to the code graph—common during massive refactors—can lead to TransactionOutOfMemoryError if executed in a single scope.14 The Vektal sync strategy incorporates the CALL {... } IN TRANSACTIONS OF 10000 ROWS logic, introduced in Neo4j 5.x, to manage these updates efficiently.15

This subquery syntax allows the sync\_to\_neo4j.py script to process updates in discrete batches, each with its own commit. This is particularly vital for the bi-temporal logic, which requires both an UPDATE to old nodes and a CREATE for new nodes. By batching these operations, the platform avoids polluting the transaction log and ensures that the "highest transaction in a store file" is represented correctly, maintaining store consistency.14

## **AST Metadata Harvesting: Extracting Structural Intelligence**

The static component of the Vektal Knowledge Graph is derived from deep harvesting of the Python codebase using the ast (Abstract Syntax Tree) module. This process goes beyond identifying function names; it extracts the operational metadata that defines the platform's behavior in the Flask and Celery environments.16

### **Extraction Patterns for Flask and Celery**

The ast.NodeVisitor implementation targets specific decorators and function calls to enrich the graph with structural dependencies.

* **Flask @app.route Extraction:** The harvester identifies ast.Call nodes where the decorator is app.route. It extracts the url\_template from the first positional argument and the allowable HTTP methods (e.g., \`\`) from the methods keyword argument.18 These are mapped to properties on an :APIRoute node, which is then linked to the decorated :Function.20  
* **Celery @app.task Extraction:** For background concurrency, the harvester extracts the queue and priority arguments. The queue name is mapped directly to the Vektal-specific queues: assistant.t1, assistant.t2, assistant.t3, scrape, or ingest.21 The priority (typically an integer between 0 and 9\) is stored to help identify high-priority failure patterns.22

| Decorator | Argument | AST Node Type | Graph Mapping |
| :---- | :---- | :---- | :---- |
| @app.route | path | ast.Constant / ast.Str | :APIRoute.url\_template |
| @app.route | methods | ast.List | :APIRoute.http\_methods |
| @app.task | queue | ast.Constant | :CeleryTask.queue |
| @app.task | priority | ast.Constant | :CeleryTask.priority |

### **Mapping Environment Variable Dependencies**

A critical aspect of Vektal's multi-supplier architecture is the use of environment variables to configure supplier-specific behavior, such as proxy selection. The harvester visits all ast.Call nodes representing os.getenv() or os.environ.get().25 These calls are mapped to central :EnvVar nodes.

By linking the calling :Function to the :EnvVar via a :DEPENDS\_ON\_CONFIG relationship, the graph can visualize the configuration surface area of the entire platform.25 If a particular environment variable is misconfigured, the graph immediately identifies every function that might be affected. This is essential for the "Proxy X failure" forensic scenario, where the graph reveals the fallback proxies defined in the same functional context.25

## **Sentry-to-Graph Grounding: Correlating Errors with Blueprints**

Observability without grounding is merely a collection of stack traces. In the Vektal platform, Sentry serves as the source of truth for errors, but these errors are currently disconnected from the static and episodic graphs. The sentry\_issue\_puller.py script bridges this gap by mapping Sentry issues directly to the versioned :Function nodes.29

### **Normalized Path Resolution and Mapping Logic**

The mapping relies on two primary data points from the Sentry stack trace: abs\_path and lineno. However, discrepancies often exist between the absolute path in the runtime container and the relative path stored in the static code graph. Vektal utilizes a normalization strategy, potentially involving Sentry's RewriteFrames integration, to ensure that the file paths align.29

The grounding query logic identifies the correct node by matching the normalized module\_path and the lineno within the range of lines occupied by the function (as harvested by the AST processor).

### **Observability Enrichment via Graphiti**

Once a Sentry issue is grounded in the static graph, the platform can immediately traverse to the associated Graphiti episodes. This enables the correlation of high-level AI agent decisions (ORACLE\_DECISION) with low-level runtime errors (FAILURE\_PATTERN).3 For instance, if a Sentry issue indicates a rate-limiting error, the developer can look for episodes where an AI agent recently increased the scraping frequency for that specific store\_id, providing immediate context for the failure.3

## **Multi-Tenant Isolation: Architecting the Per-Vendor Blast Radius**

The Vektal platform serves multiple independent vendors, each with its own set of SKUs and operational context. Multi-tenant isolation is therefore not just a security requirement but a critical performance and diagnostic feature. The graph architecture must ensure that an error for Store A does not impact the analysis or performance of Store B.5

### **Store-ID Indexing and Partitioning**

The platform implements a logical isolation strategy within a single Neo4j Aura instance. Every operational node—including Sentry Issues, Graphiti Episodes, and even the static code nodes (since code might be deployed on a per-tenant basis)—must support a store\_id property.5

To ensure high-performance traversals within a tenant's context, composite indexes are created for all primary node labels. For example:

* CREATE INDEX function\_store\_idx FOR (n:Function) ON (n.store\_id, n.function\_signature)  
* CREATE INDEX episode\_store\_idx FOR (n:Episode) ON (n.store\_id, n.timestamp)  
* CREATE INDEX issue\_store\_idx FOR (n:SentryIssue) ON (n.store\_id, n.issue\_id)

These indexes allow the Neo4j query planner to quickly narrow the search space to a specific vendor's data, effectively creating "virtual partitions" within the shared database.5 This approach is preferred over a database-per-tenant model for the initial 8+ vendors, as it simplifies maintenance and cross-tenant aggregate analysis while still providing strong isolation through query discipline.5

### **Tenant-Specific Blast Radius Analysis**

The store\_id property enables the platform to perform "blast radius" analysis confined to a single vendor. If a shared resource, such as a supplier's API, is experiencing downtime, the graph can identify every function and API route *for that specific vendor* that depends on the resource. This allows for proactive alerting and more targeted troubleshooting. The isolation ensures that the "noisy neighbor" effect—where one vendor's massive data volume impacts another's performance—is mitigated through strict index-backed filtering.5

## **Data Topology: Mapping SQLAlchemy to Graph Relationships**

The final component of the Vektal Knowledge Graph is the mapping of the relational data model to the graph structure. By visualizing the dependencies between code functions and database tables, the platform can identify potential data-integrity issues and optimize database interactions.35

### **Relational-to-Graph Transformation Logic**

Vektal utilizes SQLAlchemy models to define its relational schema. The mapping process transforms these definitions into graph nodes and relationships based on several core principles:

* **Tables to Nodes:** Each SQLAlchemy entity table (e.g., suppliers, orders) becomes a :Table node. Columns such as table\_name and schema\_version are stored as properties.36  
* **Foreign Keys to Relationships:** Instead of implicit links via keys, relationships between tables are modeled as explicit :DEPENDS\_ON or :REFERENCES edges. The direction of these edges follows the business semantics (e.g., (Order)--\>(Supplier)) rather than the direction of the foreign key constraint.36  
* **Junction Tables to Weighted Edges:** Many-to-many relationships, which require junction tables in SQL (e.g., store\_vendors), are converted into direct relationships between the primary entity nodes. Any data stored in the junction table (e.g., access\_level) is moved to properties on the relationship itself.36

| SQL Source | Graph Representation | Mapping Logic |
| :---- | :---- | :---- |
| Entity Table | Node Label (:Table) | Rows become node instances; columns become properties.36 |
| Foreign Key | Relationship Type | Meaningful verbs (e.g., PLACED\_BY) replace column names.38 |
| Junction Table | Relationship Property | Link attributes are stored on the edge, not a separate node.36 |
| Lookup Table | Node Property | Small, static lists (e.g., status codes) are stored as properties on the entity.38 |

### **Visualizing Code-to-Data Dependencies**

The power of this mapping lies in its integration with the static code graph. By analyzing the ast for database access patterns (e.g., session.query(Supplier)), the system can create :ACCESSES relationships between :Function nodes and :Table nodes.35 This enables the platform to visualize the entire data dependency chain. A developer can query for all API routes that read from a specific table, or identify which Celery tasks update a given set of attributes, providing a complete view of the platform's data topology.35

## **Forensic Playbook: Advanced Cypher Queries for Root-Cause Analysis**

The following queries represent the peak utility of the Vektal Knowledge Graph, enabling rapid forensic analysis across the code, episode, and error domains.

### **1\. Fallback Proxy Discovery in Failure Contexts**

**Scenario:** A scraping task is failing because the primary proxy is blocked. Find the secondary Proxy Y linked in the EnvVar context for this specific store\_id.

Cypher

MATCH (s:Store {store\_id: $sid})--\>(f:Function)  
MATCH (f)--\>(primary:EnvVar {name: 'PROX\_PRIMARY'})  
MATCH (f)--\>(backup:EnvVar {name: 'PROX\_BACKUP'})  
MATCH (e:Episode {type: 'FAILURE\_PATTERN', store\_id: $sid})--\>(f)  
WHERE e.content CONTAINS primary.value  
RETURN f.function\_signature, backup.value AS fallback\_config, e.timestamp  
ORDER BY e.timestamp DESC  
LIMIT 5

### **2\. Blast Radius of a Supplier API Refactor**

**Scenario:** A supplier has changed its API schema for the orders table. Identify all affected API routes and high-priority Celery tasks for Store 101\.

Cypher

MATCH (t:Table {name: 'orders', store\_id: '101'})\<--(f:Function)  
OPTIONAL MATCH (api:APIRoute)--\>(f)  
OPTIONAL MATCH (task:CeleryTask {priority: 0})--\>(f)  
RETURN f.function\_signature,   
       collect(DISTINCT api.url\_template) AS affected\_routes,   
       collect(DISTINCT task.queue) AS urgent\_queues

### **3\. Correlating Sentry Issues with Historical AI Decisions**

**Scenario:** A sudden spike in Sentry issues has been detected. Find any ORACLE\_DECISION episodes that occurred within a 15-minute window preceding these errors.

Cypher

MATCH (si:SentryIssue {store\_id: $sid})--\>(f:Function)  
MATCH (e:Episode {type: 'ORACLE\_DECISION', store\_id: $sid})--\>(f)  
WHERE duration.between(e.timestamp, si.timestamp).minutes \<= 15  
  AND e.timestamp \< si.timestamp  
RETURN si.issue\_id, si.message, e.content AS ai\_context, e.timestamp AS decision\_time  
ORDER BY si.timestamp DESC

### **4\. Identifying Inconsistent Supplier Performance across Stores**

**Scenario:** A specific Entity (e.g., "Supplier X") is linked to multiple FAILURE\_PATTERN episodes. Determine if this failure is localized to one store or widespread across the platform.

Cypher

MATCH (en:Entity {name: 'Supplier X'})\<--(e:Episode {type: 'FAILURE\_PATTERN'})  
MATCH (e)--\>(f:Function)  
RETURN e.store\_id, count(e) AS failure\_count, collect(DISTINCT f.function\_signature) AS failing\_functions  
ORDER BY failure\_count DESC

### **5\. Tracking Code Evolution during Operational Instability**

**Scenario:** A function is currently failing. Use the bi-temporal labels to find the last known stable version of the code and the CODE\_INTENT episode associated with its creation.

Cypher

MATCH (f\_fail:Function {function\_signature: $sig, store\_id: $sid})  
MATCH (f\_old:Function {function\_signature: $sig, store\_id: $sid})  
WHERE f\_old.EndDate \= f\_fail.StartDate  
MATCH (e:Episode {type: 'CODE\_INTENT'})--\>(f\_old)  
RETURN f\_old.StartDate, f\_old.EndDate, e.content AS design\_intent, f\_old.checksum

## **Future Outlook: Graph-Driven Autonomic Observability**

The unification of static code, temporal episodes, and observability streams into a high-performance Neo4j Aura instance positions the Vektal platform at the forefront of autonomic engineering. As the Knowledge Graph grows, it moves from a passive record to an active component of the system's reasoning engine. The bi-temporal versioning ensures that the AI agents responsible for ORACLE\_DECISION episodes have a perfect historical memory, allowing them to learn from past failures grounded in specific code versions.3

Furthermore, the hybrid search capabilities of Graphiti—combining semantic vector embeddings with BM25 keyword search—allow for sub-second query latency even as the data volume scales.1 This performance is critical for real-time applications, such as voice-activated diagnostic assistants or automated recovery systems that must traverse the graph to find fallback configurations during a live incident.3

In conclusion, the Vektal Knowledge Graph is more than a database; it is a unified representation of the platform's implementation, intent, and experience. By strictly adhering to multi-tenant isolation and leveraging the structural intelligence of the Python AST, Vektal ensures that its observability framework is as scalable and interconnected as the multi-supplier ecosystem it supports. This architectural foundation provides the transparency required to manage the next generation of Shopify-integrated platforms with confidence and precision.
