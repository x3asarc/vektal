## Summary

Product enrichment pipeline architecture for transforming raw scraped product data into high-quality, SEO-optimized German-language content has matured significantly. The standard approach combines traditional rule-based attribute extraction with AI-powered description generation, semantic embeddings for search, and comprehensive quality scoring.

**Key findings:**
- Hybrid enrichment (rule-based + AI) achieves best cost-quality balance
- Sentence-transformers with 768-dimensional embeddings (paraphrase-multilingual-mpnet-base-v2) are standard for German e-commerce semantic search
- Quality scoring uses weighted formula: completeness (40-50%), richness (30-40%), SEO health (20-30%)
- OpenRouter provides 75-95% cost savings vs direct API calls, with Gemini Flash 1.5 as optimal German text generator ($0.10/$0.40 per 1M tokens)
- Jinja2 is industry standard for content templating with built-in i18n support
- Product family grouping uses title-based normalization (remove size/color) with SKU pattern matching as fallback

**Primary recommendation:** Migrate existing side-project enrichment pipeline to src/core/enrichment/, integrate vendor YAML config, use paraphrase-multilingual-mpnet-base-v2 for embeddings, and implement weighted quality scoring with pgvector for PostgreSQL vector storage.

### Pattern 1: Multi-Stage Pipeline with Checkpoints
**What:** Sequential enrichment stages with intermediate save points for resumability
**When to use:** Long-running enrichment of large catalogs (1000+ products)
**Example:**
```python
# Source: Existing side-project/src/data_pipeline/pipeline.py (verified working)
class EnrichmentPipeline:
    def run(self, skip_audit=False, skip_extraction=False, ...):
        # Stage 1: Audit raw data
        if not skip_audit:
            self.df = self._step_audit()
        else:
            self._load_checkpoint('audit')

        # Stage 2: Extract attributes
        if not skip_extraction:
            self.df = self._step_extraction(self.df)

        # Stage 3: AI generation
        if not skip_ai:
            self.df = self._step_ai_generation(self.df)

        # Save checkpoint after each stage
        self._save_checkpoint(df, 'stage_name')
```

### Pattern 2: Weighted Quality Scoring
**What:** Component-based scoring with industry-standard weightings
**When to use:** All enriched products need objective quality measurement
**Example:**
```python
# Source: Research synthesis from multiple sources
def calculate_quality_score(product: dict) -> int:
    """Calculate 0-100 quality score"""
    score = 0

    # Completeness (40-50 points)
    desc_len = len(product.get('description', ''))
    if desc_len > 100: score += 40
    elif desc_len > 50: score += 30
    elif desc_len > 20: score += 15

    # Richness (30-40 points)
    if product.get('extracted_color'): score += 10
    if product.get('extracted_size'): score += 10
    if product.get('extracted_material'): score += 10

    # SEO Health (20-30 points)
    if product.get('inferred_category'): score += 10
    if len(product.get('product_type', '')) > 2: score += 10
    if len(product.get('tags', '')) > 10: score += 10

    return min(score, 100)
```

### Pattern 3: Few-Shot AI Generation with Similarity Search
**What:** Use embedding similarity to find 3 similar products as examples for AI prompts
**When to use:** Generating descriptions for products with limited vendor data
**Example:**
```python
# Source: side-project/src/data_pipeline/ai_descriptions.py (verified pattern)
def generate_description(target_product, catalog, model):
    # Find 3 most similar products using embeddings
    target_embedding = embedding_model.encode(target_product['title'])
    catalog_embeddings = np.vstack(catalog['embedding'].values)
    similarities = cosine_similarity([target_embedding], catalog_embeddings)[0]
    top_3_indices = np.argsort(similarities)[-3:][::-1]

    # Build few-shot prompt with examples
    examples = [catalog.iloc[i] for i in top_3_indices]

    1. {examples[0]['title']}: {examples[0]['description']}
    2. {examples[1]['title']}: {examples[1]['description']}
    3. {examples[2]['title']}: {examples[2]['description']}

    Schreibe nun für: {target_product['title']}"""

    return openrouter_api.generate(prompt)
```

### Anti-Patterns to Avoid
- **Regenerating embeddings on every field change:** Only regenerate when title, description, or tags change (the searchable fields). Updating price doesn't require re-embedding.
- **Single-field embeddings:** Concatenating title + description + tags (weighted) captures more semantic meaning than title-only embeddings.
- **Synchronous AI calls in request path:** Always queue enrichment as background jobs (Celery). 5-30 second AI calls block user experience.
- **No caching for identical prompts:** AI responses for "generate description for Pentart Acrylfarbe Rot 20ml" should be cached indefinitely (product doesn't change).
- **Generic error handling:** Differentiate between rate limits (retry with backoff), invalid input (skip), and service outages (alert). Don't treat all failures the same.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| German umlaut URL handling | Custom string replacement | Jinja2 filters + slugify library | Edge cases: ß→ss, ä→ae, combined with special chars. Battle-tested unicode normalization. |
| Vector similarity search | NumPy cosine similarity loops | pgvector with HNSW indexing | 28x lower latency, 16x higher throughput. HNSW handles 100k+ products efficiently. |
| API rate limiting | Sleep timers and counters | tenacity @retry decorator + cachetools | Exponential backoff, jitter, per-endpoint limits. Handles 429 errors automatically. |
| Variant detection | Regex on product titles | Existing families.py pattern + SKU matching | Already handles German size/color patterns. SKU prefix matching catches edge cases. |
| Embedding model selection | Train custom model | Pre-trained paraphrase-multilingual-mpnet-base-v2 | Trained on 1B+ sentence pairs, 50 languages. Fine-tuning costs weeks + GPU hours. |
| Template validation | Manual try/except | Jinja2 StrictUndefined + schema validation | Catches {{missing_variable}} at load time, not generation time. Prevents silent failures. |
| Quality scoring formula | Ad-hoc thresholds | Industry standard weights (PQS formula) | Completeness 40-50%, Richness 30-40%, SEO 20-30%. Proven in production e-commerce systems. |

**Key insight:** AI/ML infrastructure is commoditized. Use battle-tested libraries for embeddings, caching, and retry logic. Only build custom logic for domain-specific rules (German product categories, vendor-specific attribute patterns).

## Common Pitfalls

### Pitfall 1: Embedding Dimension Mismatch Between Models
**What goes wrong:** Switching from 384-dim model (all-MiniLM-L6-v2) to 768-dim model (paraphrase-multilingual-mpnet-base-v2) breaks existing vector database queries.
**Why it happens:** Vector similarity requires identical dimensions. You can't compare 384-dim to 768-dim vectors.
**How to avoid:**
- Choose embedding model BEFORE Phase 3 database schema design
- Document embedding dimension in schema comments: `embedding vector(768)`
- If switching models, re-embed entire catalog (budget 1-2 hours for 10k products)
**Warning signs:** `ERROR: dimension mismatch in vector operation` from pgvector queries

### Pitfall 2: German Umlaut Inconsistency in URLs
**What goes wrong:** URL handles generated as `pentart-acrylfarbe-grun` (without umlaut) don't match search queries for "grün" (with umlaut).
**Why it happens:** URL slug generation strips umlauts for SEO, but search index keeps them. Different normalization strategies.
**How to avoid:**
- Use consistent normalization: ü→ue, ö→oe, ä→ae, ß→ss
- Store both normalized (URL) and original (display) versions
- Google Germany accepts both, but be consistent
**Warning signs:** 404 errors on German product URLs, SEO tool warnings about URL mismatches

### Pitfall 3: Quality Score Threshold Too High (>90% requirement)
**What goes wrong:** Pipeline rejects 80% of products, AI generation costs skyrocket trying to reach 90+ scores.
**Why it happens:** E-commerce reality: vendor data is messy. Perfect scores (95-100) require manual curation, not achievable at scale.
**How to avoid:**
- Target >85% average score, not minimum score
- Allow 70-85 range for "good enough" products
- Focus AI generation on <70 products (truly poor quality)
- Reserve 90+ for hero products (manual curation)
**Warning signs:** Quality gate fails >50% of products, AI generation costs exceed budget

### Pitfall 4: AI Prompt Inconsistency (Different Results for Same Product)
**What goes wrong:** Re-running enrichment generates completely different descriptions for identical products.
**Why it happens:** High temperature (0.8-1.0) + no caching + insufficient examples in prompt.
**How to avoid:**
- Set temperature=0.3 for product descriptions (deterministic)
- Cache AI responses by (product_title, vendor, category) key
- Include 3+ examples in few-shot prompt (consistency)
- Use system prompt to enforce tone: "Du bist ein sachlicher E-Commerce Texter..."
**Warning signs:** A/B testing shows customers confused by tone variations, SEO penalties for duplicate content

### Pitfall 5: Batch Size Mismatch with Rate Limits
**What goes wrong:** Processing 100 products/batch hits OpenRouter rate limit (10 req/sec), entire batch fails.
**Why it happens:** Batch size chosen for speed, not API constraints. No backoff between requests.
**How to avoid:**
- Calculate: rate_limit / batch_size = minimum delay
- Example: 10 req/sec limit, batch of 10 = 1 second between batches
- Use tenacity @retry with exponential backoff for 429 errors
- Monitor OpenRouter dashboard for actual usage vs limits
**Warning signs:** `RateLimitError` exceptions, incomplete enrichment runs, wasted API credits

### Pitfall 6: Embedding Generation Before Quality Gate
**What goes wrong:** Generate embeddings for 10k products, then quality gate rejects 2k. Wasted 20% of embedding computation.
**Why it happens:** Pipeline runs stages sequentially without validation checkpoints.
**How to avoid:**
- Run quality gate BEFORE expensive operations (AI, embeddings)
- Stage order: Extract → Validate → AI (if needed) → Validate again → Embeddings
- Checkpoint after validation: "Only 8k products passed, proceeding with embeddings"
**Warning signs:** High compute costs, long pipeline runs, re-enrichment required

### German URL Handle Generation
```python
# Source: https://gist.github.com/davidsneighbour/9c6e1eb93d110b4bf59b (WordPress German slugs)
import re
from unidecode import unidecode

def generate_url_handle(title: str) -> str:
    """Generate SEO-friendly German URL handle"""
    # German umlaut transliteration
    replacements = {
        'ä': 'ae', 'ö': 'oe', 'ü': 'ue', 'ß': 'ss',
        'Ä': 'Ae', 'Ö': 'Oe', 'Ü': 'Ue'
    }

    handle = title.lower()
    for umlaut, replacement in replacements.items():
        handle = handle.replace(umlaut, replacement)

    # Remove special characters, keep hyphens
    handle = re.sub(r'[^a-z0-9\-]', '-', handle)
    handle = re.sub(r'-+', '-', handle)  # Collapse multiple hyphens
    handle = handle.strip('-')

    return handle

# Example: "Pentart Acrylfarbe Grün 20ml" → "pentart-acrylfarbe-gruen-20ml"
```

### OpenRouter AI Description Generation
```python
# Source: side-project/src/data_pipeline/ai_descriptions.py (verified working)
import requests
from cachetools import TTLCache

# Cache AI responses for 30 days (product descriptions don't change)
cache = TTLCache(maxsize=10000, ttl=2592000)

def generate_description(product: dict, examples: list, api_key: str) -> str:
    """Generate German product description via OpenRouter"""

    # Check cache first
    cache_key = (product['title'], product['vendor'], product.get('inferred_category'))
    if cache_key in cache:
        return cache[cache_key]

    system_prompt = """Du bist ein erfahrener E-Commerce Texter für Bastelschachtel.at.
    Schreibe sachliche, informative Produktbeschreibungen auf Deutsch."""

    examples_text = "\n\n".join([
        f"Titel: {ex['title']}\nBeschreibung: {ex['description'][:200]}"
        for ex in examples[:3]
    ])

    user_prompt = f"""Produkt: {product['title']}
    Marke: {product.get('vendor', 'Pentart')}
    Kategorie: {product.get('inferred_category', 'Bastelbedarf')}

    Ähnliche Produkte:
    {examples_text}

    Schreibe eine deutsche Produktbeschreibung (max 200 Zeichen, 2-3 Sätze).
    Erwähne Verwendungszweck und wichtige Eigenschaften."""

    response = requests.post(
        "https://openrouter.ai/api/v1/chat/completions",
        headers={
            "Authorization": f"Bearer {api_key}",
            "Content-Type": "application/json"
        },
        json={
            "model": "google/gemini-flash-1.5",  # $0.10/$0.40 per 1M tokens
            "messages": [
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_prompt}
            ],
            "temperature": 0.3,  # Low for consistency
            "max_tokens": 300
        }
    )

    description = response.json()['choices'][0]['message']['content'].strip()
    cache[cache_key] = description  # Cache result
    return description
```

### Embedding Generation with pgvector Storage
```python
# Source: https://www.timescale.com/blog/postgresql-as-a-vector-database-create-store-and-query-openai-embeddings-with-pgvector
from sentence_transformers import SentenceTransformer
import psycopg2

# Load model once (cache in memory)
embedding_model = SentenceTransformer('sentence-transformers/paraphrase-multilingual-mpnet-base-v2')

def generate_and_store_embedding(product: dict, conn):
    """Generate 768-dim embedding and store in PostgreSQL"""

    # Weighted concatenation for better search
    search_text = (
        f"{product['title']} " * 2 +  # Title weight: 2x
        f"{product['description']} " +  # Description weight: 1x
        f"{' '.join(product.get('tags', []))} " * 1.5  # Tags weight: 1.5x
    )

    # Generate embedding (768 dimensions)
    embedding = embedding_model.encode(search_text)

    # Store in PostgreSQL with pgvector
    with conn.cursor() as cur:
        cur.execute("""
            UPDATE products
            SET embedding = %s::vector(768),
                embedding_generated_at = NOW()
            WHERE id = %s
        """, (embedding.tolist(), product['id']))

    conn.commit()

# Query similar products
def find_similar_products(query_text: str, conn, limit=10):
    embedding = embedding_model.encode(query_text)

    with conn.cursor() as cur:
        cur.execute("""
            SELECT id, title, description,
                   1 - (embedding <=> %s::vector(768)) as similarity
            FROM products
            ORDER BY embedding <=> %s::vector(768)
            LIMIT %s
        """, (embedding.tolist(), embedding.tolist(), limit))
        return cur.fetchall()
```

### Retry Logic with Exponential Backoff
```python
# Source: https://tenacity.readthedocs.io/
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
import requests

@retry(
    retry=retry_if_exception_type(requests.exceptions.RequestException),
    wait=wait_exponential(multiplier=1, min=1, max=60),  # 1s, 2s, 4s, ... max 60s
    stop=stop_after_attempt(5)  # Give up after 5 tries
)
def call_ai_api(prompt: str, api_key: str) -> str:
    """Call AI API with automatic retry on failure"""
    response = requests.post(
        "https://openrouter.ai/api/v1/chat/completions",
        headers={"Authorization": f"Bearer {api_key}"},
        json={"model": "google/gemini-flash-1.5", "messages": [{"role": "user", "content": prompt}]},
        timeout=30
    )
    response.raise_for_status()  # Raises exception for 4xx/5xx
    return response.json()['choices'][0]['message']['content']
```

## Open Questions

Things that couldn't be fully resolved:

1. **OpenRouter model pricing volatility**
   - What we know: OpenRouter aggregates 400+ models, pricing changes frequently
   - What's unclear: Whether to lock model choice (Gemini Flash 1.5) or allow dynamic switching based on cost/quality
   - Recommendation: Start with Gemini Flash 1.5 ($0.10/$0.40), make model configurable in vendor YAML for future flexibility
   - Confidence: MEDIUM (pricing found via docs, but German-specific quality requires testing)

2. **Embedding regeneration strategy for incremental updates**
   - What we know: Only regenerate when title/description/tags change (not price/inventory)
   - What's unclear: Whether to regenerate on typo fixes, or only "significant" content changes
   - Recommendation: Track content hash (MD5 of title+description+tags), regenerate only when hash changes
   - Confidence: HIGH (standard pattern from multiple sources)

3. **Quality score threshold calibration for German e-commerce**
   - What we know: Industry standard is >85% average, with 40/30/20 component weights
   - What's unclear: Whether German craft products have different completeness expectations vs general e-commerce
   - Recommendation: Start with 85% target, measure actual distribution after first 1000 products, adjust if median <80 or >95
   - Confidence: MEDIUM (formula validated, but threshold may need domain tuning)

4. **Local LLM vs API for German descriptions**
   - What we know: Llama 3.1 8B and Mistral 7B available locally/free on OpenRouter
   - What's unclear: Quality comparison with Gemini Flash 1.5 for German product descriptions
   - Recommendation: Prototype with both, A/B test 100 products, measure quality score delta
   - Confidence: LOW (no German e-commerce benchmarks found, requires testing)

5. **Multi-field vs concatenated embeddings for product search**
   - What we know: Concatenated vectors (title+desc+tags weighted) are simpler, multi-field allow query-time weighting
   - What's unclear: Performance difference for 10k-100k product catalogs in German
   - Recommendation: Start with concatenated (simpler), migrate to multi-field if search quality <80% satisfaction
   - Confidence: MEDIUM (research shows 40-50bp improvement, but small catalogs may not benefit)
