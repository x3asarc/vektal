## Key Technical Details

### Embedding Generation Pipeline

```python
product = {
    'title': 'Pentart Acrylfarbe Rot 20ml',
    'description': 'Hochwertige Acrylfarbe für Holz und Metall',
    'tags': 'farbe,acryl,pentart'
}

# Field weighting via text repetition
search_text = (
    "Pentart Acrylfarbe Rot 20ml " +  # Title 2x
    "Pentart Acrylfarbe Rot 20ml " +
    "Hochwertige Acrylfarbe für Holz und Metall " +  # Description 1x
    "farbe acryl pentart " +  # Tags 1.5x
    "farbe acryl"  # (half of tags)
)

embedding = model.encode(search_text)  # -> (768,) numpy array
```

### Content Hashing Strategy

Only hash searchable content (title, description, tags):

```python
# Price change does NOT trigger re-embedding
hash1 = compute_content_hash({'title': 'Test', 'price': 10})
hash2 = compute_content_hash({'title': 'Test', 'price': 20})
assert hash1 == hash2  # ✓ Same hash

# Title change DOES trigger re-embedding
hash3 = compute_content_hash({'title': 'Changed'})
assert hash1 != hash3  # ✓ Different hash
```

This prevents expensive re-embedding for non-semantic changes (price, inventory).

## Decisions Made

### 1. Embedding Model: paraphrase-multilingual-mpnet-base-v2

**Chosen:** `paraphrase-multilingual-mpnet-base-v2`
**Why:**
- 768-dimensional (standard size for pgvector)
- Multilingual support (German + English)
- Best balance of quality vs speed (2-3s load time, fast inference)
- Proven performance on semantic similarity tasks

**Alternatives considered:**
- `all-mpnet-base-v2`: English only, not suitable
- `multilingual-e5-large`: 1024-dim, slower, overkill for our scale

### 2. Field Weighting: Text Repetition

**Chosen:** Repeat text (title 2x, tags 1.5x, description 1x)
**Why:**
- Simple and interpretable
- No custom training needed
- Works with any embedding model
- Easy to tune by changing repetition count

**Alternatives considered:**
- Learned weights: Requires training data
- TF-IDF: More complex, harder to maintain
- Custom embeddings: Expensive to train and update

### 3. Lazy Model Loading

**Chosen:** Load model on first use
**Why:**
- Don't pay 2-3s startup cost if not using embeddings
- Multiple processes can share same model instance
- Allows CLI commands that don't need embeddings to start fast

**Alternatives considered:**
- Eager loading: Wastes time for non-embedding tasks
- Separate service: Over-engineering for current scale

### 4. Content Hashing Strategy

**Chosen:** Hash only title + description + tags
**Why:**
- Price/inventory changes don't affect semantic meaning
- Prevents wasteful re-embedding of thousands of products
- Clear cache invalidation policy

**Alternatives considered:**
- Hash all fields: Triggers re-embedding on price changes (bad)
- Time-based invalidation: No way to detect content changes (worse)

## Integration Points

### Phase 03: Database Schema
Embeddings will be stored in `pgvector` columns:

```sql
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    title TEXT,
    description TEXT,
    embedding VECTOR(768),  -- Store embedding here
    embedding_hash TEXT,    -- Content hash for cache invalidation
    embedding_updated_at TIMESTAMP
);

CREATE INDEX ON products USING ivfflat (embedding vector_cosine_ops);
```

### Phase 11: Semantic Search
EmbeddingGenerator will power semantic search queries:

```python
# User searches for "red acrylic paint"
query_embedding = generator.generate_embedding({'title': 'red acrylic paint'})

# Find similar products via vector search
similar_products = db.execute("""
    SELECT title, description,
           1 - (embedding <=> %s) AS similarity
    FROM products
    ORDER BY embedding <=> %s
    LIMIT 10
""", [query_embedding, query_embedding])
```
