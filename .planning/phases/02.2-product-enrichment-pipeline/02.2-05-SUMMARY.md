## What Was Built

### 1. TemplateEngine (Jinja2-based)

**Purpose:** Render vendor YAML content templates with German text support

**Location:** `src/core/enrichment/templating/engine.py` (145 lines)

**Features:**
- Jinja2 Environment with StrictUndefined (fails on missing variables)
- Custom filters: `german_capitalize`, `umlaut_safe`, `truncate_german`
- `render()`: Render single template string with context
- `render_product_content()`: Render all vendor templates (title, description sections)
- `validate_template()`: Pre-validate templates before use
- `_build_context()`: Extract product fields for template variables

**Context Variables:**
- Product: product_name, product_type, sku, price, size, color, material, category
- Vendor: vendor, origin, store_name
- Derived: size_display, use_cases, technique (default: Decoupage)

**Example Usage:**
```python
template_engine = TemplateEngine()

# Simple template
result = template_engine.render(
    "{{vendor}} {{product_name}} {{size}}",
    {'vendor': 'Pentart', 'product_name': 'Acrylfarbe', 'size': '20ml'}
)
# -> "Pentart Acrylfarbe 20ml"

# Vendor YAML content templates
vendor_config = {
    'enrichment': {
        'content_templates': {
            'title': {
                'template': '{{product_name}} von {{vendor}} - {{size}}',
                'max_length': 70
            },
            'description': {
                'sections': {
                    'intro': {'template': 'Hochwertige {{material}}farbe...'},
                    'features': {'template': 'Ideal für {{use_cases}}'}
                }
            }
        }
    }
}

rendered = template_engine.render_product_content(vendor_config, product)
# -> {'title': '...', 'description': '...'}
```

### 2. EnrichmentPipeline (Orchestrator)

**Purpose:** Run all 7 enrichment steps in sequence with checkpoints

**Location:** `src/core/enrichment/pipeline.py` (312 lines)

**Architecture:**
```
EnrichmentPipeline.__init__()
├── Eager components (lightweight):
│   ├── AttributeExtractor (regex patterns)
│   ├── SEOGenerator (text transformations)
│   ├── ProductFamilyGrouper (normalization rules)
│   ├── QualityScorer (scoring formula)
│   ├── QualityGate (validation checks)
│   └── TemplateEngine (Jinja2 env)
└── Lazy components (expensive):
    ├── AIDescriptionGenerator (API connection, ~500ms)
    └── EmbeddingGenerator (768-dim model, ~2-3s)
```

**Pipeline Steps:**
1. **Extract attributes** (`_step_extract_attributes`)
   - Extracts color, size, material, category from title/description
   - Marks enrichment_status = 'extracted'
   - Checkpoint: `checkpoint_extraction.json`

2. **Apply vendor templates** (`_step_apply_templates`)
   - Renders Jinja2 templates from vendor YAML
   - Only updates if template produces content
   - Marks enrichment_status = 'templated'
   - Checkpoint: `checkpoint_templates.json`
   - **Skip condition:** No vendor_config provided

3. **AI description generation** (`_step_ai_generation`)
   - Finds products with quality score <50 or description <20 chars
   - Generates via OpenRouter API (few-shot with similar products)
   - Marks ai_generated=True, enrichment_status = 'ai_generated'
   - Checkpoint: `checkpoint_ai.json`
   - **Skip conditions:** skip_ai=True OR no openrouter_api_key

4. **Create product families** (`_step_create_families`)
   - Groups variants by normalized title (remove size/color)
   - Adds family_id, is_base_variant, variant_count
   - Checkpoint: `checkpoint_families.json`

5. **Generate embeddings** (`_step_generate_embeddings`)
   - Creates 768-dim vectors for semantic search
   - Adds embedding (as list), embedding_hash
   - Checkpoint: `checkpoint_embeddings.json`

6. **Calculate quality scores** (`_step_calculate_scores`)
   - Runs QualityScorer on all products
   - Adds data_quality_score (0-100)
   - No checkpoint (fast operation)

7. **Quality gate validation** (`_step_quality_gate`)
   - Validates batch meets minimum thresholds
   - Returns (passed: bool, report: dict)
   - **Halt condition:** passed=False AND force=False

**Skip Flags:**
- `skip_extraction`: Skip attribute extraction
- `skip_ai`: Skip AI description generation
- `skip_families`: Skip family grouping
- `skip_embeddings`: Skip embedding generation
- `skip_quality_gate`: Skip quality validation
- `force`: Bypass quality gate (proceed even if failed)

**Parameters:**
- `vendor_config`: Vendor YAML dict for templates (optional)
- `vendor_slug`: Future support for auto-loading from config/vendors/{slug}.yaml (Plan 06)
- `max_ai_products`: Limit AI generation count (useful for testing)

**Checkpoints:**
- Format: JSON files in `data/enrichment_checkpoints/`
- Naming: `checkpoint_{step_name}.json`
- Content: Full product list with all fields from that step
- Serialization: Converts numpy arrays to lists for JSON compatibility
- Loading: `pipeline.load_checkpoint('extraction')` → List[dict]

## Key Design Decisions

### 1. Lazy Loading for Expensive Components

**Problem:** AI generator needs API connection (~500ms), embedding generator loads 768-dim model (~2-3s)

**Decision:** Load on first use via @property decorators

**Impact:**
- Pipeline initializes in <100ms without AI/embeddings
- Saves time for partial re-runs (skip_ai=True, skip_embeddings=True)
- Multiple processes can share same pipeline instance without duplicate loading

**Trade-off:** Slightly more complex code, but significant performance gain for common use cases

### 2. Checkpoint After Every Step

**Problem:** Enrichment can take 10+ minutes for 1000+ products. API timeouts, memory issues, crashes lose all progress.

**Decision:** Save JSON checkpoint after each step

**Implementation:**
- `_save_checkpoint(products, 'extraction')` → `data/enrichment_checkpoints/checkpoint_extraction.json`
- Converts numpy arrays to lists for JSON compatibility
- UTF-8 encoding with ensure_ascii=False for German text

**Impact:**
- **Disk overhead:** ~1-2MB per checkpoint (100 products × 20KB each)
- **Recovery:** Can resume from any step without repeating expensive operations
- **Debugging:** Inspect intermediate state at each step

**Trade-off:** Disk I/O adds ~100-200ms per step, but worth it for long-running batches

### 3. Individual Skip Flags

**Problem:** Different scenarios need different steps:
- Developer: Testing family grouping → skip extraction (already done)
- Cost-conscious: Skip AI (save API costs)
- Performance: Skip embeddings (2-3s model load)
- Production: Skip quality gate (force mode)

**Decision:** 7 boolean flags (skip_extraction, skip_ai, skip_families, skip_embeddings, skip_quality_gate, force)

**Alternatives considered:**
- `resume_from='step_name'`: Harder to skip non-consecutive steps
- `steps_to_run=['extract', 'families']`: More typing, less discoverable

**Impact:**
- **Complexity:** 7 parameters in `run()` method
- **Flexibility:** Fine-grained control over execution
- **Discoverability:** Each flag self-documenting in IDE autocomplete

### 4. StrictUndefined Mode for Templates

**Problem:** Missing template variables cause silent rendering bugs (renders empty string or "None")

**Decision:** Jinja2 StrictUndefined mode raises exception on missing variable

**Example:**
```python
# BAD: Silently renders partial content
template = "{{vendor}} - {{missing_var}}"
result = render(template, {'vendor': 'Pentart'})
# -> "Pentart - " (missing_var disappeared)

# GOOD: Fails loud and early
env = Environment(undefined=StrictUndefined)
template = env.from_string("{{vendor}} - {{missing_var}}")
result = template.render({'vendor': 'Pentart'})
# -> jinja2.UndefinedError: 'missing_var' is undefined
```

**Impact:**
- **Vendor configs:** Must provide all template variables or templates fail
- **Error detection:** Catches typos immediately ({{colour}} vs {{color}})
- **Quality:** Forces complete, correct vendor YAML configs

## Lessons Learned

### What Went Well

1. **Clear separation of concerns:** TemplateEngine and Pipeline are independent, testable units
2. **Lazy loading pattern:** @property decorators made expensive components optional
3. **Checkpoint design:** JSON format enables easy inspection and debugging
4. **Integration tests:** Caught emoji encoding bug that unit tests wouldn't find

### What Could Be Improved

1. **Windows testing:** Should have tested on Windows earlier to catch emoji issue
2. **Progress feedback:** Pipeline steps are silent except for print statements
3. **Error handling:** No graceful degradation for API failures or missing models
4. **Documentation:** Template context variables should be documented in vendor YAML schema

### For Future Plans

1. **Test on Windows:** Always verify console output encoding
2. **Add progress bars:** Use tqdm for long-running operations
3. **Structured logging:** Replace print() with proper logger
4. **Async API calls:** Parallelize AI generation for batch efficiency
