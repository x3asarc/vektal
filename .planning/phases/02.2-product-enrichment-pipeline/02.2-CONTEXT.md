# Phase 2.2: Product Enrichment Pipeline - Context

**Gathered:** 2026-02-08
**Status:** Ready for research and planning
**Approach:** Research-first (investigate best practices before implementation decisions)

<domain>
## Phase Boundary

Transform raw scraped product data into high-quality, SEO-optimized German-language content through AI-powered enrichment. This phase sits between scraping (Phase 2.1) and database design (Phase 3), ensuring products are enriched BEFORE schema is finalized.

**In scope:**
- AI-generated German descriptions following store content framework
- Attribute extraction (colors, materials, techniques, dimensions)
- Quality scoring (0-100) based on completeness and richness
- Vector embeddings for semantic search (sentence-transformers)
- Content templating for brand consistency
- SEO optimization (meta titles, descriptions, URL handles)
- Image alt text generation following vendor patterns
- Product family grouping (variants linked to base product)

**Out of scope:**
- Scraping logic (Phase 2.1 handles this)
- Database schema design (Phase 3)
- Search UI (Phase 11)
- Real-time enrichment triggering (Phase 6 - Celery)

**Success target:** >85% average quality score across enriched products

</domain>

<research_directives>
## Critical Research Questions

### 1. Vector Embeddings Best Practices

**What to investigate:**
- What fields do sentence-transformers need for optimal semantic search?
  - Single concatenated text vs multi-field embeddings?
  - Field weighting strategies (title=2x, description=1x, tags=1.5x)?
  - Metadata filtering approaches (hybrid search patterns)
- German-language embedding models (best for e-commerce)
- Embedding dimension trade-offs (384 vs 768 vs 1024)
- When to regenerate embeddings (on every field change or only key fields)?

**Why this matters:**
Phase 11 (Search & Discovery) depends on embedding quality. Poor embeddings = poor search results. Need to know NOW what fields to include in enrichment so Phase 3 database schema captures them.

### 2. Product Enrichment Pipeline Patterns

**What to investigate:**
- Industry-standard e-commerce enrichment flows (order of operations)
- Quality scoring algorithms used in practice
  - Completeness metrics (required vs optional fields)
  - Richness metrics (description length, image count, attribute coverage)
  - SEO health metrics (meta tags, URL structure, alt text)
- Attribute extraction approaches:
  - Rule-based vs AI-based vs hybrid
  - Confidence scoring for extracted attributes
  - Validation strategies
- German SEO best practices for product content
  - Meta description length (Google Germany)
  - Keyword density guidelines
  - URL structure conventions

**Why this matters:**
We're building a production system for German e-commerce. Industry patterns prevent reinventing the wheel and ensure competitive SEO performance.

### 3. AI Description Generation Strategies

**What to investigate:**
- Template-based vs fully AI-generated descriptions
  - When to use which approach?
  - Cost-quality trade-offs
- Prompt engineering for product descriptions
  - German-language prompt patterns
  - Consistent tone/style enforcement
  - Handling technical specifications vs marketing copy
- AI model selection for German content
  - OpenRouter model comparison (cost vs quality)
  - Gemini vs Claude vs GPT for German text
  - Local LLM feasibility (no API costs)

**Why this matters:**
AI costs scale with catalog size. 10,000 products × $0.01/description = $100. Need cost-effective approach that maintains quality.

### 4. Content Template Systems

**What to investigate:**
- Jinja2 vs Mustache vs custom templating
- Variable placeholder patterns ({{product.title}} vs {title} vs $title)
- Conditional logic in templates (if/else for variant handling)
- Multi-language template management (German primary, English fallback)
- Template validation (catch missing variables before generation)

**Why this matters:**
Vendor YAML includes content templates. Need robust system that vendors can configure without breaking.

### 5. Product Family & Variant Grouping

**What to investigate:**
- Shopify variant model (parent product + variants)
- Variant detection algorithms
  - SKU pattern matching (ABC-001-RED, ABC-001-BLUE → same family)
  - Title similarity scoring
  - Attribute comparison (size/color variations)
- When to create new product vs new variant
- Family hierarchy rules (base product selection)

**Why this matters:**
Enrichment should detect variants automatically and group them correctly. Affects database schema (Phase 3) and search UX (Phase 11).

</research_directives>

<known_constraints>
## Decisions Already Made

### Technical Stack
- **Embedding model:** sentence-transformers (local, free, fast)
- **AI API:** OpenRouter (75-95% cheaper than direct APIs)
- **Language:** German-first with English fallback
- **Config format:** Vendor YAML (sections 12-22 handle enrichment)
- **Quality target:** >85% average score across catalog

### Integration Points
- **Input:** Phase 2.1 scraper output (raw product data)
- **Config source:** Vendor YAML enrichment sections
- **Output:** Enriched product data ready for Phase 3 database
- **Execution:** Celery background jobs (Phase 6 integration)

### Architectural Constraints
- Must work in Docker containers (Phase 2 infrastructure)
- Must handle 10,000+ product catalogs efficiently
- Must cache AI responses aggressively (cost control)
- Must validate before expensive AI calls (fail fast)

### SEO Requirements
- German meta descriptions (Google Germany guidelines)
- URL handles: lowercase, hyphens, no special chars
- Image alt text: descriptive, includes product name + key attributes
- Schema.org markup preparation (structured data for Google)

</known_constraints>

## Upstream and Downstream Dependencies

### Phase 2.1 (Universal Vendor Scraping Engine) - UPSTREAM
**Provides:**

- Embedding column type (vector extension?)

**Critical:** Enrichment output structure MUST be defined BEFORE Phase 3 schema design.

**Enrichment as async jobs:**
- Long-running AI calls (5-30 seconds per product)
- Batch processing (enrich 100 products at once)
- Progress tracking (10/100 products enriched)
- Retry logic (API failures, rate limits)

### Phase 11 (Search & Discovery) - DOWNSTREAM
**Embeddings power search:**
- Semantic similarity search (find similar products)
- Hybrid search (keywords + embeddings)
- Faceted filtering (attributes extracted here)

</critical_integrations>

<quality_requirements>
## Quality Scoring Framework (0-100)

**Research should investigate:**
- Completeness component (40-50 points?)
  - Required fields present (title, price, image)
  - Optional fields present (dimensions, materials, techniques)
  - Variant coverage (all size/color options captured)

- Richness component (30-40 points?)
  - Description length (minimum 100 words German?)
  - Image count (minimum 3 high-res images?)
  - Attribute count (minimum 5 extracted attributes?)

- SEO health component (20-30 points?)
  - Meta title present and optimized
  - Meta description present and within length limits
  - URL handle SEO-friendly
  - Image alt text present and descriptive

**Target:** >85% average means most products score 85-95 (high quality), with few <70 (poor quality).

</quality_requirements>
