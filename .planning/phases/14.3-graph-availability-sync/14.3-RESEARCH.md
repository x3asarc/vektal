## Executive Summary

This research establishes the technical foundation for making the knowledge graph resilient across environments. We've identified the specific probe paths for Aura (Bolt + Query API), determined the standard fallback behaviors for paused instances, and defined the metrics for sync reliability.

**Key Finding**: The Neo4j 5 Query API (HTTP) provides a robust fallback for connectivity probes that doesn't rely on the Bolt driver, enabling more reliable health checks even in minimal environments.

---

## 1. Multi-Backend Chain Strategy

We will implement a prioritized backend chain:
`aura -> local_neo4j -> local_snapshot`.

### 1.1 Aura-First Policy
- **Primary Probe (Bolt)**: Use `driver.verify_connectivity()` with a 5s timeout.
- **Secondary Probe (Query API)**: Use `httpx` to POST to `https://<dbid>.databases.neo4j.io/query`.
- **Auth**: Basic Auth (Base64 `neo4j:password`) is supported on all Aura tiers (Free/Pro/Enterprise).
- **Resume Behavior**: Connecting to a paused Aura instance triggers a resume, but it will initially timeout. The bootstrap script should detect `ServiceUnavailable` and recommend a manual resume or wait period.

### 1.2 Local Neo4j Fallback
- **Trigger**: Aura unreachable (timeout or DNS failure).
- **Action**: Attempt `docker compose up -d neo4j`.
- **Health Check**: Use `cypher-shell` inside the container:
  ```bash
  docker exec neo4j cypher-shell -u neo4j -p $PASSWORD "RETURN 1"
  ```
- **Timeouts**: 10s start period, 5s interval, 3 retries.

### 1.3 Local Snapshot (Read-Only)
- **Source**: `src/graph/local_graph_store.py` (already exists).
- **Behavior**: Parses repository filesystem and planning docs to build an in-memory graph.
- **Persistence**: `.graph/local-snapshot.json` (TTL: 15 mins memory, 60 mins disk).

## 2. Sync Reliability and Staleness

### 2.1 Sync Metadata
We will store sync status in `.graph/sync-metadata.json` to allow availability checks without a live graph.
```json
{
  "last_sync_at": "2026-02-26T12:00:00Z",
  "sync_mode": "auto",
  "last_source": "git-hook",
  "last_commit_hash": "a1b2c3d"
}
```

### 2.2 Staleness Budget
- **Calculation**: Compare `last_sync_at` with `last_commit_at`.
- **Git Probe**: `git log -1 --format=%ct` (Unix timestamp of last commit).
- **Warning Threshold**: If `last_commit_at > last_sync_at` and `delta > 1 hour`, mark as "STALE".

## 4. Neo4j Query API Reference

### 4.1 Request Format
- **URL**: `https://<database_id>.databases.neo4j.io/query`
- **Headers**:
  - `Authorization: Basic <base64>`
  - `Accept: application/json`
  - `Content-Type: application/json`
- **Body**:
  ```json
  {
    "statement": "MATCH (n:Tool) RETURN n.name LIMIT 1",
    "parameters": {}
  }
  ```

### 4.2 Response Structure (Plain JSON)
```json
{
  "data": {
    "fields": ["n.name"],
    "values": [["search_tools"]]
  }
}
```

## 5. Sentry Issue Ingestion for Autonomous Triage

To make Phase 14.3 remediation loops react to real production failures, Sentry can be used as an external issue feed.

### 5.1 Ingestion Pattern
- **Source**: Sentry REST API (project issues feed).
- **Polling**: Scheduled pull (e.g., every 5-15 minutes) plus manual pull command for operators.
- **Cursoring**: Track `last_seen` (`updated_at` + issue id) to avoid reprocessing historical issues.
- **Dedup key**: `(project, issue_fingerprint, environment, first_seen)` to avoid duplicate remediation triggers.

### 5.2 Normalization Contract
Each issue is normalized into a machine-readable envelope:
```json
{
  "source": "sentry",
  "issue_id": "1234567890",
  "title": "Neo4j connection timeout",
  "severity": "error",
  "environment": "production",
  "service": "backend",
  "normalized_category": "AURA_UNREACHABLE",
  "first_seen": "2026-03-01T10:00:00Z",
  "last_seen": "2026-03-01T10:05:00Z"
}
```

### 5.3 Routing Constraints
- Only issues mapped to Phase 14.3 graph-availability taxonomy should trigger graph remediators.
- Unknown/unmapped issue types are logged and stored, but not auto-remediated.
- Ingestion failures must never block graph bootstrap fallback selection.

### 5.4 Security and Governance
- Use auth token from environment (`SENTRY_AUTH_TOKEN`), never hardcode.
- Restrict token scope to read-only issue access.
- Keep issue payload retention minimal; only store fields needed for remediation and audit.
