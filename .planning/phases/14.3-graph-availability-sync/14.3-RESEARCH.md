## Summary

This research establishes the implementation path for Phase 14.3, focusing on a robust three-tier fallback for the knowledge graph and an autonomous self-healing layer built on the **NullClaw** architectural pattern. 

We have verified the **Neo4j Aura Query API v2** as a lightweight secondary probe that avoids driver overhead, and mapped the **Sentry API** for automated issue ingestion and normalization. The system will treat "Infrastructure Downtime" as a solvable code problem by orchestrating specialized remediation agents (`AuraRemediator`, `SnapshotRemediator`, `SyncRemediator`) through a centralized VTable-style registry.

**Primary recommendation:** Use HTTP-based Query API v2 for initial health probes as it fails faster than Bolt on paused instances. Standardize all remediation tools on the `UniversalRemediator` base class and enforce the `NanoFixerLoop` protocol.

<deep_dive_aura>
## Neo4j Aura Management (Deep Dive)

### Aura Console API v1
To programmatically manage (pause/resume) Aura instances, we must use the **Aura Console API** (separate from the Database Query API).

- **Base URL:** `https://api.neo4j.io/v1/`
- **Auth Flow:** OAuth2 Bearer Token.
  - **Token Endpoint:** `POST /oauth/token` (requires `AURA_CLIENT_ID` and `AURA_CLIENT_SECRET`).
  - **RATE LIMIT:** 15 requests/hour for token exchange. **MANDATORY:** Cache token for 1 hour.
- **Resume Endpoint:** `POST /tenants/{tenant_id}/instances/{instance_id}/resume`
- **Pause Endpoint:** `POST /tenants/{tenant_id}/instances/{instance_id}/pause`
- **Credential Generation:** Found in "Account Details" -> "API Credentials" in the Aura Console.

### Probing: Bolt vs HTTP Query API v2
| Probe Type | Timeout | Behavior on PAUSED | Recommendation |
|------------|---------|--------------------|----------------|
| **Bolt** | 2s-30s | Often hangs until 30s timeout or returns `ServiceUnavailable` | Good for deep health |
| **Query v2**| < 2s | Fails immediately with HTTP 404 or 503 | **BEST** for fast-failover |

**Implementation Strategy:** Use `httpx` to probe `/db/neo4j/query/v2` with a 2s timeout. If it fails, trigger `AuraRemediator`.
</deep_dive_aura>

## Sentry Issue Triage (Normalization)

### Ingestion Contract
- **Endpoint:** `GET /api/0/organizations/{org_slug}/issues/`
- **Query Params:** `query=is:unresolved level:error`
- **Deduplication:** Store `last_seen_id` in `.graph/sentry-cursor.json`.

### Failure Taxonomy Mapping
Sentry issues will be normalized into the Phase 14.3 categories based on keywords in `title`, `culprit`, and `tags`.

| Sentry Data | Phase 14.3 Category | Remediation Action |
|-------------|---------------------|--------------------|
| `ServiceUnavailable`, `Neo4jError` | `AURA_UNREACHABLE` | `AuraRemediator` (Resume API) |
| `docker.errors`, `ConnectionRefused` | `LOCAL_NEO4J_START_FAIL` | `DockerRemediator` (Compose restart) |
| `FileNotFound`, `json.JSONDecodeError` | `SNAPSHOT_CORRUPT` | `SnapshotRemediator` (Rebuild) |
| `TaskTimeout`, `CeleryError` | `SYNC_TIMEOUT` | `SyncRemediator` (Retry/Diagnose) |

**Pro Tip:** Use Sentry custom tags (e.g., `failure_category:infrastructure`) to simplify classification.

### Pattern 1: Aura Query API v2 Probe (Lightweight)
```python
# Fails fast on PAUSED or NETWORK_DOWN
async def probe_aura_http(hostname, user, password):
    url = f"https://{hostname}/db/neo4j/query/v2"
    auth = (user, password)
    payload = {"statement": "RETURN 1", "parameters": {}}
    async with httpx.AsyncClient(timeout=2.0) as client:
        try:
            resp = await client.post(url, auth=auth, json=payload)
            return resp.status_code == 200
        except (httpx.TimeoutException, httpx.ConnectError):
            return False
```

### Pattern 2: NullClaw Registry Lookup
```python
# src/graph/infra_probe.py
from src.graph.remediation_registry import registry

async def fix_if_needed(service):
    # DYNAMIC HEAP: No direct class imports
    tool = registry.get_tool(service)
    if tool:
        # standard protocol: Probe -> Validate -> Act -> Verify
        return await tool.fix() 
```

### Anti-Patterns to Avoid
- **Hardcoding IDs:** Never hardcode `tenant_id` or `instance_id`. Load from `.env`.
- **Direct Remediator Imports:** Violates the "Dynamic Heap" principle and makes testing/mocking harder.
- **Synchronous Probes:** Probing multiple backends should be done in parallel (`asyncio.gather`).

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: Aura Resume Timeout
**What goes wrong:** Resuming a paused Aura instance takes 2-5 minutes; a 2s probe will fail repeatedly.
**How to avoid:** `AuraRemediator` should detect "RESUMING" status via API and signal the fallback chain to wait or proceed to `local_neo4j` temporarily.

### Pitfall 2: Sentry API Rate Limits
**What goes wrong:** Excessive polling of Sentry issues triggers 429.
**How to avoid:** Use a 15-minute polling interval by default and local cursor state.

### Pitfall 3: Registry Import Cycles
**What goes wrong:** `remediation_registry` imports `UniversalRemediator` which imports `registry`.
**How to avoid:** Keep `UniversalRemediator` (base class) in `universal_fixer.py` or a dedicated `base.py`.
</common_pitfalls>
