## Implementation Decisions

### Fallback Cascade Timing
- **Aura timeout:** 2 seconds max (matches Phase 13.2 graph query timeout)
- **Aura validation:** Two-layer probe â€” Bolt connection first, Query API HTTP second
- **Local Neo4j startup:** Auto-attempt on Aura failure, 5-second startup budget via `docker compose up`
- **Snapshot fallback:** Immediate if both fail
- **Failure caching:** Yes â€” cache backend availability status (60s TTL) to skip known-dead endpoints
- **Cascade order:** Always `aura â†’ local_neo4j â†’ snapshot`, no shortcuts
- **Active Remediation Chain:** `aura -> [AuraRemediator] -> local_neo4j -> [DockerRemediator] -> local_snapshot -> [GitStalenessGuard]`

### Sync Mode Behavior
- **Default mode:** Auto-sync (background, non-blocking)
- **Auto-sync triggers:**
  - Session bootstrap (if snapshot >24h stale)
  - After code changes detected (git hook integration)
  - Periodic daemon (every 6 hours)
- **Manual sync:** Available via `gsd-tools.js sync-graph` (or `scripts/graph/sync_to_neo4j.py`)
- **Mid-session sync failure:** Emit to failure log, continue on current backend, retry next trigger
- **Blocking behavior:** Never block session start â€” degrade gracefully and sync in background

### Session Bootstrap Integration (Multi-Agent Compatibility)
- **Shared bootstrap command:** Single canonical script (`scripts/graph/bootstrap_graph_backend.py`) invoked by all agents
- **PreToolUse wiring:** Bootstrap runs before first MCP tool call via hook configuration
- **Agent compatibility:** Must work across Claude Code, Codex, and Gemini local entry points
- **Hook configuration locations:**
  - `.claude/settings.local.json` (Claude Code)
  - `.gemini/` session config (Gemini, if present)
  - `.codex/` config (Codex, if present)
- **Operator documentation:** Single runbook (`docs/ops/graph-availability-runbook.md`) explains prerequisites, fallback behavior, manual intervention

### Backend Resolution & Driver Management
- **Canonical backend enum:** `aura`, `local_neo4j`, `local_snapshot` (fixed, no other values)
- **Unified Driver Factory:** Hide sync/async complexity â€” single factory abstracts AsyncDriver (cloud) vs Driver (local)
- **Runtime manifest:** `.graph/runtime-backend.json` (generated artifact, not committed)
- **Manifest contents:** `backend`, `checked_at`, `reason`, `freshness_hours`, `is_degraded`

### Status Visibility
- **Active backend indicator:** All MCP responses include `backend_source` field (`aura`, `local_neo4j`, `snapshot`)
- **Freshness metadata:** Include `last_sync_timestamp` and `staleness_hours` in status queries
- **Stale snapshot warning:** Emit warning log if snapshot >48h old and being used
- **Status command:** `gsd-tools.js graph-status` (or `scripts/graph/graph_status.py`) shows current backend, last sync, sync mode
- **Session bootstrap log:** Print which backend resolved (visible in Claude Code session start)
- **Machine-readable contract:** Status command returns parseable JSON for CI/automation use

### Degraded Mode Guardrails
- **Snapshot mode restrictions:** Read-only queries allowed, no writes attempted
- **Staleness threshold:** If snapshot >7 days old, emit ERROR-level log recommending manual sync
- **Query compatibility:** All MCP queries must work on all three backends (schema parity enforced)
- **Data freshness disclosure:** MCP responses include `is_stale: true` flag when snapshot >24h old
- **Operation safety:** No behavior changes based on backend â€” queries return same structure regardless
- **Mutation blocking:** Governance layer blocks mutations when `backend_source == 'snapshot'`

### Security Requirements
- **Query API authentication:** Derived at runtime from `NEO4J_USER` + `NEO4J_PASSWORD` env vars
- **Username alias support:** Accept `NEO4J_USERNAME` as optional alias if needed (prefer `NEO4J_USER`)
- **No hardcoded credentials:** Scripts, docs, test fixtures must never contain credentials
- **Parameterized queries only:** All Cypher uses placeholders with `parameters` objects â€” never string concatenation
- **Sentry credentials (optional):** If Sentry integration enabled, require `SENTRY_AUTH_TOKEN`, `SENTRY_ORG_SLUG`, `SENTRY_PROJECT_SLUG` in `.env`
- **Graceful credential degradation:** Missing/invalid Sentry credentials â†’ warn + skip, do not block graph fallback

### Autonomous Self-Healing Architecture (NullClaw Pattern)

**Architectural Foundation:** Phase 14.2 established the NullClaw pattern (inspired by github.com/nullclaw/nullclaw) for hyper-modular, VTable-based remediation with explicit tool dispatch.

**Existing Implementation (Phase 14.2):**

- `src/graph/remediation_registry.py` â€” Dynamic "Heap" that auto-discovers remediation tools
- **Recursive dependency resolution:** If Neo4j fix fails â†’ auto-attempt Docker fix â†’ re-try Neo4j
- **Current tools:** DockerRemediator, RedisRemediator, GitStalenessGuard

#### Failure Resolution Loop
- **Failure log:** `FAILURE_JOURNEY.md` (existing Phase 13.2 pattern) captures all graph availability failures
- **Failure categories:**
  - `AURA_UNREACHABLE` â€” network/auth failure
  - `LOCAL_NEO4J_START_FAIL` â€” Docker/port conflict
  - `SNAPSHOT_CORRUPT` â€” invalid/missing snapshot file
  - `SYNC_TIMEOUT` â€” sync took >5 minutes
  - `BACKEND_MISMATCH` â€” schema version drift
- **New remediation tools (register in Heap):**
  - `AuraRemediator` â€” resume paused cloud instances, validate credentials
  - `SnapshotRemediator` â€” rebuild corrupted snapshots from git HEAD
  - `SyncRemediator` â€” diagnose and resolve sync pipeline failures
- **Autonomous healing agents:**
  - Bash loop: `scripts/graph/auto_heal_loop.sh` (runs every 30 min, checks failure log)
  - Healing dispatch: `scripts/graph/orchestrate_healers.py` (reads failure log, spawns specialist agents)
  - Specialist agents per category (e.g., `heal_aura.sh`, `heal_local_neo4j.sh`, `heal_snapshot.sh`)

#### Sentry-Driven Failure Intake (Optional External Signal Bridge)
- **Signal source:** Sentry project issues are treated as external failure signals, not direct remediation commands
- **Operational prerequisites:**
  - `.env` contains `SENTRY_AUTH_TOKEN` with read-only access to project issues
  - `.env` contains `SENTRY_ORG_SLUG` and `SENTRY_PROJECT_SLUG` for deterministic pull targets
  - Token scope is minimum required (read-only preferred)
- **Graceful degradation:** Missing/invalid Sentry credentials â†’ warn + skip, never block graph fallback or session bootstrap
- **Polling model:** Non-blocking pull loop (scheduled + manual trigger) fetches new/open issues
- **Cursor state:** Deduplicated local cursor state (last_checked_at, last_issue_id) prevents re-processing
- **Normalization contract:** Convert Sentry issue payloads into existing failure categories:
  - `AURA_UNREACHABLE`
  - `LOCAL_NEO4J_START_FAIL`
  - `SNAPSHOT_CORRUPT`
  - `SYNC_TIMEOUT`
  - `BACKEND_MISMATCH`
- **Routing rule:** Only mapped categories with approved severities can trigger healing agents
- **Audit trail:** Every ingested issue appends normalized records to `FAILURE_JOURNEY.md` and graph episode metadata
- **Dedup policy:** Use issue fingerprint + first_seen timestamp + environment as dedupe key
- **Manual verification:** Operator command for manual pull + triage verification (`scripts/observability/sentry_issue_puller.py --manual`)

#### Learning + Process Improvement Loop
- **Learning capture:** Every healed failure emits to `LEARNINGS.md` with:
  - Failure type + root cause
  - Solution applied
  - Recommended process change
- **Orchestration agent:** `scripts/graph/learning_orchestrator.py`
  - Runs daily
  - Parses `LEARNINGS.md` for new entries
  - Classifies learning type: `FAILSAFE_ADDITION`, `TIMEOUT_TUNING`, `FALLBACK_ORDER_CHANGE`
  - Spawns specialist agents to implement process changes
- **Process update agents:**
  - `update_timeouts.py` â€” adjust timeout constants in code
  - `add_failsafe.py` â€” generate new guardrail code
  - `update_fallback_logic.py` â€” modify cascade order/conditions
- **Verification:** All process changes run through governance gate before commit

#### Failure â†’ Learning â†’ Process Pipeline
```
1. Failure occurs â†’ logged to FAILURE_JOURNEY.md

```

#### NullClaw Design Principles (Mandatory)
- **Single source of truth:** `remediation_registry.py` is the ONLY registry â€” all tools registered, all calls go through registry
- **Hyper-modular:** Every remediation is a discrete, pluggable tool registered in the Heap
- **Predictable:** Strict protocol (Probe â†’ Validate â†’ Act â†’ Verify) â€” no ad-hoc fixes
- **Auditable:** All remediation attempts logged with outcomes for governance review
- **Recursive:** Dependency resolution is automatic (Neo4j â†’ Docker â†’ Host ENV)
- **VTable-style dispatch:** Registry maps service names to remediation tools dynamically
- **Fail-safe:** If remediation fails, system degrades gracefully (logs error, uses fallback backend)
- **Never block:** Remediation runs async â€” session starts immediately, healing happens in background
- **No direct tool imports:** All scripts must call tools via registry lookup â€” never import remediation classes directly

## Specific Ideas

- **Lightweight:** All timeouts â‰¤5s, cache aggressively, never block session start
- **Fast:** Parallel backend health checks (Aura + local Neo4j checked simultaneously)
- **High success rate:** Three-tier fallback means 99.9%+ availability (all three fail = catastrophic)
- **Autonomous healing:** No human intervention required for transient failures
- **Learning-driven:** System gets smarter with each failure category encountered
- **Self-improving:** Each failure class becomes a new process failsafe automatically
- **Build on Phase 14.2:** Extend existing NullClaw foundation (`infra_probe.py`, `universal_fixer.py`, `remediation_registry.py`)
- **Master learnings:** Root-level `LEARNINGS.md` shared across all agents (Gemini, Claude, Codex) â€” auto-injected into next session
- **No reinvention:** Use `NanoFixerLoop` protocol for all new remediation tools (DockerRemediator, RedisRemediator patterns)
- **Strategic impact:** Phase 14.3 treats "Infrastructure Downtime" as a solvable code problem, not a terminal error
- **Remediator location:** All remediation tools in `src/graph/remediators/` directory, registered via auto-discovery
- **Lightweight probes:** TCP/HTTP heartbeats avoid driver overhead, fail-fast detection
- **Git-based staleness:** Commit-hash fingerprinting for snapshot validity checks
- **CI-friendly:** All availability tests pass without requiring live Aura (use mocks/snapshots)
- **Acceptance criteria:**
  - Aura up â†’ backend selects `aura`, metadata shows `backend_source=aura`
  - Aura down, local up â†’ backend selects `local_neo4j`
  - Aura down, local down â†’ backend selects `local_snapshot`, mutations blocked

  - Sync status returns machine-readable contract with `last_checked_at`
  - Governance gate returns `GREEN` only when all fallback checks pass
  - Sentry pull returns normalized issue records (category + severity + dedupe key)
  - Ingested Sentry issues trigger remediation only for whitelisted categories
