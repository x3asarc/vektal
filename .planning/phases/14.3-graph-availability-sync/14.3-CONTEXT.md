# Phase 14.3: Graph Availability + Sync Reliability - Context

**Gathered:** 2026-03-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Infrastructure resilience layer that guarantees graph context availability through a three-tier fallback strategy (Aura → local Neo4j → snapshot) with autonomous failure resolution and learning-driven process improvements.

</domain>

<decisions>
## Implementation Decisions

### Fallback Cascade Timing
- **Aura timeout:** 2 seconds max (matches Phase 13.2 graph query timeout)
- **Local Neo4j startup:** Auto-attempt on Aura failure, 5-second startup budget
- **Snapshot fallback:** Immediate if both fail
- **Failure caching:** Yes — cache backend availability status (60s TTL) to skip known-dead endpoints
- **Cascade order:** Always `aura → local_neo4j → snapshot`, no shortcuts

### Sync Mode Behavior
- **Default mode:** Auto-sync (background, non-blocking)
- **Auto-sync triggers:**
  - Session bootstrap (if snapshot >24h stale)
  - After code changes detected (git hook integration)
  - Periodic daemon (every 6 hours)
- **Manual sync:** Available via `gsd-tools.js sync-graph` command
- **Mid-session sync failure:** Emit to failure log, continue on current backend, retry next trigger
- **Blocking behavior:** Never block session start — degrade gracefully and sync in background

### Status Visibility
- **Active backend indicator:** All MCP responses include `backend_source` field (`aura`, `local_neo4j`, `snapshot`)
- **Freshness metadata:** Include `last_sync_timestamp` and `staleness_hours` in status queries
- **Stale snapshot warning:** Emit warning log if snapshot >48h old and being used
- **Status command:** `gsd-tools.js graph-status` shows current backend, last sync, sync mode
- **Session bootstrap log:** Print which backend resolved (visible in Claude Code session start)

### Degraded Mode Guardrails
- **Snapshot mode restrictions:** Read-only queries allowed, no writes attempted
- **Staleness threshold:** If snapshot >7 days old, emit ERROR-level log recommending manual sync
- **Query compatibility:** All MCP queries must work on all three backends (schema parity enforced)
- **Data freshness disclosure:** MCP responses include `is_stale: true` flag when snapshot >24h old
- **Operation safety:** No behavior changes based on backend — queries return same structure regardless

### Autonomous Self-Healing Architecture

#### Failure Resolution Loop
- **Failure log:** `FAILURE_JOURNEY.md` (existing Phase 13.2 pattern) captures all graph availability failures
- **Failure categories:**
  - `AURA_UNREACHABLE` — network/auth failure
  - `LOCAL_NEO4J_START_FAIL` — Docker/port conflict
  - `SNAPSHOT_CORRUPT` — invalid/missing snapshot file
  - `SYNC_TIMEOUT` — sync took >5 minutes
  - `BACKEND_MISMATCH` — schema version drift
- **Autonomous healing agents:**
  - Bash loop: `scripts/graph/auto_heal_loop.sh` (runs every 30 min, checks failure log)
  - Healing dispatch: `scripts/graph/orchestrate_healers.py` (reads failure log, spawns specialist agents)
  - Specialist agents per category (e.g., `heal_aura.sh`, `heal_local_neo4j.sh`, `heal_snapshot.sh`)

#### Learning + Process Improvement Loop
- **Learning capture:** Every healed failure emits to `LEARNINGS.md` with:
  - Failure type + root cause
  - Solution applied
  - Recommended process change
- **Orchestration agent:** `scripts/graph/learning_orchestrator.py`
  - Runs daily
  - Parses `LEARNINGS.md` for new entries
  - Classifies learning type: `FAILSAFE_ADDITION`, `TIMEOUT_TUNING`, `FALLBACK_ORDER_CHANGE`
  - Spawns specialist agents to implement process changes
- **Process update agents:**
  - `update_timeouts.py` — adjust timeout constants in code
  - `add_failsafe.py` — generate new guardrail code
  - `update_fallback_logic.py` — modify cascade order/conditions
- **Verification:** All process changes run through governance gate before commit

#### Failure → Learning → Process Pipeline
```
1. Failure occurs → logged to FAILURE_JOURNEY.md
2. Auto-heal loop detects → spawns healing agent
3. Healing agent fixes → emits to LEARNINGS.md
4. Learning orchestrator detects → classifies learning type
5. Process update agent spawns → implements failsafe/improvement
6. Governance gate validates → commits if GREEN
7. Future sessions benefit from new process
```

### Claude's Discretion
- Exact timeout values (within stated ranges)
- Cache TTL tuning based on observed failure patterns
- Healing agent retry logic and backoff strategies
- Log verbosity levels per environment

</decisions>

<specifics>
## Specific Ideas

- **Lightweight:** All timeouts ≤5s, cache aggressively, never block session start
- **Fast:** Parallel backend health checks (Aura + local Neo4j checked simultaneously)
- **High success rate:** Three-tier fallback means 99.9%+ availability (all three fail = catastrophic)
- **Autonomous healing:** No human intervention required for transient failures
- **Learning-driven:** System gets smarter with each failure category encountered
- **Self-improving:** Each failure class becomes a new process failsafe automatically

</specifics>
