# Phase 2.1 Plan 11: Adaptive Learning & Success Rate Tracking Summary

**One-liner:** Self-improving scraper with metrics tracking, failure categorization, adaptive retry parameters, and automatic rediscovery triggering.

## What Was Built

### ScrapeMetrics Class
Session-based metrics tracking with categorized failure reasons:

- **Success rate calculation**: 0.0-1.0 scale, vendor-specific or overall
- **Failure categorization**: 6 types (RATE_LIMIT, TIMEOUT, SELECTOR_FAILED, NETWORK_ERROR, VALIDATION_FAILED, UNKNOWN)
- **Per-vendor statistics**: total attempts, success/fail counts, most common failure, avg retry count
- **Session reporting**: Markdown export for debugging and analysis

**Files:** `src/core/scraping/metrics.py` (295 lines)

### AdaptiveRetryEngine Class
Learns from failures and auto-adjusts retry strategies:

- **Failure-driven learning**: Different responses per failure type
  - Rate limits → Increase delay 50% (max 30s)
  - Timeouts → Increase timeout 25% (max 120s)
  - Selector failures → Enable fallback selectors after 3 failures
  - Network errors → Increase retry attempts (max 5)

- **Fallback selector chains**: Primary → learned fallback → common defaults
- **Rediscovery triggering**: When selector failures >5 in 10 attempts or success rate <50%
- **Exponential backoff with jitter**: Intelligent retry delays
- **Per-vendor learning**: Maintains separate learned parameters for each vendor

**Files:** `src/core/scraping/adaptive.py` (365 lines)

### UniversalScraper Integration
Integrated metrics and adaptive learning into existing scraper:

- **Automatic tracking**: Every scrape tracked (success/failure, duration, retry count)
- **Failure categorization**: Error messages mapped to structured failure types
- **BatchResult dataclass**: Returns success_rate, failure_breakdown, recommendations
- **Rediscovery warnings**: Logged when vendor config needs update

**Files:** `src/core/scraping/engine.py` (modified)

## Decisions Made

### 1. Session-Based Metrics (Not Persistent)
**Context:** Gap 3 requires "dynamic improvement within session"

**Chosen:** In-memory tracking, clears on restart

**Rationale:**
- User requirement: self-improving "within single session"
- Persistence adds complexity without current benefit
- Phase 14 (ML) will add long-term learning if needed

**Impact:** Fast, simple, meets stated requirement

### 2. Six Categorized Failure Types
**Context:** Need structured failure reasons for adaptive learning

**Chosen:** RATE_LIMIT, TIMEOUT, SELECTOR_FAILED, NETWORK_ERROR, VALIDATION_FAILED, UNKNOWN

**Rationale:**
- Enables targeted adaptive responses per failure type
- Covers all observed failure patterns in existing scraper
- Extensible (can add more types later)

**Impact:** Machine-learnable failure tracking

### 3. Pragmatic Adaptive Rules (Not ML)
**Context:** User wants "dynamic improvement NOW", not Phase 14 ML

**Chosen:**
- Rate limits → +50% delay
- Timeouts → +25% timeout
- Selector failures → fallback selectors
- Network errors → +1 retry attempt

**Rationale:**
- Immediate value without ML infrastructure
- Proven effective in production scraping systems
- Foundation for ML enhancement later

**Impact:** Self-improvement starts immediately

### 4. Rediscovery Thresholds
**Context:** When to recommend vendor config refresh

**Chosen:**
- Selector failures >5 in last 10 attempts
- Success rate <50% for vendor

**Rationale:**
- Catches broken selectors early
- Avoids false positives from temporary issues
- Tested against realistic failure patterns

**Impact:** Proactive config maintenance

## Technical Highlights

### Exponential Backoff with Jitter
```python
backoff = base_delay * (2 ** (attempt - 1))
jitter = ±20% random variation
max_delay = 30000ms
```

**Why:** Mimics human behavior, reduces detection risk, prevents thundering herd

### Fallback Selector Learning
```python
# Primary fails → try fallbacks → learn which works
chain = [primary, learned_successful, common_defaults...]
# Next time, prioritize learned successful fallback
```

**Why:** Adapts to vendor-specific quirks without manual config updates

### Per-Vendor Parameter Isolation
```python
# Each vendor has independent learned parameters
_vendor_params: dict[str, LearnedParams]
# Prevents one vendor's issues affecting others
```

**Why:** Vendors have different rate limits, timeouts, page structures
