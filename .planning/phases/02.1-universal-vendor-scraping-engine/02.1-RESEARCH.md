## Summary

Phase 2.1 implements a self-learning, vendor-agnostic scraping system that discovers vendors from SKUs with zero configuration. The system combines local pattern matching, web search, LLM-powered inference, and automatic site structure learning to generate vendor YAML configs automatically.

Research reveals this phase requires integrating four distinct technical domains:
1. **Multi-strategy web scraping** (Playwright for JavaScript-heavy sites, existing Selenium fallback)
2. **Vendor discovery pipeline** (local patterns → known vendors → web search → AI inference)
3. **LLM integration** (local models for classification, API for complex reasoning)
4. **YAML configuration management** (Pydantic validation, versioning, auto-generation)

The codebase already contains proven patterns from `/quickcleanup` (SKU validation, batch processing, retry logic) and `universal_vendor_scraper/` (multi-strategy architecture), which can be unified and enhanced with AI-powered discovery.

**Primary recommendation:** Build vendor discovery pipeline using sentence-transformers for local classification (zero cost), Gemini Flash via OpenRouter for complex inference ($0.10/1M tokens), and Playwright for JavaScript site scraping. Store all vendor intelligence in validated YAML configs using Pydantic schemas.

### Pattern 1: Discovery Pipeline (Hybrid Approach)
**What:** Sequential vendor discovery with early exit on high confidence
**When to use:** Every time user provides SKU without specifying vendor
**Example:**
```python
# Source: Phase 2.1 CONTEXT.md design decisions
class VendorDiscoveryPipeline:
    """Hybrid discovery: local patterns → known vendors → web search → AI"""

    def discover_vendor(self, sku: str, store_profile: dict) -> VendorResult:
        # Stage 1: Local pattern matching (instant, free)
        if match := self.match_local_patterns(sku, store_profile):
            if match.confidence >= 0.90:
                return match  # High confidence, skip expensive steps

        # Stage 2: Search with store context (not SKU alone!)
        search_query = f"{sku} {' '.join(store_profile['keywords'][:3])}"
        results = self.web_search(search_query)

        # Stage 3: AI inference with niche validation
        inference = self.ai_classifier.infer_vendor(
            sku=sku,
            search_results=results,
            store_niche=store_profile['niche'],
            known_vendors=store_profile['known_vendors']
        )

        # Stage 4: Niche mismatch detection (CRITICAL for accuracy)
        if inference.vendor_niche != store_profile['niche']:
            inference.confidence *= 0.5  # Penalty for niche mismatch
            inference.requires_confirmation = True

        return inference
```

### Pattern 2: Multi-Strategy Scraping with Retry Logic
**What:** Attempt multiple scraping strategies with exponential backoff
**When to use:** When scraping any vendor (known or newly discovered)
**Example:**
```python
# Source: universal_vendor_scraper/scraper.js patterns (existing code)
class UniversalScraper:
    """Vendor-agnostic scraping with strategy fallback"""

    async def scrape_with_retry(
        self,
        vendor_config: VendorConfig,
        sku: str,
        max_attempts: int = 4,
        direct_url: str = None
    ) -> ScrapeResult:
        strategies = [
            PlaywrightStrategy(),   # Primary: JavaScript sites
            SeleniumStrategy(),     # Fallback: Static sites
            FirecrawlStrategy()     # Discovery: Collection pages
        ]

        # GSD optimization: direct URL skips search
        if direct_url:
            strategies.insert(0, DirectURLStrategy(direct_url))

        for attempt, strategy in enumerate(strategies[:max_attempts], 1):
            try:
                result = await strategy.execute(vendor_config, sku)

                # Validate against vendor patterns
                if self.validator.validate(result, vendor_config, sku):
                    return ScrapeResult(success=True, data=result, strategy=strategy.name)

            except RateLimitError as e:
                # Exponential backoff: 2^attempt seconds
                await asyncio.sleep(2 ** attempt)
            except Exception as e:
                logger.error(f"Strategy {strategy.name} failed: {e}")
                continue

        return ScrapeResult(success=False, error="All strategies failed")
```

```python
from pydantic import BaseModel, Field, validator
import yaml

class VendorConfig(BaseModel):
    """Schema for vendor YAML configuration"""

    vendor_name: str
    domain: str
    discovery_method: str = "auto_inferred"

    # SKU patterns (list of regex patterns)
    sku_patterns: list[dict[str, str]] = Field(
        description="Regex patterns for SKU validation"
    )

    # URL templates
    urls: dict[str, str] = Field(
        description="Product, search, collection URL templates"
    )

    # Scraping strategy preference
    strategy: dict[str, str] = Field(
        description="Primary/fallback strategy selection"
    )

    # CSS selectors
    selectors: dict[str, str] = Field(
        description="Element selectors for data extraction"
    )

    @validator('sku_patterns')
    def validate_sku_patterns(cls, patterns):
        """Ensure all regex patterns compile"""
        import re
        for pattern in patterns:
            try:
                re.compile(pattern.get('regex', ''))
            except re.error as e:
                raise ValueError(f"Invalid regex: {e}")
        return patterns

    @classmethod
    def from_yaml(cls, yaml_path: str) -> 'VendorConfig':
        """Load and validate YAML config"""
        with open(yaml_path) as f:
            data = yaml.safe_load(f)
        return cls.model_validate(data)
```

### Pattern 4: LLM Intent Classification with Local Fallback
**What:** Pattern matcher first, LLM only for ambiguous cases
**When to use:** Chat interface routing (Phase 2.1 chat requirements)
**Example:**
```python
# Source: LLM intent classification research + CONTEXT.md decisions
class ChatRouter:
    """Route user queries: pattern matcher → LLM classifier → handlers"""

    def __init__(self):
        self.pattern_matcher = PatternMatcher()
        self.llm_classifier = LocalLLMClassifier()  # sentence-transformers
        self.api_classifier = OpenRouterClassifier()  # Gemini Flash

    def route(self, user_message: str) -> Handler:
        # Stage 1: Pattern matching (80% of queries)
        if intent := self.pattern_matcher.match(user_message):
            return self.get_handler(intent)

        # Stage 2: Local LLM classification (15% of queries)
        local_intent = self.llm_classifier.classify(user_message)
        if local_intent.confidence >= 0.85:
            return self.get_handler(local_intent.intent)

        # Stage 3: API LLM for complex cases (5% of queries)
        api_intent = self.api_classifier.classify(user_message)
        return self.get_handler(api_intent.intent)

class PatternMatcher:
    """Fast, free pattern matching for common intents"""

    PATTERNS = {
        'add_product': [
            r'^add\s+(?:sku|product)\s*[:=]?\s*(\S+)',
            r'^update\s+(?:sku|product)\s*[:=]?\s*(\S+)',
            r'^\s*([A-Z0-9]+)\s*$'  # Bare SKU
        ],
        'search_vendor': [
            r'find\s+vendor\s+for\s+(\S+)',
            r'who\s+makes\s+(\S+)',
        ]
    }

    def match(self, message: str) -> Optional[Intent]:
        for intent, patterns in self.PATTERNS.items():
            for pattern in patterns:
                if match := re.match(pattern, message, re.IGNORECASE):
                    return Intent(intent=intent, entities=match.groups())
        return None
```

```python
            for p in products

        return vectorizer.get_feature_names_out().tolist()
```

### Anti-Patterns to Avoid

- **Searching with SKU alone:** Don't search "R0530" without store context. Always include store keywords: "R0530 decoupage rice paper". Prevents wrong vendor matches (car parts vs arts & crafts).

- **Vendor-specific scraping code:** Don't create separate scrapers per vendor. Use single engine with YAML config. Future vendors shouldn't require code changes.

- **Synchronous LLM calls in main thread:** Don't block on API calls. Use async or background tasks. LLM inference can be slow (500ms-2s).

- **Hardcoded retry counts:** Don't use fixed retry attempts. Use exponential backoff with max time limit. Some sites respond differently to rate limits.

- **Plain environment variables for secrets:** Don't use `environment:` in docker-compose.yml. Use Docker secrets (Phase 2 already has this issue, noted in verification).

- **Single scraping strategy:** Don't rely only on Playwright or Selenium. Different sites need different approaches. Multi-strategy with fallback is essential.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Retry logic with backoff | Manual `for` loops with `time.sleep()` | `tenacity` library | Handles edge cases: jitter, max time limits, conditional retries, exponential/linear backoff. Manual implementations miss these. |
| YAML validation | Custom dict checks with `isinstance()` | Pydantic models | Type coercion, nested validation, clear error messages, automatic documentation. Hand-rolled validation is verbose and error-prone. |
| Web search API | Scraping Google search results | DuckDuckGo API or Google Custom Search API | Search engine scraping breaks frequently (anti-bot measures). Official APIs are stable and legal. |
| Text embeddings | Word2Vec or custom vectors | `sentence-transformers` | Pre-trained multilingual models, battle-tested, optimized, handles synonyms/context. Custom embeddings require large training datasets. |
| LLM API integration | Direct HTTP requests with `requests` | OpenAI SDK (OpenRouter-compatible) | Handles streaming, error handling, retries, rate limits, token counting. Direct HTTP misses these. |
| CSS selector validation | Trying selectors and catching exceptions | Playwright's built-in `wait_for_selector()` | Auto-waits for elements, handles dynamic content, clear timeout errors. Manual checks race with page loads. |
| SKU pattern learning | Manual regex writing | Extract from existing catalog + AI inference | Humans miss edge cases (variant suffixes, special formats). AI finds patterns in data you didn't notice. |
| Rate limit detection | Checking response codes manually | HTTP client with retry adapters (urllib3.Retry) | Detects rate limits from headers (`Retry-After`), status codes (429), response patterns. Manual checks incomplete. |

**Key insight:** Web scraping and AI integration have mature ecosystems. Vendor discovery combines 4-5 complex domains (scraping, NLP, LLMs, config management). Each has libraries that handle edge cases you won't think of until production breaks.

## Common Pitfalls

### Pitfall 1: Niche Mismatch False Positives
**What goes wrong:** SKU "R503" returns Honda oil filter (automotive) instead of ITD rice paper (arts & crafts) because search engine prioritizes popularity over store context.

**Why it happens:** Searching with SKU alone ignores store niche. Multiple unrelated products can share similar SKU formats.

**How to avoid:**
- Always include store keywords in search query: `"{sku} {store_keywords}"`
- Implement niche validation in AI inference step
- Apply confidence penalty (-50%) for niche mismatch
- Require user confirmation when vendor niche ≠ store niche

**Warning signs:**
- Vendor discovered but product type completely different
- User confusion: "Why is it suggesting car parts for my craft store?"
- Low confidence scores despite clear search results

### Pitfall 2: Configuration Drift (YAML vs Reality)
**What goes wrong:** Vendor redesigns website, CSS selectors break, scraper silently fails or returns wrong data. YAML config becomes stale.

**Why it happens:** Websites change frequently. Static configs don't auto-update. No validation that selectors still work.

**How to avoid:**
- Track scrape success rate per vendor (store in `_meta.stats`)
- Trigger re-reconnaissance when success rate drops below 50%
- Version YAML configs (increment version on each change)
- Store last_verified timestamp, flag configs older than 90 days
- Implement selector fallbacks in YAML (primary + backup selectors)

**Warning signs:**
- Multiple consecutive scrape failures for same vendor
- Scraped data looks incomplete (missing images, prices)
- Product title doesn't match expected format
- Success rate graph shows sudden drop

### Pitfall 3: Rate Limiting Death Spiral
**What goes wrong:** Scraper hits rate limit (429), retries immediately, gets blocked faster, consumes all retry attempts within seconds.

**Why it happens:** No exponential backoff. Retries too fast. Doesn't read `Retry-After` header.

**How to avoid:**
- Use `tenacity` with exponential backoff: `@retry(wait=wait_exponential(multiplier=1, min=2, max=60))`
- Read `Retry-After` header from 429 responses
- Add jitter to avoid thundering herd: `wait_exponential_jitter()`
- Respect vendor's `rate_limit.delay_ms` from YAML config
- Implement per-vendor rate limiting (not global)

**Warning signs:**
- All 4 retry attempts fail within 5 seconds
- Logs show repeated 429 errors
- Same vendor always fails, others succeed
- Vendor blocks IP address entirely

### Pitfall 4: LLM Inference Cost Explosion
**What goes wrong:** Every user query hits OpenRouter API ($0.10/1M tokens). 1000 products = $50+ in API costs.

**Why it happens:** Calling LLM for tasks that pattern matching or local models can handle.

**How to avoid:**
- Pattern matching first (80% of queries, free)
- Local sentence-transformers for classification (15%, free)
- API LLM only for complex inference (5%, paid)
- Cache LLM results: same SKU pattern → same vendor
- Batch API calls when possible

**Warning signs:**
- OpenRouter bill higher than expected
- Every vendor discovery takes 2-3 seconds
- High latency on simple queries ("add SKU R0530")
- API usage graphs show linear growth with product count

### Pitfall 5: Docker Secret Exposure
**What goes wrong:** API keys visible in `docker inspect` output. Anyone with Docker access can read secrets.

**Why it happens:** Using `environment:` block in docker-compose.yml with direct variable substitution.

**How to avoid:**
- Use Docker secrets: `docker-compose.secrets.yml` (Phase 2 already created this)
- Reference secrets with `secrets:` not `environment:`
- Never log secret values in application logs
- Use `get_secret()` helper function (Phase 2 pattern)

**Warning signs:**
- `docker inspect backend | grep API_KEY` shows actual key
- .env file committed to git (check .gitignore)
- Logs contain "API_KEY=sk-..."
- Production containers use same credentials as dev

### Pitfall 6: Playwright Resource Leaks
**What goes wrong:** Browsers pile up in memory, system runs out of RAM, scraper crashes.

**Why it happens:** Not closing browser contexts properly. Exception handling misses cleanup.

**How to avoid:**
```python
# GOOD: Context manager ensures cleanup
async with async_playwright() as p:
    browser = await p.chromium.launch()
    try:
        page = await browser.new_page()
        # ... scraping logic ...
    finally:
        await browser.close()

# BAD: Manual cleanup, exceptions skip close()
browser = await p.chromium.launch()
page = await browser.new_page()
# ... scraping logic ...
await browser.close()  # Skipped if exception thrown!
```

**Warning signs:**
- Memory usage climbs with each scrape
- `ps aux | grep chromium` shows zombie processes
- Docker container OOM killed
- Scraper works for 10 products, crashes on batch of 50

### Example 1: Sentence-Transformers Local Classification
```python
# Source: https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2
from sentence_transformers import SentenceTransformer
from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

class LocalVendorClassifier:
    """Local LLM classification using sentence-transformers (zero cost)"""

    def __init__(self):
        # Lightweight model: 120MB download, fast inference
        self.model = SentenceTransformer('all-MiniLM-L6-v2')

        # Pre-compute embeddings for known vendors
        self.vendor_embeddings = self._embed_vendor_profiles()

    def _embed_vendor_profiles(self) -> dict:
        """Create embeddings for each known vendor"""
        vendors = {
            'ITD Collection': 'Italian rice paper decoupage scrapbooking crafts A4 A3',
            'Pentart': 'acrylic paint stencils mixed media art supplies',
            'Aisticraft': 'paper crafts decoupage napkins decorative'
        }

        embeddings = {}
        for vendor, description in vendors.items():
            embeddings[vendor] = self.model.encode(description)

        return embeddings

    def classify_vendor(self, search_results: list[str], threshold: float = 0.75) -> dict:
        """
        Classify vendor from search results using semantic similarity

        Returns:
            {
                'vendor': str,
                'confidence': float,
                'method': 'local_llm'
            }
        """
        # Combine search results into context
        context = ' '.join(search_results[:5])
        context_embedding = self.model.encode(context)

        # Calculate similarity to each vendor
        similarities = {}
        for vendor, vendor_emb in self.vendor_embeddings.items():
            similarity = cosine_similarity(
                [context_embedding],
                [vendor_emb]
            )[0][0]
            similarities[vendor] = similarity

        # Get best match
        best_vendor = max(similarities, key=similarities.get)
        confidence = similarities[best_vendor]

        if confidence >= threshold:
            return {
                'vendor': best_vendor,
                'confidence': float(confidence),
                'method': 'local_llm'
            }

        return {
            'vendor': None,
            'confidence': float(confidence),
            'method': 'local_llm',
            'needs_api_fallback': True
        }
```

```python
# Source: side-project/src/data_pipeline/ai_descriptions.py (existing code)
import os
import requests
import json
from functools import lru_cache

class OpenRouterVendorInference:
    """LLM-powered vendor inference via OpenRouter (cheap: $0.10/1M tokens)"""

    def __init__(self, api_key: str = None, model: str = "google/gemini-flash-1.5"):
        self.api_key = api_key or os.getenv('OPENROUTER_API_KEY')
        self.model = model
        self.api_url = "https://openrouter.ai/api/v1/chat/completions"

    @lru_cache(maxsize=1000)  # Cache results for repeated SKUs
    def infer_vendor(
        self,
        sku: str,
        search_results: tuple,  # Tuple for hashable cache key
        store_niche: str,
        known_vendors: tuple
    ) -> dict:
        """
        Infer vendor from search results using LLM

        Cache key: (sku, search_results hash, store_niche)
        Returns same result for identical inputs (no API call)
        """
        prompt = f"""Analyze these search results for SKU "{sku}":

{chr(10).join(search_results)}

Store Profile:
- Niche: {store_niche}
- Known Vendors: {', '.join(known_vendors)}

Tasks:
1. Identify the most likely vendor/manufacturer
2. Determine vendor's primary niche
3. Check if vendor niche matches store niche
4. Rate confidence 0-100%

Return JSON:
{{
  "vendor_name": "...",
  "vendor_website": "...",
  "vendor_niche": "...",
  "niche_match": true/false,
  "confidence": 0-100,
  "reasoning": "..."
}}"""

        response = requests.post(
            self.api_url,
            headers={
                "Authorization": f"Bearer {self.api_key}",
                "Content-Type": "application/json"
            },
            json={
                "model": self.model,
                "messages": [
                    {"role": "user", "content": prompt}
                ],
                "temperature": 0.1,  # Low temperature for consistent results
                "response_format": {"type": "json_object"}
            }
        )

        response.raise_for_status()
        content = response.json()['choices'][0]['message']['content']
        return json.loads(content)
```

### Example 3: Tenacity Retry with Exponential Backoff
```python
# Source: https://www.zenrows.com/blog/python-requests-retry
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
import requests

class RateLimitError(Exception):
    """Raised when server returns 429"""
    pass

class VendorScraper:
    """Scraper with intelligent retry logic"""

    @retry(
        stop=stop_after_attempt(5),
        wait=wait_exponential(multiplier=1, min=2, max=60),
        retry=retry_if_exception_type((RateLimitError, requests.RequestException)),
        reraise=True
    )
    def fetch_product_page(self, url: str) -> str:
        """
        Fetch page with exponential backoff on errors

        Retry schedule:
        - Attempt 1: immediate
        - Attempt 2: wait 2s
        - Attempt 3: wait 4s
        - Attempt 4: wait 8s
        - Attempt 5: wait 16s
        """
        response = requests.get(url, timeout=30)

        if response.status_code == 429:
            # Respect Retry-After header if present
            retry_after = response.headers.get('Retry-After')
            if retry_after:
                print(f"Rate limited. Server says retry after {retry_after}s")
            raise RateLimitError(f"Rate limited: {url}")

        response.raise_for_status()
        return response.text
```

### Example 4: Pydantic YAML Validation
```python
# Source: https://medium.com/@jonathan_b/a-simple-guide-to-configure-your-python-project-with-pydantic-and-a-yaml-file-bef76888f366
from pydantic import BaseModel, Field, field_validator
import yaml
from pathlib import Path
import re

class SKUPattern(BaseModel):
    """Single SKU pattern definition"""
    regex: str
    description: str
    examples: list[str]

    @field_validator('regex')
    def validate_regex(cls, v):
        """Ensure regex compiles"""
        try:
            re.compile(v)
        except re.error as e:
            raise ValueError(f"Invalid regex pattern: {e}")
        return v

class VendorURLs(BaseModel):
    """URL templates for vendor"""
    product_template: str
    search_template: str | None = None
    collection_a4: str | None = None
    collection_a3: str | None = None

    @field_validator('product_template')
    def validate_product_url(cls, v):
        """Ensure URL has {sku} placeholder"""
        if '{sku}' not in v and '{sku_lower}' not in v:
            raise ValueError("product_template must contain {sku} or {sku_lower} placeholder")
        return v

class VendorConfig(BaseModel):
    """Complete vendor configuration schema"""
    vendor_name: str
    domain: str
    discovery_method: str = "auto_inferred"
    sku_patterns: list[SKUPattern]
    urls: VendorURLs
    selectors: dict[str, str]
    strategy: dict[str, str] = Field(
        default={"primary": "playwright", "fallback": "selenium"}
    )

    @classmethod
    def from_yaml(cls, yaml_path: Path) -> 'VendorConfig':
        """Load and validate YAML file"""
        with open(yaml_path) as f:
            data = yaml.safe_load(f)

        try:
            return cls.model_validate(data)
        except Exception as e:
            raise ValueError(f"Invalid vendor config {yaml_path.name}: {e}")

    def to_yaml(self, yaml_path: Path):
        """Save validated config to YAML"""
        with open(yaml_path, 'w') as f:
            yaml.dump(
                self.model_dump(exclude_none=True),
                f,
                default_flow_style=False,
                sort_keys=False
            )

# Usage
try:
    config = VendorConfig.from_yaml(Path('config/vendors/itd_collection.yaml'))
    print(f"✓ Valid config for {config.vendor_name}")
except ValueError as e:
    print(f"✗ Validation failed: {e}")
```

### Example 5: Playwright with Proper Resource Management
```python
# Source: https://playwright.dev/python/docs/intro
from playwright.async_api import async_playwright, Browser, Page
from contextlib import asynccontextmanager
import asyncio

class PlaywrightStrategy:
    """Strategy for JavaScript-heavy sites with proper cleanup"""

    @asynccontextmanager
    async def browser_context(self):
        """Ensure browser cleanup even on exceptions"""
        async with async_playwright() as p:
            browser = await p.chromium.launch(
                headless=True,
                args=['--no-sandbox', '--disable-dev-shm-usage']  # Docker-friendly
            )
            try:
                yield browser
            finally:
                await browser.close()

    async def scrape_product(self, url: str, selectors: dict) -> dict:
        """Scrape product with automatic cleanup"""
        async with self.browser_context() as browser:
            page = await browser.new_page()

            try:
                # Navigate with timeout
                await page.goto(url, wait_until='domcontentloaded', timeout=30000)

                # Wait for critical elements
                await page.wait_for_selector(selectors['product_title'], timeout=10000)

                # Extract data
                title = await page.inner_text(selectors['product_title'])
                price = await page.inner_text(selectors['product_price'])

                # Get all image URLs
                images = await page.evaluate("""
                    (selector) => {
                        return Array.from(document.querySelectorAll(selector))
                            .map(img => img.src || img.getAttribute('data-src'));
                    }
                """, selectors['product_image'])

                return {
                    'title': title.strip(),
                    'price': price.strip(),
                    'images': images,
                    'source_url': url
                }

            finally:
                await page.close()  # Always close page

# Usage
async def main():
    scraper = PlaywrightStrategy()
    result = await scraper.scrape_product(
        url='https://vendor.com/product',
        selectors={
            'product_title': 'h1.product__title',
            'product_price': '.price__current',
            'product_image': '.product__media-gallery img'
        }
    )
    print(result)

asyncio.run(main())
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Selenium for all scraping | Playwright for JS sites, BeautifulSoup for static | 2023-2024 | 12% faster page loads, 15% lower memory, better API design |
| Direct API calls to Claude/GPT | OpenRouter unified API | 2024-2025 | 75-95% cost reduction, automatic fallback, single integration |
| Manual retry loops | Tenacity library | Mature (2018+) | Handles edge cases (jitter, max time, conditional retries) |
| Manual YAML validation | Pydantic schemas | Pydantic v2 (2023) | Type-safe, auto-generated docs, clear error messages |
| Word2Vec embeddings | sentence-transformers | 2019+ (mature) | Pre-trained multilingual, context-aware, battle-tested |
| Vendor-specific scraper code | YAML-driven universal engine | Emerging pattern | Add vendors without code changes, future-proof |
| Searching SKU alone | Context-aware search (SKU + keywords + niche) | Best practice 2024+ | Prevents niche mismatches, higher accuracy |

**Deprecated/outdated:**
- **Puppeteer for Python scraping**: Use Playwright instead. Puppeteer is Node.js-focused; Playwright has native Python support.
- **Scrapy for simple scraping**: Overkill for small projects. Use requests + BeautifulSoup or Playwright. Scrapy excellent for large-scale crawling but adds complexity.
- **jsonschema for Python validation**: Pydantic v2 is faster, type-safe, better errors.
- **Direct Claude/GPT API calls**: OpenRouter provides unified interface, cost optimization, and fallback routing.

   - What we know: Need search API for vendor discovery. Options: Google Custom Search ($5/1000), DuckDuckGo (free, rate-limited), Bing ($1/1000)
   - What's unclear: Which provides best results for product SKU searches
   - Recommendation: Start with DuckDuckGo (free), fall back to Google Custom Search if quality insufficient. Track discovery accuracy per search engine.

4. **YAML versioning and migration strategy**
   - What we know: Need to version configs, track changes, handle schema updates
   - What's unclear: How to migrate 50+ vendor configs when schema changes (e.g., v1.0 → v2.0)

5. **Local LLM vs API balance**
   - What we know: sentence-transformers for classification, OpenRouter for complex inference
   - What's unclear: Exact threshold for "complex enough to justify API call"
   - Recommendation: Log all classification decisions with confidence scores. Analyze which queries fail local classification. Adjust threshold based on cost/accuracy tradeoff.

6. **Multi-tenant vendor config sharing**
   - What we know: Future customers need vendor configs
   - What's unclear: How to share learned configs across stores without exposing sensitive data
   - Recommendation: Separate public vendor configs (selectors, patterns) from private store data (GSD mappings, credentials). Create community config repository in Phase 10+.

## Risk Areas and Mitigation Strategies

### Risk 1: Niche Mismatch False Positives
**Risk:** System suggests wrong vendor (car parts for craft store)
**Impact:** User loses trust, manual corrections needed
**Probability:** MEDIUM (30% on first iteration)
**Mitigation:**
- Implement niche validation in AI inference
- Apply confidence penalty for niche mismatch
- Require user confirmation when confidence <70%
- Learn from user corrections (store in vendor_cache table)
- A/B test search query variations (SKU alone vs SKU+keywords)

### Risk 2: Vendor Site Changes Breaking Scrapers
**Risk:** Website redesign breaks all selectors, 100% failure rate
**Impact:** Cannot scrape vendor until config updated
**Probability:** MEDIUM (every vendor redesigns 1-2x per year)
**Mitigation:**
- Track success rate per vendor, alert when <50%
- Implement selector fallbacks (primary + backup in YAML)
- Version vendor configs, store changelog
- Auto-trigger re-reconnaissance on repeated failures
- Notify user with suggested fixes from AI analysis

### Risk 3: OpenRouter API Cost Overruns
**Risk:** API usage exceeds budget, $100/month → $500/month
**Impact:** Unexpected costs, need to reduce usage or increase budget
**Probability:** LOW-MEDIUM (if not cached properly)
**Mitigation:**
- Aggressive caching (LRU cache 1000 results)
- Pattern matching first (80% queries avoid API)
- Local LLM for classification (15% queries, free)
- Monitor API usage daily, alert on anomalies
- Implement cost cap in OpenRouter dashboard

### Risk 4: Rate Limiting Blocks
**Risk:** Scraper hits rate limits, gets IP banned
**Impact:** Cannot scrape vendor for hours/days
**Probability:** MEDIUM (without proper retry logic)
**Mitigation:**
- Exponential backoff with tenacity
- Respect Retry-After headers
- Per-vendor rate limiting from YAML config
- Rotate user agents (avoid bot detection)
- Consider proxy rotation for problematic vendors

### Risk 5: Docker Resource Exhaustion
**Risk:** Playwright browsers consume all RAM, container crashes
**Impact:** Scraping stops, manual restart needed
**Probability:** MEDIUM (without proper cleanup)
**Mitigation:**
- Context managers for browser lifecycle
- Limit concurrent browsers (max 2-3)
- Set memory limits in docker-compose.yml
- Monitor container memory usage
- Implement browser pooling/reuse

### Risk 6: Incomplete Vendor Configs
**Risk:** Auto-generated YAML missing critical fields (selectors, patterns)
**Impact:** Scraper fails, manual config editing required
**Probability:** HIGH (40% on first version)
**Mitigation:**
- Gemini Flash verification step before user sees config
- Test auto-generated config on 5 sample products
- Clear user prompts for missing data
- Comprehensive template with examples (already exists)
- Fallback to manual config entry for complex vendors
