## The Problem That Prompted This Phase

**Original Issue**: `src/core/image_scraper.py` doesn't do strict SKU matching - it just clicks the first search result, causing incorrect product images to be scraped.

**User's Manual Workaround**: Created `/quickcleanup` folder with:
- Strict SKU pattern validation
- Batch processing with retry logic
- GSD optimization (pre-mapped SKU→URL)
- Firecrawl integration (manual) for direct product page discovery

**Stage 1.5: Vendor Disambiguation Strategies**

**Strategy B: SKU Pattern Matching**
```python
# Learn SKU patterns from existing products
store_sku_patterns = {
    "ITD Collection": r"^R\d{4}[A-Z]?$",
    "Pentart": r"^P-\d{5}$",

# Check if new SKU matches known pattern

    if re.match(pattern, sku):

```

**Strategy C: Progressive Enhancement**
```python
# Search with increasing specificity
search_queries = [
    f"{sku} {vendor}",           # Try known vendors first
    f"{sku} {store_keywords}",   # Add store context
    f"{barcode} {store_niche}",  # Try barcode with niche
    f"{sku}",                    # Fallback: SKU alone
]

for query in search_queries:
    results = web_search(query)
    if high_confidence_match(results):
        break  # Stop when we find a clear winner
```

**Strategy D: Niche Filtering**
```python
# Reject vendors from incompatible niches
niche_compatibility = {
    "arts_and_crafts": ["arts", "crafts", "hobby", "scrapbooking"],
    "automotive": ["car", "auto", "vehicle", "parts"],
    "electronics": ["electronic", "resistor", "circuit"],
}

def is_compatible_niche(vendor_niche, store_niche):
    if vendor_niche == store_niche:
        return True  # Perfect match

    # Check for related niches
    compatible = niche_compatibility.get(store_niche, [])
    return any(keyword in vendor_niche for keyword in compatible)
```

## First-Time User Onboarding (Critical for Accuracy!)

### Adaptive Onboarding Strategy

**Core Principle**: Use catalog data when available, ask questions when not

```python
# Adaptive approach based on catalog size
catalog_size = len(shopify_products)

if catalog_size >= 50:
    # RICH DATA: Full store intelligence from catalog
    approach = "catalog_primary"
    questionnaire = "minimal"  # Just confirm findings
    confidence = "high"

elif catalog_size >= 10:
    # SOME DATA: Hybrid approach
    approach = "hybrid"
    questionnaire = "moderate"  # Fill gaps
    confidence = "medium"

else:
    # LITTLE/NO DATA: Questionnaire critical
    approach = "questionnaire_primary"
    questionnaire = "full"  # Detailed input needed
    confidence = "low"
```

**Source of Truth Priority**:

**Large Catalog (50+ products)** - Trust the data:
```
Vendor Discovery Priority:
1. Catalog SKU patterns (learned from 247 existing products)
2. Catalog keywords (extracted from titles/descriptions)
3. Catalog vendor field (existing vendor relationships)
4. Questionnaire input (fallback/override)

Example:
  User adds SKU "R0530"
  → Check: Do we have "R####" pattern in catalog? YES (127 ITD products)
  → Infer: This is likely ITD Collection
  → Confidence: 95% (no questionnaire needed)
```

**Small Catalog (10-49 products)** - Blend both:
```
Vendor Discovery Priority:
1. Questionnaire niche + keywords (user knows their business)
2. Catalog patterns (limited but useful)
3. Hybrid scoring (combine both signals)

Example:
  User adds SKU "R0530"
  → Catalog: Only 5 products, unclear patterns
  → Questionnaire: "I sell decoupage supplies"
  → Search: "R0530 decoupage"
  → Confidence: 75% (hybrid approach)
```

**Empty/New Store (<10 products)** - Ask the user:
```
Vendor Discovery Priority:
1. Questionnaire input (PRIMARY source of truth)
2. User's stated niche + keywords
3. Catalog will become primary as it grows

Example:
  User adds SKU "R0530"
  → Catalog: Empty or too small
  → Questionnaire: "Arts & Crafts, keywords: decoupage, rice paper"
  → Search: "R0530 decoupage rice paper"
  → Confidence: 70% (questionnaire-driven)
```

```python
            'niche': store_profile['questionnaire']['niche'],  # Trust user input

            'known_vendors': [],  # None yet
            'niche': store_profile['questionnaire']['niche'],
            'confidence': 0.70,
            'source': 'questionnaire'

    if context['source'] == 'catalog_analysis':
```

## Universal Vendor-Agnostic Scraping (Original Vision)

### Core Principle
**Vendor-agnostic scraping logic** that learns/adapts to ANY vendor, not just the 5 currently supported (ITD Collection, Pentart, Aisticraft, FN Deco, Paper Designs).

### Key Requirements

**1. Universal Scraping Engine**
- ONE scraping algorithm that works for all vendors
- Vendors don't require custom code - only configuration
- System "learns" vendor patterns through configuration
- Future customers with completely different vendors work seamlessly

**2. Multi-Strategy Framework**
- **Playwright**: JavaScript-heavy sites, SPAs, dynamic content
- **Selenium**: Static HTML, reliable for simple sites
- **Firecrawl**: Site discovery, collection page crawling, direct product page extraction
- **Intelligent Selection**: System chooses best strategy automatically OR uses them in conjunction

**3. Firecrawl Discovery Pattern** (User's Manual Process to Automate)
```
1. Give Firecrawl a vendor's collection page (e.g., ITD A4 collection)
2. Firecrawl crawls ALL product pages within that collection
3. Extracts direct product page URLs
4. System matches URLs to SKUs from user's input list
5. Scraper uses direct links (GSD optimization) = MUCH FASTER
```

**4. Vendor Configuration via YAML**
- Each vendor gets ONE YAML file: `config/vendors/{vendor_name}.yaml`
- YAML contains:
  - SKU patterns (regex for validation)
  - URL templates (collection pages, search patterns)
  - CSS selectors (product cards, images, titles)
  - Scraping strategy preferences (playwright, selenium, firecrawl)
  - GSD mappings (optional pre-mapped SKU→URL)
  - Rate limiting rules
- **Benefits**:
  - Easy to add new vendors without code changes
  - Future customers can add their own vendors
  - Patterns from existing vendors inform new vendor setup

**5. Best of All Approaches**

```bash
shopify-tools images backup --vendor ITD

# Apply scraped images to Shopify

```

- Shopify API client (existing `ShopifyClient`)

## Lessons from `/quickcleanup`

### What Worked Brilliantly
1. ✅ **Strict SKU validation** - Prevented A3/A4 mixups
2. ✅ **Batch processing** - 50 SKUs per batch was optimal
3. ✅ **Retry logic** - 4 attempts with different strategies
4. ✅ **GSD optimization** - Pre-mapped URLs = 10x faster
5. ✅ **Safe backups** - CSV + local downloads before apply
6. ✅ **Comprehensive tracking** - Every scrape result logged

### What Was Painful
1. ❌ **Manual Firecrawl step** - Had to use website, export, manually integrate
2. ❌ **Vendor-specific code** - ITD Collection logic can't be reused for Pentart
3. ❌ **Slow initial discovery** - Searching each SKU one-by-one is slow
4. ❌ **No unified CLI** - Separate Node.js + Python scripts

### Patterns to Replicate for ALL Vendors
- SKU pattern validation (but configurable per vendor)
- Size inference from SKU structure (ITD: "L" suffix = A3)
- Collection page discovery → product page extraction
- Direct URL matching for known products
- Backup before destructive operations

## Success Criteria (What Must Be TRUE)

**For Self-Learning Discovery** (NEW - The "Holy Grail"):
1. User provides ONLY a SKU/barcode → System infers vendor automatically (confidence >70%)
2. System visits vendor site and learns structure WITHOUT human configuration
3. YAML config file auto-generates from learned patterns (SKU format, URLs, selectors)
4. B2B-only vendors detected and alternative workflow suggested
5. Vendor confidence score increases with each successful scrape
6. Site structure changes detected and YAML auto-updates
7. User reaction: "Oh my goodness! I can't believe it just did that!"

**For Niche Safety** (CRITICAL - Added 2026-02-08):
1. Onboarding questionnaire builds store profile (niche, keywords, vendor scope)
2. Store profile auto-analyzes existing Shopify catalog for validation
3. SKU search includes store context keywords (not just SKU alone)
4. Vendor niche mismatch detected and rejected automatically (focused mode)
5. User confirmation required when confidence <70% or niche mismatch (flexible mode)
6. System learns from existing vendors and SKU patterns in catalog
7. Cross-niche products flagged for review: "Honda oil filter in Arts & Crafts store?"

**For Universal Engine**:
1. User can add a NEW vendor by creating a YAML file without touching code (OR system auto-generates it)
2. System automatically chooses best scraping strategy for each vendor
3. Firecrawl discovery works for vendors with collection pages
4. SKU validation prevents mismatches (like A3/A4 confusion)
5. Batch processing with retry logic works for all vendors
6. Multiple vendor formats unified into single scraping algorithm

**For Performance**:
1. GSD optimization (direct URLs) makes scraping 10x faster than search-based
2. Retry logic achieves >80% success rate (vs 65% in `/quickcleanup`)
3. Parallel batch processing (multiple vendors simultaneously)
4. Web search + AI inference completes in <5 seconds
5. Site reconnaissance completes in <30 seconds for new vendor

**For User Experience**:
1. Zero-configuration: `shopify-tools discover-product R0530` (ONE command)
2. System explains what it's doing in real-time: "Searching web... Found ITD Collection... Learning site structure..."
3. Dry-run mode shows what WOULD happen before applying
4. Backup/restore workflow prevents data loss
5. Progress tracking shows current batch, SKU, and strategy being used
6. User can review and approve auto-generated YAML before first scrape

**For Future Customers**:
1. NO documentation needed for adding vendors (system figures it out)
2. Example YAML configs exist but are optional (for manual override)
3. No Bastelschachtel-specific logic in core engine
4. Works with ANY vendor worldwide, any language, any SKU format

## Implementation Decisions (Captured 2026-02-08)

The following decisions were captured during `/gsd:discuss-phase 2.1` session:

### Discovery Strategy

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Strategy sequencing** | Hybrid | Local patterns first → known vendors → web search → AI. Speed when confident, accuracy when uncertain. |
| **Store data priority** | Catalog-first | Full Shopify store analysis at signup. Use existing catalog patterns before web search. |
| **User input handling** | Optional acceleration | User CAN provide extra info (vendor name, keywords) to speed up discovery, but not required. |

### Chat Interface Architecture

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Routing approach** | Pattern matcher → LLM classifier | 80% of queries hit pattern matching (free, fast). Only 20% need LLM for understanding. |
| **LLM for classification** | Gemini Flash via OpenRouter | $0.10/1M tokens. Classifies intent, routes to handlers. LLM decides WHERE not HOW. |
| **Response generation** | Handlers, not LLM | Handlers generate responses. LLM only for truly ambiguous cases. |

### AI/LLM Integration

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **High-volume ops** | Local models | sentence-transformers for embeddings/classification. Zero cost per query. |
| **Caching** | Aggressive | Same query = cached result. Patterns learned and stored. |
| **API fallback** | When local fails | Only call external API when local models don't understand. |
| **Complex reasoning** | Claude Sonnet | For site structure learning (rare, worth paying for accuracy). |

### YAML Configuration

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Generation approach** | Auto-generate with verification | System generates complete YAML from reconnaissance. User reviews before first use. |
| **Template created** | `config/vendors/_template.yaml` | 700+ line comprehensive template with 22 sections covering scraping AND enrichment. |
| **Versioning** | Automatic with changelog | Versions increment on change. Old versions backed up. Changelog tracks what changed. |

### Verification Layer

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Verification method** | LLM-powered | Gemini Flash validates URLs, selectors, patterns before user sees config. |
| **UX during verification** | "Verifying results..." | Simple spinner with status updates. |
| **On issues found** | Interactive prompts | User gets specific questions to clarify/confirm. Responses stored in YAML for audit. |
| **On critical failure** | Block until resolved | User must address critical issues before proceeding. |

### Confirmation UX

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Default detail level** | Minimal | "Found ITD Collection. Correct?" - quick, non-intrusive. |
| **Detailed view** | User preference (deferred) | Settings toggle for detail level deferred to user settings phase. |
| **Confidence thresholds** | 90%+ auto, 70-90% confirm, <70% require selection | Standard three-tier approach. |

### Enrichment

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Enrichment config location** | In vendor YAML | Sections 12-22 of template cover keywords, tags, content templates, SEO, attributes. |
| **Enrichment pipeline location** | New Phase 2.2 | Dedicated phase for AI enrichment, separate from scraping engine. |
| **Vendor keywords** | Auto-applied to products | When product assigned to vendor, it inherits vendor keywords, tags, patterns. |

### Deferred Decisions

| Topic | Deferred To | Notes |
|-------|-------------|-------|
| Shopify Sidekick integration | Phase 10 | Research API availability during chat interface phase. |
| User preference toggles | Settings phase | Detail level, confirmation preferences, etc. |
| Semantic search | Phase 11 | Uses embeddings generated in Phase 2.2. |
| Product chatbot | Phase 10 | Uses search under the hood. |
| Vertex AI integration | Phase 11 or 14 | For premium semantic search. |
