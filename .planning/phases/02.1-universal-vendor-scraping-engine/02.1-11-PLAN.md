<task type="auto">
  <name>Task 2: Create adaptive retry engine</name>
  <files>src/core/scraping/adaptive.py, src/core/scraping/__init__.py</files>
  <action>
Create `src/core/scraping/adaptive.py` with AdaptiveRetryEngine:

1. **AdaptiveRetryEngine class**:
   - Takes ScrapeMetrics instance for failure analysis
   - Maintains per-vendor learned parameters

2. **learn_from_failure() method**:
   - Input: vendor_name, failure_reason, current_params
   - Update learned parameters based on failure type:
     - RATE_LIMIT: Increase delay by 50%, max 30s
     - TIMEOUT: Increase timeout by 25%, max 120s
     - SELECTOR_FAILED: Flag for selector re-discovery
     - NETWORK_ERROR: Increase retry attempts

3. **get_retry_params() method**:
   - Input: vendor_name, attempt_number
   - Return dict with:
     - delay_ms: Learned or default delay
     - timeout_ms: Learned or default timeout
     - max_attempts: Based on failure history
     - use_fallback_selectors: True if primary failing

4. **should_trigger_rediscovery() method**:
   - Input: vendor_name
   - Return True if:
     - SELECTOR_FAILED count > 5 in last 10 attempts
     - Success rate < 50% in session
   - Signals that vendor config needs update

5. **get_fallback_selector_chain() method**:
   - Input: vendor_name, selector_type (title, images, etc.)
   - Return list of selectors to try in order
   - Primary from config, then common defaults
   - Learn which fallback works best

6. **_calculate_backoff() method**:
   - Exponential backoff with jitter
   - Base on failure history
   - Cap at reasonable maximum (30s)

7. **reset_vendor_learning() method**:
   - Clear learned parameters for a vendor
   - Called after config update/re-discovery
