# Phase 13.2: Oracle Framework Reuse - PRE-CONTEXT SCOPE

**Gathered:** 2026-02-16  
**Purpose:** Separate shared Oracle framework enablement from Phase 13.1 enrichment implementation.

<hard_facts>
## Hard Facts
- Phase goal from roadmap: create one reusable Oracle framework for cross-phase verification use.
- Dependency constraints:
  - Phase 13 established execution verification and governance contracts.
  - Phase 13.1 requires content/visual verification for enrichment arbitration.
- Current project state facts:
  - Oracle-like execution verification already exists in governance paths.
  - Content and visual verification are emerging needs across enrichment, resolution, and future optimization phases.
  - Duplicated verifier logic would create drift and policy inconsistency.
</hard_facts>

<portfolio_snapshot>
## Portfolio Snapshot
- Completed phases: 1 through 13 complete.
- Active phase: 13.1 (enrichment v2, semantic-retrieval-first scope).
- Planned next phases: 13.2 (oracle reuse), then 14 and 15.
- Reuse opportunity:
  - Phase 13: execution oracle patterns.
  - Phase 13.1: content/visual oracle requirements.

</portfolio_snapshot>

<direction_options>
## Direction Options (Explicit)
- A) Shared Oracle Core + Adapters (Recommended)
  - Scope now: define one orchestration contract + adapters (execution/content/visual/policy).
  - Defers to later: autonomous self-healing behavior expansion.
  - Hard-fact basis: existing verifier needs already span multiple phases.
  - Tradeoff/risk: small upfront framework cost.
- B) Phase-local Verifiers
  - Scope now: each phase builds bespoke verifier logic.
  - Defers to later: cross-phase unification.
  - Hard-fact basis: fastest local delivery.
  - Tradeoff/risk: rapid drift, duplicate policy handling, inconsistent telemetry.
- C) Execution-only Oracle
  - Scope now: keep only apply/write verification reusable.
  - Defers to later: content/visual/policy adapters.
  - Hard-fact basis: already partially implemented in Phase 13.
  - Tradeoff/risk: leaves 13.1 and 14 with parallel verifier implementations.
</direction_options>

<recommendation>
## Recommendation
- Recommended option: A
- Why (project-wide): delivers one stable verifier substrate and avoids fragmented second-opinion implementations.
- Dependency impact: minor, with high downstream simplification.
- Locked by user directive: yes.
</recommendation>
