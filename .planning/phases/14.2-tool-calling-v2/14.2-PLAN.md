# Phase 14.2: Tool Calling 2.0 Integration
**Status**: PLANNED
**Depends on**: Phase 14.1 (complete 2026-02-23)
**Branch**: `claude/tool-calling-rag-integration-1TpRu`
**Tagged**: `[developer-facing]`

## Goal

Integrate Anthropic's "Tool Calling 2.0" patterns into the existing RAG/Graphiti/Neo4j
workflow and the assistant tool pipeline. The result is lower token consumption, higher tool
accuracy, and a self-describing tool registry discoverable through the knowledge graph itself.

Source concepts: AI Jason — "Anthropic killed Tool Calling" (2026-02 release)

---

## Why This Matters for This Codebase

Phase 14.1 built the active RAG layer:
- MCP server with 3 tools (`query_graph`, `get_dependencies`, `retrieve_intent`)
- Semantic cache (0.92 cosine threshold)
- Search-then-Expand bridge (max 5 nodes, depth 2, 8192 token budget)
- Convention guardrails + session lifecycle

Phase 14.2 upgrades _how Claude calls those tools_:
1. Examples in tool schemas raise accuracy 72% → ~90%
2. Deferred loading prevents schema bloat when toolset grows (Phase 15 adds more)
3. Batch/programmatic queries collapse multi-ping-pong loops into single calls
4. Context filtering keeps raw graph noise outside the effective window

These are orthogonal to Phase 15's Self-Healing work and can ship without blocking it.

## Video Concepts → Codebase Mapping

### C1: Programmatic / Code-Execution Tool Calling

**Problem**: `get_dependencies_tool` already issues two separate `query_graph()` calls
internally. Tier 3 agents that need to resolve 20 products against conventions call
`retrieve_intent` 20 times, burning 20 round trips.

**Solution**:
- Add `batch_query` MCP tool — accepts `queries: list[str]`, runs them in a loop server-side,
  returns aggregated results in one call.
- Add `batch_dependencies` MCP tool — accepts `file_paths: list[str]`, returns combined
  dependency map.
- When using the Anthropic SDK (OpenRouter/Gemini Flash), pass `betas=["code-execution-2026-01-20"]`
  so the model can emit Python snippets to loop over Graphiti episode results, conditional logic,
  and post-processing — all within a single LLM round trip.

**Files touched**:
- `src/graph/mcp_server.py` — add `batch_query_tool`, `batch_dependencies_tool`, register in
  `list_tool_contracts()` and `call_tool()`
- `src/tasks/graphiti_sync.py` — emit episodes in batches (list[Episode]) not one by one
- `src/jobs/graphiti_ingestor.py` — chunk list and loop internally; don't surface individual
  failures up the call chain

**Token reduction estimate**: 30–50% for multi-entity Tier 3 resolution flows.

### C2: Context Window Optimization (Effective Window vs Nominal Window)

**Problem**: Raw Cypher results and full vendor HTML can be verbose. Even with the 8192-token
budget in `search_expand_bridge.py`, a node dump with full `description` fields can consume
the budget without conveying more signal than a summary.

**Solution**:
- Add `compact_output: bool = False` parameter to `query_graph_tool`. When `True`, strip
  node fields to `{path, type, summary}` only. Default `False` preserves current behaviour.
- Vendor web scrapers: adopt the `web_fetch_tool_2026_02_09` filtering pattern — extract
  structured content (product title, price, description) from raw HTML before it reaches the
  assistant context. This applies to `src/api/v1/` vendor adapter endpoints.
- `search_expand_bridge.py`: reduce the per-node token estimate from ~50 to ~30 when
  `compact_output=True`, allowing 2.7× more nodes within the same 8192-token budget.

**Files touched**:
- `src/graph/mcp_server.py` — `compact_output` param on `query_graph_tool`
- `src/graph/search_expand_bridge.py` — compact vs full serialization mode
- Vendor adapter layer (identified in Phase 15 scoping, not this phase)

**Token reduction estimate**: ~24% on web-fetch-heavy flows; ~40% on large graph retrievals
with `compact_output=True`.

### C3: Deferred Tool Loading + Tool Search Meta-Tool

**Problem**: Currently 3 MCP tools + 6 assistant tools = 9 total. Phase 15 adds refactor
tools, optimization tools, and pattern-detection tools. Beyond 10 tools, schema loading
consumes significant prompt tokens on every request.

**Solution**:
- Add a `search_tools` MCP tool: accepts a natural language query, returns matching tool
  names + their schemas from the knowledge graph. This is the "Tool Search" pattern from the
  video.
- Store tool schemas as `Tool` nodes in Neo4j (extend `src/core/synthex_entities.py`).
  Relationships: `Tool -[REQUIRES_INTEGRATION]-> Integration`, `Tool -[ALLOWED_IN]-> Tier`.
- Set `"deferred_loading": true` in `.claude/settings.local.json` once tool count exceeds 10.
  This hides all tool schemas except `search_tools` from the initial prompt; Claude calls
  `search_tools("resolve product")` to discover and load just the schemas it needs.
- `AssistantToolRegistry` model: add `schema_json: JSON` column (with Alembic migration) to
  persist the full JSON schema alongside the metadata that already exists.

**Files touched**:
- `src/graph/mcp_server.py` — add `search_tools_tool`, register it
- `src/core/synthex_entities.py` — add `ToolNode` entity + `REQUIRES_INTEGRATION`, `ALLOWED_IN` edges
- `src/models/assistant_tool_registry.py` — add `schema_json` column
- `migrations/` — new Alembic migration for `schema_json`
- `.claude/settings.local.json` — add `"deferred_loading": true` when ready
- `src/graph/query_templates.py` — add `tool_search` Cypher template

**Token reduction estimate**: Scales with tool count — at 15 tools saves ~40% prompt tokens
per request.

### C4: Input Examples in Tool Schemas

**Problem**: All 3 MCP tools and 6 assistant tools have description-only schemas. The video
cites a jump from 72% to 90% accuracy for complex nested parameters and date/format-specific
inputs simply by adding `input_examples`.

**Solution**:
- Add `input_examples` key to every tool schema in `list_tool_contracts()` in `mcp_server.py`.
- Extend `AssistantToolRegistry.metadata_json` to carry an `input_examples` array that
  `tool_projection.py` passes through when building the effective toolset for the LLM call.

**Concrete examples to add**:

`query_graph`:
```json
"input_examples": [
  {"query": "who imports src/core/graphiti_client.py"},
  {"query": "what conventions exist for error handling"},
  {"query": "show decisions made about the token budget"}
]
```

`get_dependencies`:
```json
"input_examples": [
  {"file_path": "src/core/graphiti_client.py", "direction": "imported_by", "depth": 2},
  {"file_path": "src/graph/mcp_server.py", "direction": "both", "depth": 1}
]
```

`retrieve_intent`:
```json
"input_examples": [
  {"query": "why was 8192 chosen as the token budget"},
  {"query": "what architectural decisions exist for Neo4j vector index"},
  {"query": "any known bugs related to session lifecycle hooks"}
]
```

`batch_query` (new):
```json
"input_examples": [
  {"queries": ["who imports graphiti_client.py", "what calls semantic_cache.py"]},
  {"queries": ["conventions for rate limiting", "decisions about Celery queues"]}
]
```

`search_tools` (new):
```json
"input_examples": [
  {"query": "tools for reading product data"},
  {"query": "tools allowed in tier 2 that mutate data"}
]
```

**Files touched**:
- `src/graph/mcp_server.py` — `list_tool_contracts()` + each tool's schema dict
- `src/assistant/tool_projection.py` — propagate `input_examples` from registry to LLM payload
- `src/models/assistant_tool_registry.py` — `metadata_json` already exists; document convention

**Accuracy improvement**: 72% → ~90% on complex nested param / format-specific inputs.

## Success Criteria

1. All 5 MCP tools (3 existing + `batch_query` + `search_tools`) have `input_examples`.
2. `search_tools("resolve product")` returns at least one matching tool schema from Neo4j.
3. `batch_query(queries=["A", "B", "C"])` returns aggregated results in **one** MCP call.
4. Token consumption for a 10-entity Tier 3 resolution flow drops by ≥30% vs baseline.
5. `compact_output=True` on `query_graph_tool` reduces token usage by ≥20% vs full output.
6. `deferred_loading: true` set in `.claude/settings.local.json` with `search_tools` as the
   discovery entry point.
7. `AssistantToolRegistry.schema_json` column exists, migrated, and populated for all 6
   default tools.
8. No existing tests broken. All new behaviour covered by unit tests.
