### Core Concepts

#### C1: Programmatic/Code-Execution Tool Calling

**Problem**: When using natural language tool calling, each invocation requires a full inference pass, and intermediate results pile up in context whether they're useful or not.

**Solution**: Provide Claude with a code execution tool and describe what functions are available in that environment. When Claude invokes the tool with code, your application executes it locally where those functions are defined.

**How it works**:
```python
# Instead of this (3 separate LLM round-trips):
result1 = call_tool("query_graph", {"query": "who imports X"})
result2 = call_tool("query_graph", {"query": "who calls Y"})
result3 = call_tool("query_graph", {"query": "what depends on Z"})

# Do this (1 LLM round-trip with programmatic execution):
code = """
queries = ["who imports X", "who calls Y", "what depends on Z"]
results = [query_graph(q) for q in queries]
return aggregate_results(results)
"""
result = call_tool("code_execution", {"code": code})
```

**Application to this codebase**:
- Add `batch_query(queries: list[str])` MCP tool
- Add `batch_dependencies(file_paths: list[str])` MCP tool
- Use Celery `group()` for episode batch emission

**Token savings**: 30-50% on multi-entity flows

#### C2: Context Window Optimization (Effective Window vs Nominal Window)

**Problem**: Raw graph results with full descriptions consume the 8192-token budget inefficiently. A node with a 300-token description may only contribute 10 tokens of signal.

**Solution**: Compact output modes filter before entering LLM context.

**Key insight**: "The idea is simple: run loops and conditionals in a code execution environment, call tools there, then only return the parts that matter to the model. Less sludge in the context window."

**Application to this codebase**:
- Add `compact_output: bool = False` to `query_graph_tool`
- When `True`: return `{path, type, summary}` only (strip verbose fields)
- Adjust token estimates: 30 tokens/node (compact) vs 50 tokens/node (full)
- Result: 2.7× more nodes within same 8192 budget

**Token savings**: ~40% on large graph retrievals

---

#### C3: Tool Search (Dynamic Discovery)

**Problem**: Loading 15+ tool schemas upfront consumes 1,200+ tokens/request in prompt overhead.

**Solution**: Expose only a `search_tools` meta-tool initially. Claude calls it to discover capabilities on-demand.

**Pattern**:
```python
# Instead of loading all tools upfront:
tools = [tool1_schema, tool2_schema, ..., tool15_schema]  # 1200 tokens

# Do this:
tools = [search_tools_schema]  # 80 tokens

# Claude discovers on demand:
result = call_tool("search_tools", {"query": "tools for updating prices"})
# Returns only product_mutation_tool schema (150 tokens)
```

**Application to this codebase**:
- Store tool schemas as `ToolNode` entities in Neo4j
- Add `search_tools` MCP tool that queries the graph
- Set `deferred_loading: true` in `.claude/settings.local.json`

**Token savings**: Scales with tool count - at 15 tools saves ~40% per request

---

#### C4: Input Examples in Tool Schemas

**Problem**: JSON schema tells an agent what's valid, not what's smart.

**Key research finding**: "Tool examples teach conventions and 'how people actually call this thing,' which in practice can be the difference between a tool that works 60% of the time and one that works 90% of the time."

**Accuracy improvement**: 72% → 90% on nested/conditional params

**Example**:
```json
{
  "name": "query_graph",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {"type": "string"}
    }
  },
  "input_examples": [
    {"query": "who imports src/core/graphiti_client.py"},
    {"query": "what conventions exist for error handling"},
    {"query": "show decisions made about the token budget"}
  ]
}
```

**Application to this codebase**:
- Add `input_examples` to all 3 MCP tools
- Add `input_examples` to all 6 assistant tools in `metadata_json`
- Propagate through `tool_projection.py`

**Impact**: Reduces retry/correction round trips by ~18%

#### Tool Discovery Protocol

MCP defines `tools/list` endpoint for tool discovery:

```python
async def list_tools():
    """List all available tools - always returns fresh data."""
    session = client.get_session("my_server")
    tools = await session.list_tools()

    for tool in tools:
        print(f"Tool: {tool.name}")
        print(f"Description: {tool.description}")
        print(f"Schema: {tool.inputSchema}")
```

**Protocol details**:
- Clients use `*/list` methods to discover available primitives
- Servers send update notifications when tools change
- Supports real-time notifications via JSON-RPC 2.0

**Application to this codebase**:
- `list_tool_contracts()` already implements this pattern
- Add notification support for tool schema changes (Phase 15)
- Tool nodes in graph enable semantic search over `list_tools()` results

#### Tool Calling Pattern

**From [mcp-use Python SDK](https://mcp-use.com/docs/python/client/direct-tool-calls)**:

```python
# Call a specific tool with arguments
result = await session.call_tool(
    name="add",
    arguments={"a": 1, "b": 2}
)

# Handle the result
if getattr(result, "isError", False):
    print(f"Error: {result.content}")
else:
    print(f"Tool result: {result.content}")
```

**Error handling pattern**:
```python
try:
    result = await session.call_tool("risky_operation", {"param": "value"})
    if result.isError:
        print(f"Tool reported error: {result.content}")
    else:
        print(f"Success: {result.content}")
except Exception as e:
    print(f"Connection or protocol error: {e}")
```

**Application to this codebase**:
- Already following this pattern in `mcp_server.py`
- Add batch variants: `batch_query`, `batch_dependencies`
- Ensure error handling covers both tool errors and protocol errors

### Key Findings

#### Batch Query Patterns

**Pattern 1: Multiple queries in same session** (from Context7):

```python
with driver.session(database="neo4j") as session:
    # Auto-commit transactions - each query commits independently
    result1 = session.run("CREATE (p:Person {name: $name})", name="Alice")
    result2 = session.run("CREATE (p:Person {name: $name})", name="Bob")
    result3 = session.run("MATCH (p:Person) RETURN count(p) AS total")
```

**Pattern 2: Explicit transaction** (better for atomicity):

```python
with driver.session() as session:
    with session.begin_transaction() as tx:
        tx.run("CREATE (p:Person {name: $name})", name="Alice")
        tx.run("CREATE (p:Person {name: $name})", name="Bob")
        tx.run("CREATE (p:Person {name: $name})", name="Charlie")
        # Auto-commits on exit
```

**Pattern 3: Transaction functions with retry** (production-ready):

```python
from neo4j import unit_of_work

@unit_of_work(timeout=5.0)
def batch_create_tx(tx, names):
    for name in names:
        tx.run("CREATE (p:Person {name: $name})", name=name)

with driver.session() as session:
    session.execute_write(batch_create_tx, ["Alice", "Bob", "Charlie"])
```

**Application to this codebase**:
- Use transaction functions for `batch_query` tool
- Retry logic automatic with `@unit_of_work`
- Timeout protection prevents worker hangs

async def main():

#### Vector Search Capabilities

**From [Neo4j Vector Search Docs](https://neo4j.com/developer/genai-ecosystem/vector-search/)**:

**Similarity scoring**:
> "The scores are bounded between 0.0 and 1.0, where the closer to 1.0 the score is, the more similar the indexed vector is to the query vector."

**k-NN queries**:
```cypher
CALL db.index.vector.queryNodes(
    'tool_embedding_index',
    $top_k,
    $query_embedding
) YIELD node, score
WHERE node:Tool
RETURN node.name AS name, score
ORDER BY score DESC
```

**Native vector type** (Neo4j 2025.10+):
- First-class `VECTOR` type for storing embeddings
- End-to-end support (drivers → Bolt → Cypher → storage)
- Constraints on vector dimensions

**Application to this codebase**:
- Create `tool_embedding_index` on `Tool.embedding`
- Use 384-dim embeddings (same as Phase 14.1)
- Cosine similarity function

---

#### Vector Index Configuration

**From [Neo4j Vector Indexes Manual](https://neo4j.com/docs/cypher-manual/current/indexes/semantic-indexes/vector-indexes/)**:

```cypher
CREATE VECTOR INDEX tool_embedding_index IF NOT EXISTS
FOR (n:Tool)
ON n.embedding
OPTIONS {
    indexConfig: {
        `vector.dimensions`: 384,
        `vector.similarity_function`: 'cosine'
    }
}
```

**Index types**:
- **HNSW** (default): Hierarchical Navigable Small World - fast approximate k-NN
- **Flat**: Exact k-NN - slower but perfect recall

**Application to this codebase**:
- Use HNSW for tool search (speed > perfect recall)
- 384 dimensions matches sentence-transformers model from Phase 14.1

**Projected savings**: 35-50% reduction in tokens/request
