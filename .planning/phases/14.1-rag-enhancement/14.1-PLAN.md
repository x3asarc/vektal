# Phase 14.1: Neo4j-Vector-Hybrid-RAG Enhancement

**Goal:** Upgrade the Phase 14 codebase knowledge graph from a passive data store into an active, dual-layered reasoning engine with MCP tool access, memory engineering, semantic caching, and search-then-expand retrieval.

**Depends on:** Phase 14 (Codebase Knowledge Graph - provides Neo4j schema, entities, scanner, embeddings, query templates)

**Cross-reference:** See `14.1-CROSS-REFERENCE.md` for detailed feature-by-feature gap analysis against YAML blueprint.

---

## Success Criteria (what must be TRUE)

1. Claude Code can query the knowledge graph via MCP tools (`query_graph`, `get_dependencies`, `retrieve_intent`) without Python imports or shell commands.
2. Decision, Convention, and BugRootCause nodes exist in the graph with EXPLAINS, RESOLVED_BY, and SUPERCEDES relationships.
3. Search-then-Expand bridge retrieves context in two phases (vector search → graph traversal) with enforced constraints (max 5 initial nodes, depth 2, 8192 token budget).
4. Semantic cache serves identical or near-identical queries (cosine >= 0.92) without re-executing graph traversal, reducing latency by >50% on cache hits.
5. Session initialization automatically pulls top Convention nodes to provide architectural context.
6. Null graph results fall back to filesystem search with discrepancy flagging.

## Plans: 6 plans in 3 waves

### Wave 1: Memory Engineering + Entity Extensions (Foundation)

#### Plan 14.1-01: Memory Entity Types and Relationships

**What:** Extend the entity schema with Decision, Convention, BugRootCause, and Conversation node types plus EXPLAINS, RESOLVED_BY, and SUPERCEDES relationship edges.

**Why:** The existing entity types (OracleDecision, FailurePattern, etc.) capture runtime events but lack architectural memory. Convention nodes capture project standards ("always use RFC 7807 errors"). Decision nodes capture "why we chose X over Y." BugRootCause nodes link symptoms to verified fixes. These are the long-term memory layer that makes the RAG system intelligent across sessions.

**Tasks:**

1. **Add memory entity types** to `src/core/synthex_entities.py`:
   - `DecisionEntity`: captures architectural decisions with `context`, `rationale`, `alternatives`, `status` (active/superseded), `phase_ref`
   - `ConventionEntity`: captures project standards with `rule`, `scope` (global/module/file), `enforcement` (hard/soft), `examples`
   - `BugRootCauseEntity`: captures verified bug root causes with `symptom`, `root_cause`, `fix_description`, `affected_files`, `resolved_by_commit`
   - `ConversationEntity`: captures key conversation threads with `topic`, `outcome`, `participants`, `session_id`

2. **Add memory relationship edges**:
   - `ExplainsEdge`: links any entity to a Decision/Convention that explains it
   - `ResolvedByEdge`: links BugRootCause to the fix (commit, file change, etc.)
   - `SupersedesEdge`: links newer Decision/Convention to the one it replaces (with `reason`)

3. **Add episode types** to `EpisodeType` enum:
   - `DECISION_RECORDED = "decision_recorded"`
   - `CONVENTION_ESTABLISHED = "convention_established"`
   - `BUG_ROOT_CAUSE_IDENTIFIED = "bug_root_cause_identified"`

4. **Extend Neo4j schema** in `src/core/codebase_schema.py`:
   - Add unique constraints for Decision, Convention, BugRootCause, Conversation
   - Add indexes for Convention.scope and Decision.status

5. **Seed initial Convention nodes** from existing project knowledge:
   - Parse `CLAUDE.md` rules → Convention nodes (RFC 7807 errors, KISS policy, LOC limits, etc.)
   - Parse `STANDARDS.md` → Convention nodes
   - Parse `PROJECT.md` Key Decisions table → Decision nodes

#### Plan 14.1-02: Reasoning Provenance and Discrepancy Tracking

**What:** Add reasoning trace storage (past queries + logic) and discrepancy flagging when graph returns null but filesystem has data.

**Why:** Without reasoning provenance, the same query gets re-derived from scratch every time. With it, the system can show "I answered a similar question 3 days ago and here's what I found." Discrepancy flagging closes the operational guardrail gap where silent null returns mask stale graph state.

**Tasks:**

1. **Add ReasoningTrace entity** to synthex_entities.py:
   - `query_text`: the original question/query
   - `cypher_generated`: the Cypher query executed
   - `template_used`: which template matched (if any)
   - `result_summary`: condensed result (not full data)
   - `duration_ms`: execution time
   - `was_cache_hit`: boolean

2. **Add trace storage** to `src/graph/query_interface.py`:
   - After every successful query, emit a `ReasoningTrace` episode
   - Store via existing Celery `emit_episode` (fire-and-forget)
   - Include `query_embedding` for future similarity matching

3. **Add discrepancy detection** to `src/core/graphiti_client.py`:
   - When `query_with_fallback()` returns null from graph but filesystem search finds results:
     - Log discrepancy with file paths and query details
     - Emit `GRAPH_DISCREPANCY` episode to graph
     - Return filesystem results with `source: "filesystem_fallback"` metadata
   - Add `GRAPH_DISCREPANCY` to EpisodeType enum

4. **Add discrepancy dashboard query template**:
   - `recent_discrepancies` template: last 50 discrepancies with file paths
   - Useful for identifying graph sync gaps

### Wave 2: Retrieval Intelligence (Bridge + Cache)

#### Plan 14.1-03: Search-then-Expand Bridge

**What:** Build the two-phase retrieval mechanism that first finds semantically relevant nodes via vector search, then expands context through graph traversal with bounded constraints.

**Why:** This is the core innovation of the YAML blueprint. Current queries are either pure vector (find similar files) OR pure graph (follow imports). The bridge combines both: "Find the 5 most relevant nodes, then follow their relationships 2 hops deep, within an 8192 token budget." This produces context that is both semantically relevant AND structurally complete.

**Tasks:**

1. **Create bridge module** `src/graph/search_expand_bridge.py`:
   - `search_then_expand(query_embedding, config)` → `BridgeResult`
   - `BridgeConfig` dataclass:
     - `max_initial_nodes: int = 5`
     - `max_traversal_depth: int = 2`
     - `max_context_tokens: int = 8192`
     - `expand_relationships: List[str] = ["IMPORTS", "CALLS", "IMPLEMENTS", "EXPLAINS"]`
   - Phase 1 (Search): Vector similarity query → top-k initial nodes
   - Phase 2 (Expand): For each initial node, follow relationships up to depth
   - Token budgeting: estimate tokens per node (path + purpose ≈ 50 tokens), stop expanding when budget reached
   - Deduplication: nodes seen in expansion are not double-counted

2. **Create BridgeResult dataclass**:
   - `initial_nodes: List[Dict]` - vector search results
   - `expanded_nodes: List[Dict]` - graph traversal additions
   - `total_tokens_estimated: int`
   - `truncated: bool` - whether budget was exceeded
   - `relationships_followed: List[str]` - which edge types were traversed

3. **Wire bridge into query_interface.py**:
   - Add `query_with_bridge(query_text)` function
   - For natural language queries that don't match a template, use bridge
   - Bridge results include both semantic matches and structural context

4. **Add Cypher for bounded expansion**:
   - `MATCH (n)-[r*1..2]->(m)` with relationship type filter
   - Parameterized depth from BridgeConfig
   - LIMIT on expanded nodes per initial node

#### Plan 14.1-04: Semantic Cache Layer

**What:** Add a similarity-based cache that serves results for near-identical queries without re-executing graph traversal.

**Why:** Many queries are repeated or near-identical across sessions ("what imports db.py", "show me the auth flow"). A semantic cache with 0.92 cosine threshold eliminates redundant graph traversal and LLM calls, reducing latency by >50% on cache hits.

**Tasks:**

1. **Create cache module** `src/graph/semantic_cache.py`:
   - `SemanticCache` class with configurable TTL and threshold
   - `CacheConfig`:
     - `similarity_threshold: float = 0.92`
     - `max_entries: int = 1000`
     - `ttl_seconds: int = 3600` (1 hour default)
   - `lookup(query_embedding) -> Optional[CachedResult]`:
     - Compare query embedding against cached query embeddings
     - If cosine similarity >= threshold, return cached result
     - Track hit/miss metrics
   - `store(query_embedding, query_text, result, duration_ms)`:
     - Store result with timestamp for TTL eviction
     - LRU eviction when max_entries exceeded
   - `invalidate(file_paths: List[str])`:
     - Invalidate cache entries whose results reference changed files
     - Called by incremental sync when files are modified

2. **Storage backend**: In-memory dict with numpy for cosine similarity
   - Cache entries: `{embedding, query_text, result, timestamp, referenced_paths}`
   - Bulk cosine similarity using numpy dot product (fast)
   - Persistence: Optional JSON dump on shutdown, load on startup

3. **Wire cache into query flow**:
   - In `query_interface.py`: check cache BEFORE template matching or graph query
   - If cache hit: return immediately with `source: "semantic_cache"`
   - If cache miss: execute query, store result in cache
   - In `incremental_sync.py`: call `cache.invalidate(changed_files)` after sync

4. **Add cache metrics**:
   - Hit rate, miss rate, average hit latency, average miss latency
   - Expose via `get_cache_stats()` function
   - Log cache hit/miss at DEBUG level

**Files touched:**
- `src/graph/semantic_cache.py` (create)
- `src/graph/query_interface.py` (modify)
- `src/graph/incremental_sync.py` (modify)
- `tests/unit/test_semantic_cache.py` (create)

### Wave 3: Agentic Interface (MCP + Lifecycle)

#### Plan 14.1-05: MCP Tool Server

**What:** Expose the knowledge graph as an MCP (Model Context Protocol) server with three tools: `query_graph`, `get_dependencies`, and `retrieve_intent`.

**Why:** This is the highest-impact missing feature. Without MCP, Claude Code cannot access the knowledge graph during normal operation. With MCP, every Claude Code session automatically has graph intelligence available as native tools.

**Tasks:**

1. **Create MCP server** `src/graph/mcp_server.py`:
   - Use `mcp` Python SDK (add to requirements.txt)
   - Implement three tools:

   **Tool 1: `query_graph`**
   - Input: `query: str` (natural language or Cypher)
   - Behavior: Routes through `query_interface.py` (template match → bridge → natural language)
   - Output: `{results: [...], source: "template"|"bridge"|"cache", duration_ms: float}`

   **Tool 2: `get_dependencies`**
   - Input: `file_path: str`, `direction: "imports"|"imported_by"|"both"`, `depth: int = 1`
   - Behavior: Executes appropriate import template with bounded depth
   - Output: `{file: str, dependencies: [{path, purpose, depth}], impact_radius: int}`

   **Tool 3: `retrieve_intent`**
   - Input: `query: str` (search for decisions, conventions, bug root causes)
   - Behavior: Vector similarity search against Decision + Convention + BugRootCause nodes
   - Output: `{results: [{type, content, confidence, created_at}], source: "vector"}`

2. **MCP server configuration**:
   - Stdio transport (standard for Claude Code MCP servers)
   - Server name: `synthex-knowledge-graph`
   - Version: matches project version

3. **Claude Code integration** `.claude/settings.local.json`:
   - Add MCP server entry pointing to `python src/graph/mcp_server.py`
   - Ensure server starts with correct PYTHONPATH and env vars

4. **Startup script** `scripts/graph/start_mcp_server.sh`:
   - Checks Neo4j availability before starting
   - Falls back gracefully if graph unavailable
   - Sets PYTHONPATH and loads .env

#### Plan 14.1-06: Session Lifecycle Hooks and Convention Guardrails

**What:** Add session initialization (pull top Convention nodes) and on-query hooks (cache check + Convention verification before suggesting changes).

**Why:** Lifecycle hooks make the RAG system proactive rather than reactive. On session start, Claude Code automatically knows the project's architectural conventions. On every query, the system checks the cache first and verifies suggestions against conventions.

**Tasks:**

1. **Session init hook** in MCP server:
   - On server startup (or first tool call), query top 3 Convention nodes by reference count
   - Return as `system_context` in initial tool response
   - Conventions cached in-memory for session duration
   - Convention format: `{rule, scope, enforcement, examples}`

2. **Convention checking** in `src/graph/convention_checker.py`:
   - `check_against_conventions(proposed_change: str) -> List[ConventionViolation]`
   - For each active Convention node, check if proposed change conflicts
   - Uses vector similarity between change description and convention rule text
   - Threshold: 0.7 similarity = potential conflict, flag for review
   - Returns violations with convention text, confidence, and suggested alternative

3. **On-query hook** in query_interface.py:
   - Before executing any query: check semantic cache (from Plan 14.1-04)
   - Before returning architectural suggestions: check conventions (from step 2)
   - Add `conventions_checked: List[str]` to QueryResult

4. **CALLS verification guardrail** in `src/graph/refactor_guard.py`:
   - `verify_refactor_safety(file_path: str) -> RefactorSafetyReport`
   - Queries `function_callers` and `function_callees` for affected file
   - Returns: `{safe: bool, downstream_files: List[str], callers_count: int, risk_level: "low"|"medium"|"high"}`
   - Risk levels: 0-2 callers = low, 3-10 = medium, 11+ = high

5. **Wire convention check into MCP retrieve_intent tool**:
   - When retrieving intent for architectural questions, include relevant conventions
   - Convention nodes returned alongside Decision nodes for complete context

## What This Does NOT Include (Deferred)

These items from the YAML blueprint are deferred because they require architectural changes beyond the current scope or have lower ROI:

1. **Tree-sitter parser** - Current Python AST works; Tree-sitter adds multi-language support but requires new dependency and parser rewrites. Deferred to Phase 15 or future phase when TypeScript parsing becomes critical.

2. **File watcher (active-on-save)** - Git hooks + consistency daemon cover 99% of updates. Active-on-save requires a filesystem watcher daemon (inotify/fsevents), adding complexity for marginal benefit. Deferred.

3. **1536-dimension embeddings** - Would require switching from local sentence-transformers to OpenAI ada-002 (API cost) or larger local model (memory cost). Current 384-dim is sufficient for code structure matching. Deferred.

4. **20% overlap chunking** - Current function-level chunking produces clean semantic boundaries. Overlap helps in prose but adds noise in structured code. Deferred.

5. **Conversation nodes in graph** - Chat sessions already persisted in PostgreSQL. Duplicating in Neo4j adds sync complexity. Deferred until a specific cross-session retrieval use case emerges.
