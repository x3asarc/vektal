## Summary

Phase 14.1 is correctly scoped as a bridge phase between the current "queryable graph foundation" and a true agent-usable retrieval system. External evidence confirms that the project can implement MCP tools with the official Python MCP SDK over stdio, execute bounded vector + graph retrieval with Neo4j 5.x primitives, and enforce predictable query safety with parameterized Cypher plus bounded expansion.

The largest implementation risk is not external technology risk; it is internal integration debt. Current code in `src/graph/query_templates.py` returns stubbed empty results, and `src/jobs/graphiti_ingestor.py` still contains a TODO placeholder instead of a real `Graphiti.add_episode(...)` ingestion call. If these are not addressed first, later plans (reasoning traces, discrepancy episodes, semantic cache hit-quality metrics) will appear implemented in code structure but remain non-functional at runtime.

**Primary recommendation:** Implement Phase 14.1 in strict dependency order with a hard "functional query path gate" before Wave 2: first make real graph query execution + real episode ingestion operational, then layer bridge/cache/MCP on top.

## Architecture Patterns

### Pattern 1: Functional Query Path Gate (Pre-Wave-2)

**What:** Make base query and episode ingestion fully functional before bridge/cache/MCP features.  
**Why:** Later features depend on real query outputs and persisted episodes, not stubs.  
**Implementation contract:**

1. `execute_template()` must run Cypher against Neo4j and return records.
2. `GraphitiIngestor.ingest_episode()` must call real Graphiti ingestion API.
3. Add one integration test proving: query executes -> trace episode emitted -> persisted retrieval path works.

### Pattern 2: Bounded Search-Then-Expand

**What:** Two-phase retrieval:
1. vector seed (`topK <= 5`)
2. bounded traversal (`depth <= 2`, relationship allowlist)

**Reference contract from Neo4j docs:**
- Use `db.index.vector.queryNodes(indexName, numberOfNearestNeighbours, queryVector)`.
- Similarity score range is `0..1`, higher is better.
- Keep traversal bounded with quantified path depth and explicit filtering.

**Suggested Cypher shape:**

```cypher
CALL db.index.vector.queryNodes($index_name, $top_k, $query_embedding)
YIELD node, score
WITH node, score
WHERE score >= $min_score
MATCH p = (node)-[*1..2]->(m)
WHERE ALL(rel IN relationships(p) WHERE type(rel) IN $allowed_relationships)
RETURN node, score, collect(DISTINCT m) AS expanded
LIMIT $global_limit
```

### Pattern 3: Semantic Cache First in Query Pipeline

**What:** Cache lookup before template/bridge traversal.  
**Why:** Avoid repeated graph traversal and keep latency stable for repeated prompts.  
**Contract:**

1. Normalize embeddings before cosine comparison.
2. Gate hit by similarity threshold (`0.92` initially, calibrate empirically).
3. Track `hit/miss`, latency, and invalidations by changed file paths.
4. Store `referenced_paths` for deterministic invalidation from incremental sync.

### Pattern 4: MCP Adapter Layer (Not Direct DB in Tools)

**What:** MCP tools call query/service layer, not raw Cypher directly in tool handlers.  
**Why:** Keeps one policy path for cache, guardrails, discrepancy handling, and tracing.  
**Tool contracts:**

- `query_graph(query: str)`
- `get_dependencies(file_path: str, direction: str, depth: int)`
- `retrieve_intent(query: str)`

### Pattern 5: Memory/Provenance as Typed Episodes

**What:** Model `Decision`, `Convention`, `BugRootCause`, `ReasoningTrace` with clear edge semantics.  
**Why:** Prevent "blob memory" and support deterministic retrieval by entity type + relationship.

## Plan-by-Plan Contracts

### 14.1-01 Memory Entities and Relationships

- Add new entity/edge classes in `src/core/synthex_entities.py` with strict field-level validation.
- Update enum and helpers in lockstep:
  - `EpisodeType`
  - `create_episode_payload`
  - `_generate_episode_id` branching in `src/tasks/graphiti_sync.py`
- Extend Neo4j schema (`src/core/codebase_schema.py`) with:
  - unique keys for new node families
  - supporting indexes for `Convention.scope`, `Decision.status`
- Seeding should be idempotent (MERGE-based), with `--dry-run`.

### 14.1-02 Reasoning Provenance and Discrepancy

- Trace emission must occur after successful query execution path (template or bridge).
- Discrepancy events should be emitted only when:
  - graph result is empty/null
  - filesystem fallback has concrete matches
- Add dedicated template for discrepancy inspection (`recent_discrepancies`).

### 14.1-03 Search-Then-Expand Bridge

- Use vector procedure for initial candidates and hard cap at `max_initial_nodes=5`.
- Enforce traversal depth ceiling at query level (`*1..2`) and code-level config.
- Include relationship allowlist to prevent noisy cross-domain expansion.
- Stop expansion once token budget reaches 8192 (heuristic first, tokenizer-backed if needed).

### 14.1-04 Semantic Cache

- Use in-memory cache initially (LRU + TTL) with numpy cosine similarity.
- Threshold `0.92` should be treated as initial operating point, then tuned with real query logs.
- Invalidation hook belongs in `incremental_sync.py` after changed file detection.
- Return metadata fields:
  - `source: semantic_cache|template|bridge|filesystem_fallback`
  - `was_cache_hit: bool`

### 14.1-05 MCP Tool Server

- Implement server over stdio with official MCP SDK.
- Prefer low-level server when explicit `inputSchema`/`outputSchema` validation is required.
- Add project-scoped MCP server configuration (`.mcp.json` recommended by Claude docs, or equivalent local settings in this repo conventions).
- Startup should validate Neo4j availability and degrade cleanly.

### 14.1-06 Lifecycle Hooks and Guardrails

- Use server lifespan/init logic to preload top conventions once per session.
- Convention checking should be advisory-first (flag violations) before hard-blocking.
- Add CALLS-based refactor risk scoring with deterministic thresholds.
- Ensure convention checks run through same query pipeline as all other retrieval paths.

## Don’t Hand-Roll

| Problem | Don’t Build | Use Instead | Why |
|---|---|---|---|
| MCP transport/protocol | Custom JSON-RPC protocol server | Official MCP Python SDK | Avoids protocol drift and schema incompatibilities |
| ANN retrieval implementation | Manual nearest-neighbor scan in Python | Neo4j vector index procedures | Native index query is supported and optimized |
| Query timeout framework | Ad-hoc timeout wrappers in each query | Driver `Query(..., timeout=...)` or centralized query wrapper | Consistent timeout behavior and easier observability |
| Memory graph ingestion | Custom one-off writers | Graphiti ingestion flow with typed entities/edges | Keeps temporal/event semantics consistent |

## Common Pitfalls

### Pitfall 1: "Implemented" Features on Top of Stubs
- **What goes wrong:** Bridge/cache/MCP compile but return empty data.
- **Why:** Base template execution and ingestion are still placeholders.
- **Avoid:** Add pre-wave gate requiring real query + persisted episode integration test.

### Pitfall 2: Enum Drift Breaking Episode Emission
- **What goes wrong:** New episode types raise `ValueError` or produce weak IDs.
- **Why:** `EpisodeType` and `_generate_episode_id` are tightly coupled.
- **Avoid:** Update enum + ID strategy + tests in one change set.

### Pitfall 3: Unbounded Traversal Explosion
- **What goes wrong:** High latency and noisy context windows.
- **Why:** Missing depth/relationship/limit constraints.
- **Avoid:** Depth cap, relationship allowlist, global result limit, token budget checks.

### Pitfall 4: Cache Pollution and Stale Context
- **What goes wrong:** Wrong-but-similar answers persist after code changes.
- **Why:** Weak invalidation and no referenced-path tracking.
- **Avoid:** Track `referenced_paths` and invalidate on incremental sync events.

### Pitfall 5: MCP Tool Direct-DB Access
- **What goes wrong:** Policy bypass (no cache, no discrepancy tracking, no conventions).
- **Why:** Tool handlers execute standalone logic.
- **Avoid:** Route all tools through one query service pipeline.

## Code Examples (Implementation-Oriented)

### 1) Neo4j vector seed + bounded expansion

```python
cypher = """
CALL db.index.vector.queryNodes($index_name, $top_k, $query_embedding)
YIELD node, score
WITH node, score WHERE score >= $min_score
MATCH p = (node)-[*1..2]->(m)
WHERE ALL(rel IN relationships(p) WHERE type(rel) IN $allowed_rels)
RETURN node, score, collect(DISTINCT m) AS expanded
LIMIT $limit
"""
params = {
    "index_name": "codebase_embeddings",
    "top_k": 5,
    "query_embedding": query_embedding,
    "min_score": 0.70,
    "allowed_rels": ["IMPORTS", "CALLS", "IMPLEMENTS", "EXPLAINS"],
    "limit": 200,
}
```

### 2) MCP stdio server skeleton

```python
import asyncio
import mcp.server.stdio
import mcp.types as types
from mcp.server.lowlevel import Server, NotificationOptions
from mcp.server.models import InitializationOptions

server = Server("synthex-knowledge-graph")

@server.list_tools()
async def list_tools():
    return [
        types.Tool(
            name="query_graph",
            description="Query knowledge graph",
            inputSchema={"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]},
        )
    ]

async def main():
    async with mcp.server.stdio.stdio_server() as (read_stream, write_stream):
        await server.run(
            read_stream,
            write_stream,
            InitializationOptions(
                server_name="synthex-knowledge-graph",
                server_version="0.1.0",
                capabilities=server.get_capabilities(notification_options=NotificationOptions(), experimental_capabilities={}),
            ),
        )

if __name__ == "__main__":
    asyncio.run(main())
```
