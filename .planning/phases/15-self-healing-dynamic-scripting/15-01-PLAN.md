Enhance existing SandboxRunner (created by Gemini) with full Docker integration:

**Container Configuration:**
- Use `docker-py` library for container orchestration
- Base image: `backend:latest` (existing project image)
- Security hardening:
  - `--security-opt no-new-privileges:true`
  - `--cap-drop=ALL` (drop all Linux capabilities)
  - `--read-only=true` (immutable root filesystem)
  - `user='1000:1000'` (non-root execution)
  - Custom seccomp profile (15-01-PLAN.md Task 3)
- Resource limits:
  - `mem_limit="512m"` (hard limit)
  - `mem_reservation="256m"` (soft limit)
  - `cpu_quota=50000` (50% of one core)
  - `pids_limit=100` (max processes)
  - `network_mode="none"` (no network)
- Filesystem:
  - Mount `.sandbox/{run_id}/` as `/app` (rw)
  - tmpfs `/tmp` with 100MB limit
  - Clone `src/`, `tests/`, `pyproject.toml`, `mypy.ini` to sandbox dir

**6-Gate Implementation:**
1. Syntax Gate (host-side, <5s):
   - Use `ast.parse()` for Python validation (already implemented)
   - No Docker needed - pure static analysis
   - Fail fast before sandbox spin-up

2. Type Gate (sandbox, <30s):
   - Run `mypy --strict --config-file=mypy.ini src/` in container
   - Parse output for type errors
   - Return YELLOW on warnings, RED on errors

3. Unit Gate (sandbox, <120s):
   - Detect affected modules from changed files
   - Run `pytest -x --tb=short -q tests/unit/{module}/` for each
   - Stop on first failure for fast feedback
   - Return RED if any test fails

4. Contract Gate (sandbox, <30s):
   - Run `pytest -x -q tests/api/test_*_contract.py` for API changes
   - Validate Pydantic schema compatibility
   - Return RED on contract violations

5. Governance Gate (sandbox, <30s):
   - Run `python scripts/governance/risk_tier_gate.py --changed-files <files>`
   - Parse JSON output for Critical/High issues
   - Return RED on blockers, YELLOW on Medium

6. Rollback Gate (host-side, <30s):
   - Analyze git diff for reversibility
   - Check if schema migrations exist (if models/ changed)
   - Document rollback procedure in metadata
   - Return YELLOW if risky, GREEN if safe

**Timeout Handling:**
- Container execution timeout: 300s (5 minutes total)
- Individual gate timeouts enforced via `subprocess.run(timeout=...)`
- Exceeded timeout → kill container → return RED verdict

**Outcome Classification:**
- GREEN: All 6 gates pass → eligible for production
- YELLOW: Minor warnings → requires approval
- RED: Any gate fails → block + escalate

**Error Handling:**
- Docker daemon not running → graceful failure with clear error
- Container crashes → capture logs, return RED
- Filesystem setup errors → fail fast with diagnostic info

Create SQLAlchemy model to persist sandbox execution metadata:

```python
from sqlalchemy import Column, Integer, String, Text, TIMESTAMP, JSON, Numeric, Enum as SQLEnum
from sqlalchemy.sql import func
from src.models import db
import enum

class SandboxVerdict(enum.Enum):
    GREEN = "green"
    YELLOW = "yellow"
    RED = "red"

class SandboxRun(db.Model):
    __tablename__ = 'sandbox_runs'

    id = Column(Integer, primary_key=True)
    run_id = Column(String(36), unique=True, nullable=False)  # UUID

    # Remediation context
    failure_fingerprint = Column(String(255))  # Links to Sentry/failure
    remediation_type = Column(String(50))  # 'code_change', 'config', 'optimization'

    # Changes tested
    changed_files = Column(JSON, nullable=False)  # {path: content}
    blast_radius_files = Column(Integer, default=0)
    blast_radius_loc = Column(Integer, default=0)

    # Verification results
    verdict = Column(SQLEnum(SandboxVerdict), nullable=False)
    gate_results = Column(JSON)  # {gate_name: {status, duration_ms, output}}
    confidence = Column(Numeric(3, 2))  # Remediation confidence score

    # Execution metadata
    duration_ms = Column(Integer)
    container_id = Column(String(64))
    logs = Column(Text)

    # Timestamps
    created_at = Column(TIMESTAMP, server_default=func.now())
    completed_at = Column(TIMESTAMP)

    # Rollback info
    rollback_notes = Column(Text)
```

Create restrictive Seccomp profile blocking dangerous syscalls:

**Base from Docker default + additional restrictions:**
- Block: `mount`, `pivot_root`, `unshare`, `clone` with CLONE_NEWUSER flag
- Block: `ptrace`, `process_vm_readv`, `process_vm_writev` (debugging/injection)
- Block: `bpf`, `perf_event_open` (kernel instrumentation)
- Allow: Standard syscalls (read, write, open, stat, execve, etc.)

**Profile Structure:**
```json
{
  "defaultAction": "SCMP_ACT_ERRNO",
  "architectures": ["SCMP_ARCH_X86_64", "SCMP_ARCH_X86", "SCMP_ARCH_X32"],
  "syscalls": [
    {
      "names": [
        "accept", "accept4", "access", "alarm", "bind", "brk", "chdir",
        "clock_gettime", "close", "dup", "dup2", "execve", "exit",
        "fchdir", "fchmod", "fcntl", "fork", "fstat", "futex",
        "getcwd", "getpid", "getuid", "ioctl", "kill", "listen",
        "lseek", "mmap", "mprotect", "munmap", "open", "openat",
```
