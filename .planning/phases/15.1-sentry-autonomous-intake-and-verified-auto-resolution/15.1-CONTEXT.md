# Phase 15.1: Sentry Autonomous Intake + Verified Auto-Resolution - Context

**Gathered:** 2026-03-04  
**Status:** Ready for planning

<domain>
## Phase Boundary

Pull live Sentry production issues, route each through our self-healing pipeline, verify the remediation by calling the Sentry API, and close the loop by marking issues resolved only after confirmation there are no recurring events. The goal is to stop seeing the “Remediation failed … issue recurring” warnings because we only promote proven fixes.
</domain>

<decisions>
## Implementation Decisions

### Puller + Classifier Behavior
- Intake uses `scripts/observability/sentry_issue_puller.py` (cursor state, query=`is:unresolved level:error`, health JSON).  
- Health monitor (`scripts/daemons/health_monitor.py`) keeps polling Sentry, logs `status`/`issue_count`, and triggers `orchestrate_healers.py`.
- Every issue should produce a `FailureEvent` that includes `error_type`, `affected_module`, and `timestamp` so we can map it to the Neo4j/Graphiti root-cause.

### Auto-Resolution Criteria
- Call `src/core/sentry_client.py` with a real `SENTRY_AUTH_TOKEN`; if missing, log and skip closure (no mock `resolved` fallback).  
- The Sentry issue is “validated” only when `status == resolved` and the `activity` list contains no events after `SandboxRun.completed_at`.  
- If status remains `pending` or new events appear, keep the issue open and log a remediation failure message before raising it to the approval queue.

### Remediation Feedback Loop
- The existing `SentryFeedbackLoop` now tracks `SandboxRun.failure_fingerprint` and queries the API for that issue (statuses, activity, regression).  
- Successful validation promotes the fix to the template catalog; recurring events mark the run as failed and write to `FAILURE_JOURNEY.md`.  
- Include the Sentry issue ID in the remediation metadata (`run.failure_fingerprint`, `sandbox_runs` table) so telemetries tie back cleanly.

### Observability & Telemetry
- Emit new events to `.graph/sentry-issue-log.jsonl` (issue_id, timestamp, status, remediation_result, resolved_at).  
- Health monitor writes `auto_heal_running` flags to `.graph/health-cache.json` so other hooks know when closure is in-progress.  
- Firecrawl/Perplexity tool-health reports highlight network restrictions; note them in `15.1-RESEARCH.md`.
</decisions>

<specifics>
## Specific Ideas

- Add a guard in `get_sentry_client()` so the mock resolved response only runs in offline test mode, never during this phase.  
- Introduce a new daemon `scripts/graph/sentry_auto_resolver.py` (or extend `sentry_feedback_loop`) that runs every 5 minutes to re-check unresolved issues and log regression trends.  
- Update `scripts/daemons/health_monitor.py` to call the resolver when `sentry.status == issues` and to capture the `neo4j`/`deps` statuses as context for Sentry events.  
- Use Playwright to create smoke tests hitting the Sentry display page (if available) to confirm UI/hook behavior; capture these in the plan if feasible.
</specifics>
