# Phase 15.1 x Phase 16 Integration Map

Date: 2026-03-04  
Status: ACTIVE DESIGN INPUT

## Why They Overlap

Phase 15.1 owns remediation logic for Sentry issues.  
Phase 16 owns context/memory observability contracts.  
Combining them makes remediation flows explainable, auditable, and cross-terminal visible.

## Direct Integration Points

1. **Issue intake -> canonical memory event**
   - 15.1 source: `scripts/observability/sentry_issue_puller.py`
   - 16 contract: `src.memory.event_schema.create_event` + `src.memory.event_log.append_event`
   - outcome: every pulled issue is in `.memory/events/*.jsonl`.

2. **Issue triage -> graph-first context assembly**
   - 15.1 source: `src/graph/sentry_feedback_loop.py`
   - 16 contract: `src.assistant.context_broker.assemble_context`
   - outcome: triage attempts graph first and records fallback reason when degraded.

3. **Health monitor -> gate-compatible telemetry**
   - 15.1 source: `scripts/daemons/health_monitor.py`
   - 16 contract: telemetry fields (`graph_attempted`, `fallback_reason`, `assembled_tokens`, `latency_ms`)
   - outcome: Sentry auto-heal activity can be scored by Context OS reports.

4. **Batch completion -> deterministic view refresh**
   - 15.1 source: intake/resolve batch completion
   - 16 contract: `python scripts/memory/materialize_views.py --mode incremental`
   - outcome: working/short/long memory views stay current after issue cycles.

5. **Closure decision -> binary governance evidence**
   - 15.1 source: issue verified resolved
   - 16 contract:
     - `python scripts/governance/context_os_gate.py --window-hours 24`
     - `python scripts/context/verify_phase16.py --mode quick`
   - outcome: Sentry closure runs cannot silently degrade context OS health.

## Implementation Sequence (Recommended)

1. Add canonical event emission in puller + feedback loop.
2. Add graph-first broker call in remediation decision path.
3. Add incremental materializer trigger after batches.
4. Add tests for event payload + broker telemetry fields.
5. Add post-run check enforcing Context OS gate stays `GREEN`.

## Success Criteria

1. Every Sentry issue lifecycle step appears in `.memory/events/*.jsonl`.
2. Triage uses graph-first context with reason-coded fallback telemetry.
3. Cross-terminal operators can see live issue flow without manual log digging.
4. `context_os_gate.py` remains `GREEN` under active 15.1 issue processing.

## Layered Gate Strategy

1. **Primary gate owner (15.1):**
   - Sentry remediation correctness and recurrence handling decide closure.
2. **Compatibility gate owner (Phase 16 contracts):**
   - memory/event/telemetry conformance must be present and non-regressing.
3. **Classification rule when Context OS gate is RED:**
   - `sentry_path_regression`: blocking for 15.1 closure.
   - `external_context_drift`: non-blocking for 15.1 closure, but must be logged and remediated separately.
