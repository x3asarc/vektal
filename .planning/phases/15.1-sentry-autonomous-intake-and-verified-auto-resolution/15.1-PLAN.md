## Objective

Close the Sentry loop end to end:
1. Pull unresolved production issues.
2. Route each issue into the existing remediation pipeline.
3. Mark resolved only when Sentry confirms closure with no post-fix recurrence.

## Scope Now

1. Intake correctness (pagination, normalization, health output).
2. Single-fire auto-heal orchestration from health monitor.
3. Verified resolution logic in feedback loop and Sentry client.
4. Required tests and binary gate evidence.

## Not In Scope

1. New UI or dashboard work.
2. Non-Sentry incident sources.
3. Broad Context OS feature expansion outside Sentry-path compatibility.

## Success Metrics (Only)

1. Every pulled issue reaches one explicit lifecycle outcome: `verified_resolved` or `recurrence_logged`.
2. Production path cannot return mock-resolved when `SENTRY_AUTH_TOKEN` is missing.
3. Health monitor does not trigger duplicate auto-heal runs for the same issue fingerprint in one cycle.
4. Sentry lifecycle emits canonical Phase 16 memory events with required triage telemetry.
5. 15.1 closes only on Sentry-path correctness; unrelated context drift is tracked separately.
