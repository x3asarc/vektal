## 1. Sentry API Contracts

- **Issue listing:** `GET /api/0/projects/{org_slug}/{project_slug}/issues/` with `query=is:unresolved level:error`, `limit`, `cursor` paginated via `Link` header (`rel="next"`, `results="true"`).  
- **Issue update:** `PUT /api/0/organizations/{org_slug}/issues/{issue_id}/` with body `{ "status": "resolved" }` (also `resolvedInNextRelease`, `ignored`).  
- **Status change schema:** `payload_type=STATUS_CHANGE` uses `GroupStatus`/`GroupSubStatus`; update path accepts `fingerprint`, `project_id`, `new_status`. Use this to mark closure once the auto-heal passes verification.

## 2. Verification & Feedback Requirements

- `SentryFeedbackLoop` must call `get_issue(issue_id)` and ensure `status == resolved` plus no `activity` dated after `SandboxRun.completed_at`.  
- When the API request fails (network, token missing), log the event, mark the run `pending`, and do not auto-close.  
- For observed `issue recurring` warnings, correlate `SandboxRun` with `failure_fingerprint` (format `SENTRY-123:module:error`); use new metadata fields (issue_id, timeline) to fill `.graph` logs.
