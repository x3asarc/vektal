<task type="auto">
  <name>Task 2: Implement RFC 7807 error handling</name>
  <files>src/api/core/errors.py</files>
  <action>
Create src/api/core/errors.py implementing RFC 7807 Problem Details format.

Required components:

1. ProblemDetails class with static methods:
   - `validation_error(error: ValidationError, status=400)` - Convert Pydantic ValidationError to Problem Details with field-level errors
   - `business_error(error_type: str, title: str, detail: str, status=400, **extensions)` - Generic business logic errors
   - `not_found(resource: str, identifier: Any)` - 404 errors
   - `rate_limit_exceeded(retry_after: str)` - 429 errors
   - `unauthorized(detail: str = "Authentication required")` - 401 errors
   - `forbidden(detail: str = "Insufficient permissions")` - 403 errors

2. Error response format (RFC 7807):
   ```python
   {
       "type": "https://api.shopify-supplier.com/errors/{error_type}",
       "title": "{Human-readable title}",
       "status": {HTTP status code},
       "detail": "{Specific error message}",
       "fields": {...}  # Extension for validation errors only
   }
   ```

3. `register_error_handlers(app)` function that registers:
   - ValidationError handler (Pydantic)
   - HTTPException handler (Werkzeug)
   - RateLimitExceeded handler (Flask-Limiter)
   - 404, 401, 403, 429, 500 handlers
   - Generic Exception catch-all (sanitize in production)

4. Security considerations:
   - Production mode: Generic error messages, no stack traces
   - Development mode: Detailed errors for debugging
   - Check `app.config.get("ENV") == "production"` or `app.debug == False`

Example validation error response:
```json
{
  "type": "https://api.shopify-supplier.com/errors/validation-error",
  "title": "Validation Failed",
  "status": 400,
  "detail": "Request validation failed on one or more fields",
  "fields": {
    "email": "Invalid email format",
    "password": "Must be at least 8 characters"
  }
}
```

  <files>src/api/core/pagination.py</files>

Create src/api/core/pagination.py with cursor-based and offset-based pagination.

Required components:

1. Pydantic schemas for pagination parameters:
   ```python
   class CursorPaginationParams(BaseModel):
       cursor: str | None = None
       limit: int = Field(default=50, ge=1, le=100, description="Items per page")

   class OffsetPaginationParams(BaseModel):
       page: int = Field(default=1, ge=1, description="Page number")
       limit: int = Field(default=50, ge=1, le=100, description="Items per page")
   ```

2. Cursor encoding/decoding functions:
   ```python
   def encode_cursor(last_id: int, last_timestamp: str) -> str:
       """Encode cursor for opaque pagination (base64 JSON)."""

   def decode_cursor(cursor: str) -> tuple[int, str]:
       """Decode cursor to extract (id, timestamp)."""
   ```

3. Pagination response builder:
   ```python
   def build_cursor_response(items: list, has_next: bool, limit: int, cursor_field: str = "created_at") -> dict:
       """Build pagination response with next_cursor if has_next."""

   def build_offset_response(items: list, page: int, limit: int, total: int) -> dict:
       """Build offset pagination response with page info."""
   ```

4. Usage guidance in docstrings:
   - Cursor pagination: For products, jobs, vendors (large/changing datasets)
   - Offset pagination: For admin views, user lists (small/static datasets)

Use base64.urlsafe_b64encode/decode for cursor encoding (URL-safe).

2. Tier limit definitions (from RESEARCH.md recommendations):
   ```python
   TIER_LIMITS = {
       UserTier.TIER_1: "100 per day",    # $29/mo Starter
       UserTier.TIER_2: "500 per day",    # $99/mo Professional
       UserTier.TIER_3: "2000 per day",   # $299/mo Enterprise
   }

   DEFAULT_UNAUTHENTICATED_LIMIT = "10 per hour"
   DEFAULT_LIMITS = ["200 per day", "50 per hour"]
   ```

3. Dynamic key function for rate limiting:
   ```python
   def get_rate_limit_key():
       """Return user ID for authenticated users, IP for anonymous."""
       if current_user.is_authenticated:
           return f"user:{current_user.id}"
       return f"ip:{get_remote_address()}"
   ```

4. Dynamic limit function for tier-based limits:
   ```python
   def get_user_tier_limit():
       """Return rate limit string based on user tier."""
       if not current_user.is_authenticated:
           return DEFAULT_UNAUTHENTICATED_LIMIT
       return TIER_LIMITS.get(current_user.tier, DEFAULT_UNAUTHENTICATED_LIMIT)
   ```

5. Limiter factory function:
   ```python
   def create_limiter(app):
       """Create and configure Flask-Limiter with Redis backend."""
       import os
       redis_url = os.getenv("REDIS_URL", "redis://redis:6379/1")

       limiter = Limiter(
           key_func=get_rate_limit_key,
           app=app,
           storage_uri=redis_url,
           storage_options={"socket_keepalive": True},
           default_limits=DEFAULT_LIMITS
       )
       return limiter
   ```

6. Decorator for tier-based endpoints:
   ```python
   # Usage: @limiter.limit(get_user_tier_limit)
   # Applied per-endpoint in route handlers
   ```

Note: The limiter instance will be created in Plan 02 during app initialization.
