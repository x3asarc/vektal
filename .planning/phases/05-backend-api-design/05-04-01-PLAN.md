       """Register before_request/after_request hooks for user API version enforcement."""

   - Add explicit TODO marker for Phase v2 data-shape transformations

Create a dedicated blueprint for user-controlled version migration lifecycle.

1. Create src/api/v1/versioning/__init__.py:
   - Define `versioning_bp = Blueprint('versioning_v1', __name__)`
   - Import routes for registration
   - Export `__all__ = ['versioning_bp']`

2. Create src/api/v1/versioning/schemas.py with Pydantic models:
   - `ApiVersionStatusResponse`
     - `current_version: str`
     - `available_versions: list[str]`
     - `lock_until: datetime | None`
     - `rollback_available: bool`
   - `MigrateToV2Response`
     - `previous_version: str`
     - `new_version: str`
     - `migration_steps: list[str]`
     - `rollback_available_until: datetime | None`
   - `RollbackToV1Response`
     - `previous_version: str`
     - `new_version: str`

3. Create src/api/v1/versioning/routes.py endpoints:
   - `GET /version`:
     - returns current user's version state
   - `POST /migrate-to-v2`:
     - requires auth
     - if already v2, return idempotent success response
     - call `run_user_migration(current_user.id, 'v2')`
     - on success: set `api_version='v2'`, set `api_version_locked_until=now()+24h`, commit
     - on failure: return ProblemDetails.business_error("migration-failed", ..., 422)
   - `POST /rollback-to-v1`:
     - requires auth
     - allowed only when user is v2 and lock window still active
     - set `api_version='v1'`, clear lock, commit
     - on invalid state/window: return ProblemDetails.business_error("rollback-not-allowed", ..., 409)

4. Error behavior:
   - All endpoint failures must return RFC 7807 payloads
   - Do not redirect API clients for migration/rollback errors
  </action>

- routes.py defines version, migrate-to-v2, rollback-to-v1 handlers

  </verify>
