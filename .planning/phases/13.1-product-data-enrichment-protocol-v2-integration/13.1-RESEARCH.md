## Summary

Phase 13.1 should deliver one **canonical enrichment protocol** that integrates existing `src/core/enrichment` assets with Phase 12 capability gates and Phase 13 reliability controls. The objective is to produce high-quality retrieval-ready product data, not merely fill random attributes.

The research outcome is clear:
1. Keep merchant data authoritative by default.
2. Use profile-driven enrichment depth (`Quick`, `Standard`, `Deep`) with tier-aware economics.
3. Route uncertain/conflicting outcomes through a shared Oracle framework.
4. Expand enrichment payload to a broad, conditional semantic contract (identity, taxonomy, variants, composition, media semantics, provenance).
5. Reuse idempotency, queue isolation, and observability controls from Phase 13.

Because Oracle reuse is needed outside enrichment, a separate Phase 13.2 is appropriate and now locked in roadmap as a small integration phase.

**Primary recommendation:** implement 13.1 with a retrieval-readiness contract and field eligibility matrix, then consume shared Oracle adapters from 13.2 instead of building local verifier logic.

## Architecture Patterns

### Pattern 1: Audit-Then-Enrich
1. Audit store metafield definitions + scope.
2. Build allowed/blocked write plan.
3. Execute enrichment against allowed surface.
4. Emit blocked-field report and upgrade guidance.

### Pattern 2: Merchant-First Arbitration
1. If merchant value is meaningful: keep.
2. If missing/generic: enrich.
3. If conflict: Oracle suggestion/hold path, not blind overwrite.

### Pattern 3: Profile Gears
- `Quick`: text extraction and minimal mapping.
- `Standard`: richer mapping/validation.
- `Deep`: visual checks + second-opinion + advanced conflict handling.

### Pattern 4: Reusable Oracle Adapters
- `Execution Oracle`: persistence/side-effect verification.
- `Content Oracle`: semantic second-opinion.
- `Visual Oracle`: image-grounded checks (for example color contradiction).
- `Policy Oracle`: threshold/scope/guardrail adjudication.

Unified Oracle output:
- `decision`, `confidence`, `reason_codes`, `evidence_refs`, `requires_user_action`.

## Retrieval-Readiness Payload (Broad, Conditional)

Mandatory research conclusion: enrichment scope must exceed hex/finish/material and be conditional by product applicability.

Field families:
1. Identity/taxonomy normalization.
2. Variant and pack semantics.
3. Physical/material/composition/finish attributes.
4. Compliance/logistics descriptors when available.
5. Media semantics (visual signals + OCR cues).
6. Trust/provenance/confidence/conflict lineage.
7. Retrieval support surfaces (facet fields, sparse keyword fields, synonym/alias expansions, multilingual normalization fields).

Use an **eligibility matrix** per product class to avoid irrelevant-field noise and retrieval degradation.

## Don’t Hand-Roll

| Problem | Don’t build | Use instead | Why |
|---|---|---|---|
| per-phase verifier forks | bespoke enrichment verifier | shared Oracle framework + adapters | consistent behavior and telemetry |
| unrestricted enrichment writes | direct mutation by inference only | audit + policy-gated write plan | safe permission handling |
| always-full reruns | recompute everything every time | hash/idempotent selective rerun | lower cost and latency |

## Common Pitfalls

1. **Schema drift** from ungoverned namespace/key creation.  
2. **Confidence illusions** from single-modality inference.  
3. **Retrieval noise** from over-populating optional fields without applicability checks.  
4. **Runtime starvation** when deep enrichment competes with interactive queues.  
5. **Merchant trust erosion** when inferred values silently overwrite curated values.

## Success Metrics (Recommended)

1. **Retrieval-Readiness Coverage**  
`% products passing critical enrichment contract`.

2. **Color/Finish Structured Accuracy Delta**  
Audited delta between inferred structured color/finish values and accepted ground truth.

3. **Field Acceptance Quality**  
`% enriched fields accepted without manual correction` by field family/profile.

4. **Semantic Search Uplift**  
Change in `Recall@K / nDCG@K` on fixed gold query benchmark pre/post enrichment.
