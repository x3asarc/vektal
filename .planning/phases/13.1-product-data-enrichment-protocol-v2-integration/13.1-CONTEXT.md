# Phase 13.1: Product Data Enrichment Protocol v2 Integration - Context

**Gathered:** 2026-02-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Create a comprehensive, production-grade product enrichment protocol as a selectable platform module (peer to existing product surfaces), integrating existing Phase 2.2 capabilities and side-project learnings into current API/job/database/frontend/governance flows.

</domain>

<decisions>
## Implementation Decisions

### Pre-Context Alignment
- Course selected: A
- High-level direction: integrated module-first enrichment protocol, not isolated script flow.
- Portfolio rationale: leverage completed 2.2 foundation, eliminate drift, and align with Phase 13 reliability/governance before Phase 14 optimization.

### Invocation, Access, and Goal Priority
- Primary objective priority is retrieval quality for downstream vectorization/semantic search; SEO gains are secondary outcomes.
- Enrichment is user-invoked by explicit choice, not automatic default background execution.
- Candidate scope is any product set selected by the user.
- Access should be gated above Tier 1 (Tier 2/3 policy decision to be finalized during planning against existing capability matrix).

### Product Surface & Platform Integration
- Enrichment must be user-selectable as its own platform feature/workspace, similar to existing major surfaces (chat, precision workflow, settings).
- Enrichment work must integrate with the main project execution surfaces (`.planning`, API/jobs/db/docker/runtime contracts), not remain standalone under `side-project`.
- Existing `side-project` artifacts are treated as source material and benchmark input, not the deployment boundary.

### Scope Strategy (Combine, Do Not Duplicate)
- Merge best parts of existing Phase 2.2 and side-project protocol behavior into one canonical enrichment v2 flow.
- Avoid parallel duplicate implementations; keep one canonical path with clear migration/cutover steps.
- Use current best practices with lightweight code and high-throughput operation.

### Runtime Packaging & Ops Constraints
- Separate Docker container is optional, not mandatory; preference is feature-level separation in product surface and contracts.
- Dependency policy must prevent repeated heavyweight ML runtime downloads (explicit concern: repeated ~13GB PyTorch/CUDA pulls).
- Protocol should default to model/runtime choices that avoid unnecessary heavyweight GPU dependency acquisition unless explicitly enabled.

### Confidence and Conflict Handling Direction
- Field precedence and conflict policy should be evidence-driven from research + verifier logic, not arbitrary hardcoding.
- If precedence cannot be resolved confidently, batch/item should be flagged for explicit user decision with internal developer visibility.
- Low-confidence inferred values should support a secondary opinion pass (separate model/agent) before final write recommendation.
- Existing phase artifacts and codebase should be checked/reused first for any verifier/oracle workflow before introducing a new pattern.

### Locked Policy Recommendations (User-Backed)
- Merchant protection policy: `A` (merchant-first, enrichment-augmented).
  - Merchant-provided values are default authority and are not auto-overwritten.
  - If merchant value is missing/generic, enrichment may fill.
  - High-confidence contradictions become suggested changes with verifier pass, not blind overwrite.
- Enrichment run profiles: `A` (`Quick`, `Standard`, `Deep`).
  - `Quick` (Tier 2): text-only, low-cost pass.
  - `Standard` (Tier 2/3): multi-source structured mapping.
  - `Deep` (Tier 3): includes vision-driven attributes and secondary-opinion verification.
- Second-opinion trigger: `B` (low-confidence and structural conflict).
- Permission failure behavior: `B` (partial apply with blocked-field report), with breaker-based stop rules for repeated permission-denied sequences.
- Language output model: `C` (user-selectable per run; German and English support in same protocol).
- Contract strictness: `C` (strict critical fields; soft optional fields with debt reporting).

### Localization Direction
- German is mandatory for current Bastelschachtel usage.
- English support should be enabled in same protocol path if operationally feasible.
- User-selectable output language should be supported as a feature contract (store/user preference driven).

### Shopify Capability, Schema, and Permission Gate
- Before enrichment execution, run a mandatory Shopify capability audit for each connected store:
  - current product/category metafield definitions and populated fields,
  - namespace/key/type inventory and conflicts,
  - current app permission scope (read/write capabilities for affected resources).
- Enrichment planning and apply behavior must be constrained by that audit result:
  - only create/update definitions/values that are permitted and valid for the store,
  - emit clear blocked reasons when scope is insufficient,
  - provide a deterministic upgrade path for missing permissions.

### Metafield Enrichment Requirements
- Metafield enrichment should be comprehensive and Shopify-best-practice aligned:
  - support product and category enrichment targets where available,
  - prefer structured, typed fields over free-form text,
  - keep canonical mappings to avoid duplicate/fragmented metadata keys.
- Finishing effects (for example matte, glossy, metallic) should be extracted and mapped into structured metafields.

### Color Enrichment Requirements
- If color signal exists in product title but structured color field is missing, enrich color metadata rather than leaving null.
- Color payload should include a hex code.
- Hex code should be derived using image-aware analysis (Vision AI) so structured color reflects product appearance, not string guessing only.
- Implementation should support confidence-aware fallback behavior when image evidence is weak or conflicting.

### Semantic Retrieval Readiness
- Enrichment outputs must be shaped for downstream vectorization and semantic search quality:
  - structured, normalized attribute fields,
  - consistent metadata taxonomy,
  - clear provenance/confidence signals for inferred fields.
- Semantic payload should be comprehensive and conditional by product eligibility (not a fixed minimal trio):
  - identity and taxonomy: canonical title, normalized product_type, category path/taxonomy nodes, brand/vendor normalization;
  - commercial structure: pack size, unit normalization, variant relationships, bundle/set semantics;
  - physical/technical attributes: dimensions, weight, material composition, finish/effect, compatibility/use-case constraints;
  - compliance/logistics attributes where available: HS code, country of origin, safety/composition disclosures;
  - media semantics: dominant/secondary colors, style cues, extracted on-image text;
  - quality and trust signals: provenance, confidence, contradiction flags, manual overrides.
- Planning/research must define an eligibility matrix so fields are extracted only when relevant to that product class.

### Oracle / Verifier Architecture (Cross-Phase, Reusable)
- Do not build a one-off verifier only for enrichment.
- Build one shared Oracle framework with domain-specific adapters:
  - `Execution Oracle` for did-it-apply checks (already aligned with Phase 13 verification primitives).
  - `Content Oracle` for enrichment truth arbitration (confidence, conflicts, second opinion).
  - `Visual Oracle` for vision-grounded attribute verification (for example vision-to-hex consistency and text-vs-image contradiction checks).
  - `Policy Oracle` for threshold/permission/guardrail adjudication.
- Keep one orchestration contract and multiple oracle strategies rather than many isolated tools:
  - consistent telemetry, idempotency, and retry behavior,
  - domain-specific prompt/rule packs per use case,
  - reusable across Phase 13.1, 14, and 15.
- Oracle output contract should be deterministic:
  - `decision` (`accept`, `suggest`, `hold`, `reject`)
  - `confidence`
  - `reason_codes`
  - `evidence_refs`
  - `requires_user_action` (bool)

### Research Breadth Mandate (Locked)
- Research for 13.1 must not be limited to already-decided fields (hex, finish/effect, material).
- Research must produce a broader enrichment factor map for semantic retrieval quality, including:
  - attribute families, optional-vs-critical classification, and category-specific applicability;
  - Shopify-native modeling options (core fields vs metafields vs metaobjects where applicable);
  - retrieval-oriented modeling patterns (facets, dense/sparse support fields, synonym/alias surfaces, multilingual normalization);
  - quality-control patterns (confidence calibration, contradiction handling, evaluator coverage, override learning hooks).

### Claude's Discretion
- Container topology can remain shared service-first if protocol boundaries are clean and measurable.
- Exact embedding/runtime implementation details may be chosen for lowest operational friction while preserving retrieval quality.
- Cutover mechanics (shadow, staged, or direct) can be proposed in planning based on risk and testability.

</decisions>

<specifics>
## Specific Ideas

- "Not isolated" is a hard requirement; enrichment must be represented as an integrated product capability.
- "Best of both, combine" is the implementation principle for 2.2 + side-project.
- Avoid repeated large ML framework downloads became a concrete operational pain point and must be addressed in protocol design.
- Shopify schema and scope introspection is a first-class pre-step, not an optional check.
- Vision AI is the preferred mechanism for deriving product color hex when missing structured color data.
- Enrichment should directly improve later vectorization and semantic retrieval phases.
- Secondary-opinion handling for low-confidence values should be designed as part of this phase's governed flow.
- Vision-to-hex should include contradiction checks (title/text color vs image-derived color) and conflict flags.
- Idempotent enrichment runs should reuse cached results when product source hash has not changed.
- Throughput controls must reuse Phase 13 throttling/rate-limit safeguards so enrichment does not starve chat/runtime traffic.
- Proposed Phase 13.1 success metrics:
  - Retrieval-readiness coverage rate.
  - Color/finish structured accuracy delta on audited samples.
  - Semantic retrieval uplift on gold query set (post-vectorization benchmark coupling).

</specifics>
