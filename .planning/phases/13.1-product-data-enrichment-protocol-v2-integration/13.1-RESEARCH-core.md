## Summary

Phase 13.1 should be implemented as a governed enrichment capability, not as a standalone pipeline clone. The current codebase already provides a substantial enrichment engine (`src/core/enrichment/*`), tier/capability routing (Phase 12), reliability/governance controls (Phase 13), and resolution confidence pathways (Phase 8/10). The correct approach is to unify these into one canonical protocol that is retrieval-first, merchant-safe, and operationally bounded.

Primary recommendation: deliver a **semantic-retrieval enrichment contract** that is broader than color/finish/material and conditional by product type. Enrichment should produce normalized attributes, provenance/confidence lineage, and contradiction flags suitable for both approval workflows and downstream vectorization.

Secondary recommendation: implement one reusable Oracle framework with adapters:
- `Execution Oracle` (already grounded in Phase 13 verification primitives),
- `Content Oracle` (second-opinion arbitration),
- `Visual Oracle` (vision-backed attribute checks such as color contradiction),
- `Policy Oracle` (permission/threshold adjudication).

Explicit alignment retained from context:
- Product-surface parity is mandatory.
- Heavy runtime re-download avoidance is mandatory operationally.
- Retrieval modeling includes facets, sparse keyword surfaces, synonym/alias normalization, and multilingual normalization.
- Color/finish structured accuracy remains a tracked metric.

## Architecture Patterns

### Pattern 1: Capability Audit Before Enrichment
1. Read current store metafield definitions + used namespaces/keys/types.
2. Resolve app permission scope against intended writes.
3. Build `allowed_write_plan` + `blocked_write_plan`.
4. Run enrichment against `allowed_write_plan`; report blocked fields with upgrade guidance.

### Pattern 2: Merchant-Protected Arbitration
1. If merchant value exists and is non-generic: keep value.
2. If inferred value conflicts:
   - run oracle pass (content + optional visual),
   - emit suggestion/conflict ticket instead of overwrite.
3. If merchant value missing/generic: fill with confidence/provenance.

### Pattern 3: Profile-Based Depth
- `Quick` (Tier 2): text extraction + mapping only.
- `Standard` (Tier 2/3): multi-source mapping + validations.
- `Deep` (Tier 3): visual checks + second-opinion oracle + advanced conflict handling.

### Pattern 4: Hash-Driven Idempotent Enrichment
- Hash product semantic inputs (title/body/media refs/critical attributes).
- If unchanged and policy version unchanged: reuse cached enrichment result.
- If changed: run only affected extraction blocks where feasible.

## Retrieval-Oriented Enrichment Factor Map (Broad Scope)

Minimum trio is insufficient. Research indicates the payload should include:

1. **Identity and taxonomy**
- normalized title
- product type
- category/taxonomy nodes
- brand/vendor canonical + aliases

2. **Variant and pack semantics**
- option normalization (size/color/style)
- pack quantity and unit normalization
- bundle/set indicators

3. **Material and composition**
- material list
- composition percentages when available
- finish/effect vocabulary

4. **Physical and compatibility attributes**
- dimensions/weight/volume normalized
- use-case tags and compatibility constraints

5. **Compliance/logistics attributes**
- HS code
- country of origin
- relevant safety/compliance descriptors when available

6. **Media semantics**
- dominant/secondary colors
- visual-text/OCR cues
- contradiction marker between textual and visual claims

7. **Trust and control metadata**
- provenance per field (`merchant_provided`, `ai_inferred`, `vision_verified`, etc.)
- confidence per field
- conflict ticket link/reference

8. **Retrieval support metadata**
- facet-ready attributes for filters
- sparse keyword support fields
- synonym/alias surfaces
- multilingual normalization fields

## Don’t Hand-Roll

| Problem | Don’t build | Use instead | Why |
|---|---|---|---|
| Per-feature verifier logic | one-off enrichment verifier | shared Oracle adapter framework | cross-phase consistency + telemetry reuse |
| Unbounded enrichment writes | direct writes without capability gate | Shopify capability audit + policy adapter | deterministic permission behavior |
| Monolithic full reruns | always recompute all fields | hash/idempotent selective rerun | cost + latency control |

## Common Pitfalls

### Pitfall 1: Schema Drift Across Metafields
- Cause: creating keys ad hoc per run/vendor.
- Prevent: canonical namespace/key registry + typed contracts.

### Pitfall 2: False Confidence From Single Modality
- Cause: relying only on text or only on image.
- Prevent: multimodal contradiction checks and confidence routing.

### Pitfall 3: Retrieval Regression Despite More Fields
- Cause: noisy optional fields polluting embedding payload.
- Prevent: critical/optional split + eligibility matrix by product class.

### Pitfall 4: Enrichment Starving Runtime Traffic
- Cause: high-volume batch jobs share same token/worker budget as interactive paths.
- Prevent: queue isolation and throttling tied to Phase 13 budgets.

## Recommended Success Metrics (Phase 13.1)

1. **Retrieval-Readiness Coverage**  
`% of products passing critical enrichment contract`.

2. **Color/Finish Structured Accuracy Delta**  
Audited delta between inferred structured color/finish and accepted ground truth.

3. **Field Acceptance Quality**  
`% enriched fields accepted without manual correction`, segmented by field family and profile depth.

4. **Search Uplift Benchmark**  
Improvement in `Recall@K / nDCG@K` on a fixed gold query set (baseline vs post-enrichment).
