## Executive Summary

Phase 13.1 sits at a critical junction: it must unify historical enrichment implementations into one production-grade capability while preserving the reliability and governance guarantees established in Phase 13. The core challenge is not feature invention; it is controlled unification. The project already contains extraction, AI generation, quality scoring, and embedding machinery in `src/core/enrichment`, plus strong governance contracts for approval, idempotency, verification, and observability. The winning approach is therefore to compose these primitives under one canonical, retrieval-first enrichment protocol with clear capability gates, typed output contracts, and shared verifier orchestration.

The primary business objective is semantic retrieval quality for downstream vectorization. This changes how enrichment success must be measured: not by “number of fields filled,” but by retrieval readiness and ranking improvement. A narrow enrichment payload (for example only color hex, finish, and material) will not deliver robust semantic search. Research supports a broader, conditional payload model: identity/taxonomy normalization, variant and pack semantics, physical/compliance attributes, media semantics, and provenance/confidence lineage. These attributes should be extracted only when relevant to the product class through an eligibility matrix; otherwise noise increases and retrieval quality can degrade.

Merchant protection is a hard boundary. Enrichment should augment merchant authority, not silently overwrite it. Where conflicts arise, second-opinion adjudication should run via a shared Oracle framework and produce deterministic outcomes (`accept`, `suggest`, `hold`, `reject`) with reason codes and evidence references. This shared Oracle architecture should not be enrichment-only: it should be reusable across execution verification, content arbitration, visual checks, and policy gating. That is why introducing a small Phase 13.2 for Oracle reuse is technically sound and reduces duplication risk across 13.1/14/15.

Operationally, 13.1 should reuse Phase 13 hardening directly: hash-driven idempotency, queue/budget isolation, explicit permission audit against Shopify capability scope, and telemetry parity. Enrichment profiles (`Quick`, `Standard`, `Deep`) map cleanly to tiered runtime economics and user intent. Language should be selectable per run, with German and English first-class.

Context-critical constraints retained explicitly:
- Enrichment is delivered as a first-class platform workspace capability, not a side-project-only flow.
- Runtime packaging defaults must avoid repeated heavyweight runtime downloads unless explicitly enabled.
- Retrieval payload design must include facet, sparse keyword, synonym/alias, and multilingual normalization surfaces.
- Color/finish structured accuracy delta is a first-class quality metric.

Recommended outcome: deliver 13.1 as the canonical enrichment protocol and payload contract, while 13.2 establishes reusable Oracle adapters consumed by 13.1 and later phases.

## Domain Overview

### Problem Space
Build a single enrichment protocol that:
1. Is Shopify-schema aware and permission-safe.
2. Produces retrieval-optimized structured data for vectorization.
3. Preserves merchant intent and safety contracts.
4. Scales economically with tiered depth.

### Glossary (short)
- **Retrieval-readiness:** a product satisfies critical structured fields and provenance required for semantic indexing.
- **Eligibility matrix:** rules that determine which fields apply to which product classes.
- **Oracle adapter:** domain-specific verifier implementation behind shared contract.
- **Conflict ticket:** explicit record of unresolved contradiction needing user/developer decision.
- **Profile gear:** execution depth mode (`Quick`, `Standard`, `Deep`).

### Evolution Context
- Phase 2.2 built enrichment mechanics.
- Phase 12 introduced capability/tier routing.

## Core Theme 1: Canonical Protocol Unification

### Definition and Purpose
Unify legacy 2.2 + side-project behavior into one governed enrichment protocol integrated with platform APIs/jobs/state.

### Architecture/Workflow
1. Input selection (user-invoked product set).
2. Shopify capability/schema audit.
3. Profile execution (`Quick`/`Standard`/`Deep`).
4. Oracle arbitration for uncertain/conflicting outputs.
5. Dry-run/approval-compatible output and apply.
6. Retrieval payload + telemetry persistence.

### Benefits
- Eliminates drift and duplicate implementations.
- Preserves one source of truth for enrichment logic.
- Improves downstream search consistency.

### Tradeoffs/Risks
- Requires careful migration and compatibility mapping.
- Larger first integration wave than local patching.

### Implementation Notes
- Keep protocol contract versioned.
- Keep field mapping registry canonical.
- Keep profile semantics explicit and testable.

## Core Theme 2: Semantic Search-Oriented Enrichment Scope

### Definition and Purpose
Move from narrow attribute filling to retrieval-meaningful semantic structuring.

### Required Factor Families
1. **Identity/Taxonomy**
   - canonical title, normalized product type, category path, aliases
2. **Variant/Pack Semantics**
   - option normalization, unit normalization, bundle/set metadata
3. **Physical/Material/Finish**
   - composition, finish/effect, dimensions, weight
4. **Compliance/Logistics**
   - HS code, origin, constraints (if present)
5. **Media Semantics**
   - visual color, OCR cues, contradiction markers
6. **Trust Metadata**
   - source, confidence, reason codes, overrides
7. **Retrieval Support Surfaces**
   - facet-friendly normalized fields
   - sparse keyword support fields
   - synonym/alias expansions
   - multilingual normalization attributes

### Benefits
- Higher retrieval precision and recall.
- Better explainability for AI-generated changes.
- Stronger future SEO and optimization support.

### Tradeoffs
- More extraction complexity.
- More schema governance and optionality management.

### Key Rule
Use an eligibility matrix so enrichment remains conditional and relevant by product class.

## Core Theme 3: Merchant Protection + Conflict Arbitration

### Definition and Purpose
Ensure merchant-provided truth remains authoritative while enabling useful augmentation.

### Pattern
- Missing/generic merchant values: enrich directly.
- Conflicting values: route to Oracle, return suggestion/conflict ticket.
- No blind overwrite of meaningful merchant values.

### Benefits
- Reduces trust erosion.
- Keeps legal/operational accountability clear.

### Risks
- Higher review load if confidence calibration is poor.

### Mitigation
- Tune thresholds per field family.
- Batch suggestions to reduce UI noise.

## Core Theme 4: Shared Oracle Framework (Cross-Phase Reuse)

### Definition and Purpose
One reusable verifier substrate with domain adapters.

### Adapter Model
- **Execution Oracle:** persistence/side-effect confirmation.
- **Content Oracle:** second-opinion semantic arbitration.
- **Visual Oracle:** vision-backed checks (for example text-vs-image color mismatch).
- **Policy Oracle:** permission/threshold compliance decisions.

### Unified Output Contract
- `decision`
- `confidence`
- `reason_codes`
- `evidence_refs`
- `requires_user_action`

### Benefits
- Consistent telemetry and governance.
- Reduced implementation duplication.
- Cleaner integration into future phases.

### Tradeoffs
- Requires small up-front framework work (captured in Phase 13.2).

## Core Theme 5: Operational Hardening Reuse from Phase 13

### Definition and Purpose
Apply existing reliability contracts to enrichment workloads.

### Required Reuse
- Idempotency via product-state hash + policy version.
- Queue isolation and throttling to protect chat/interactive traffic.
- Structured retention/observability for enrichment outcomes.
- Deferred/held states for unresolved verification.

### Failure Modes

- over-budget deep runs,
- stale cache mismatches,

### Prevention

- versioned cache keys,
- conflict ticket triage rules.

## Best Practices and Recommendations

### Setup
- Build namespace/key/type registry before writes.
- Define critical vs optional fields per product family.
- Add per-profile budgets and queue controls.

### Implementation
- Always capability-audit store before write planning.
- Always attach provenance/confidence per enriched field.
- Use Oracle pass for low-confidence/conflict paths.

### Testing
- Contract tests for profile outputs.
- Conflict arbitration tests (merchant vs inferred).
- Permission-failure partial apply tests.
- Retrieval benchmark tests on fixed gold query set.

### Maintenance
- Version policy and payload schema.
- Monitor manual override rate per field family.
- Track contradiction drift by supplier/product family.

### Scaling
- Separate deep profile queue from interactive path.
- Use hash-based cache reuse.
- Tune HNSW/IVFFlat + metadata filter indexes for workload shape.
- Keep runtime packaging lightweight by default to prevent repeated heavy dependency pulls during normal operation.

## Edge Cases, Pitfalls, and Failure Modes

1. **Title says one color; image indicates another**
- Detect: Visual Oracle contradiction reason code.
- Prevent/Fix: hold/suggest, not auto-overwrite.

2. **Metafield schema mismatch at write time**
- Detect: blocked-field report + reason code.
- Prevent/Fix: preflight schema audit; plan update path.

3. **Over-enrichment noise harming retrieval**
- Detect: retrieval metrics stagnate while field volume rises.
- Prevent/Fix: tighten eligibility matrix and optional field weighting.

4. **Deep profile cost spikes**
- Detect: rising token/cost per product with low acceptance uplift.
- Prevent/Fix: stricter deep triggers and adaptive quotas.

5. **Merchant correction churn**
- Detect: high manual correction frequency on same field/supplier.
- Prevent/Fix: feed policy updates and negative constraints.
